## 2026-06-28

### D-001 · 공식 로그 훅을 기본 수집 방식으로 사용

**상황**

AX 해커톤은 AI 대화 로그 전체를 편집·발췌 없이 제출하도록 요구합니다. 프로젝트에는 이미 `.claude/settings.json`, `.codex/hooks.json`, `tools/save_log.py`가 존재했습니다.

**검토한 선택지**

1. 매 세션 종료 후 대화를 수동 복사합니다.
2. 임의 형식의 새 로거를 처음부터 작성합니다.
3. 대회가 제공한 공식 로그 훅을 기준으로 사용합니다.

**결정**

- 공식 `log-hooks.zip`의 훅과 저장 스크립트를 기본 수집 방식으로 사용합니다.

**근거**

- 대회가 공식 제공한 방식이므로 제출 형식과 평가 의도에 가장 가깝습니다.
- 초기 프로젝트 세 파일의 해시가 공식 ZIP과 일치했습니다.
- 저장 스크립트가 선택한 대화 이벤트의 원본 JSONL 행을 그대로 보존합니다.

**영향 및 재검토 조건**

- `logs/` 파일은 수동으로 편집하지 않습니다.
- 공식 훅이 특정 실행 환경에서 작동하지 않을 때만 원본 보존 원칙을 유지하는 보완 수집기를 사용합니다.

### D-002 · Codex IDE용 프로젝트 경로 기반 감시기 보완

**상황**

현재 세션은 Codex CLI가 아닌 IDE 확장에서 실행됐고, 공식 CLI Stop 훅만으로 마지막 응답이 즉시 반영되지 않는 현상이 확인됐습니다. Windows 한글 경로에서는 표준입력 인코딩 문제도 있었습니다.

**검토한 선택지**

1. CLI에서만 작업합니다.
2. 매 응답 후 수동으로 로그를 동기화합니다.
3. 세션 원본을 읽어 같은 필터링 규칙으로 자동 동기화합니다.

**결정**

- 공식 저장 스크립트의 UTF-8 입력 처리를 보완합니다.
- IDE 세션은 프로젝트 `cwd`를 기준으로 감지하는 `tools/watch_logs.py`로 자동 동기화합니다.

**근거**

- 현재 사용하는 IDE 환경을 유지하면서 수동 누락 위험을 줄일 수 있습니다.
- 원본 Codex 세션 JSONL에서 대화 이벤트를 읽으므로 별도 대화 재작성 과정이 없습니다.
- 세션 ID별 파일로 분리해 다른 세션과 혼합하지 않습니다.

**영향 및 재검토 조건**

- 감시기는 원본 세션 파일을 수정하지 않고 파생 로그만 갱신합니다.
- Codex IDE가 공식 훅을 완전히 지원하게 되면 중복 여부를 검토해 보완 감시기를 단순화할 수 있습니다.

### D-004 · 플러그인 이름을 `OpenBell Guard`로 결정

**상황**

플러그인의 문제와 사용자 가치를 짧게 전달하면서 manifest에 사용할 안정적인 ID가 필요했습니다.

**검토한 선택지**

- `market-open-guard`
- `brokerage-resilience-copilot`
- `kakaopaysec-market-guard`
- `openbell-guard`

**결정**

- 표시 이름은 `OpenBell Guard`, 플러그인 ID는 `openbell-guard`를 사용합니다.

**근거**

- 개장 시점이라는 핵심 상황을 짧게 전달합니다.
- 특정 회사 내부 제품처럼 오인될 표현을 피하면서 대상 문제를 설명합니다.
- Codex 플러그인 식별자에 적합한 소문자 kebab-case입니다.

**영향 및 재검토 조건**

- 기획서, manifest, Skill, README와 질문지에서 같은 이름을 사용합니다.
- 상표·제출 규정 또는 manifest 검증 문제가 발견되면 새 결정 기록 후 변경합니다.

### D-005 · Skill·로컬 스크립트 중심 MVP, MCP 제외

**상황**

Codex 플러그인은 Skill, MCP, 앱, hooks 등을 포함할 수 있지만 해커톤 심사자가 별도 내부 시스템이나 인증정보 없이 실행할 수 있어야 합니다.

**검토한 선택지**

1. Grafana·Prometheus MCP를 처음부터 연결합니다.
2. 설명 중심 Skill만 제공합니다.
3. Skill과 결정론적 로컬 분석 스크립트를 결합합니다.

**결정**

- MVP는 `SKILL.md`, Python 분석·마스킹 스크립트, 참고자료와 합성 fixture로 구성합니다.
- `.mcp.json`, 앱과 플러그인 내부 hooks는 MVP에서 제외합니다.

**근거**

- 별도 API 키 없이 재현할 수 있습니다.
- 통계 계산은 동일 입력에 동일 결과를 내는 코드가 더 신뢰할 수 있습니다.
- Codex는 여러 결과의 의미 종합과 보고서 작성에 집중할 수 있습니다.
- OpenAI 모범 사례는 실제 반복 수작업을 제거할 필요가 있을 때만 MCP를 추가하도록 권장합니다.

**영향 및 재검토 조건**

- 실제 조직 적용 단계에서 읽기 전용 관측 시스템 연결이 필요하고 인증·보안 모델이 확인되면 MCP를 별도 검토합니다.

### D-006 · 합성 데이터·읽기 전용·근거 분리 안전 모델 채택

**상황**

금융 로그에는 비밀정보와 개인정보가 포함될 수 있고, 공개 자료만으로 실제 장애의 상세 근본 원인을 알 수 없습니다.

**검토한 선택지**

1. 실제 운영 로그와 시스템을 전제로 설계합니다.
2. 공개 기사만 자연어로 요약합니다.
3. 합성 텔레메트리로 기술 흐름을 검증하고 실제 적용 한계를 명시합니다.

**결정**

- 합성 fixture를 사용하고 기본 동작을 읽기 전용으로 제한합니다.
- 원본 입력은 수정하지 않고 마스킹된 파생본을 사용합니다.
- 결과를 `확인된 사실`, `원인 가설`, `추가 확인 필요`로 분리합니다.
- 실제 주문, 운영 변경, 운영 주소 부하 테스트와 자동 법적 판단을 금지합니다.

**근거**

- 내부 데이터 없이도 심사자가 전체 흐름을 재현할 수 있습니다.
- 공개 근거를 넘어선 기업 내부 원인 단정을 방지합니다.
- 금융 데이터의 비밀정보 노출과 운영 시스템 부작용을 최소화합니다.

**영향 및 재검토 조건**

- 실제 운영 효과나 무장애를 주장하지 않습니다.
- 스테이징 적용은 사용자가 환경을 명시하고 데이터 보호가 확인된 경우에만 별도 검토합니다.

### D-007 · 공식 영문 복사본 대신 한국어 프로젝트 특화 가이드 유지

**상황**

`docs/`에 OpenAI 공식문서의 영문 복사본을 보관했으나 일부 UI 태그와 불필요한 내용이 포함됐고 OpenBell Guard 적용 방법은 설명하지 않았습니다.

**검토한 선택지**

1. 영문 원문 전체 복사본만 유지합니다.
2. 영문 원문과 한국어 해설본을 모두 유지합니다.
3. 공식 URL과 확인일을 남기고 로컬에는 프로젝트 특화 해설본을 유지합니다.

**결정**

- 영문 복사본은 제거하고 OpenBell Guard에 맞춘 한국어 가이드 4개를 유지합니다.
- `docs/SOURCES.md`에 공식 원문 URL과 확인일을 기록합니다.
- 스키마와 최신 CLI 동작은 공식 원문과 실제 검증기를 우선합니다.

**근거**

- 한국어 해설본이 프로젝트 구성원과 심사 준비에 더 빠르게 활용됩니다.
- 전체 원문 복사보다 필요한 문맥만 제공해 Codex 입력 부담을 줄입니다.
- 해설과 공식 요구사항을 구분하면 문서 성격과 책임 범위가 분명해집니다.

**영향 및 재검토 조건**

- 공식문서 변경 시 출처 확인일과 해당 가이드를 갱신합니다.
- 제출물에는 `docs/`를 기본적으로 포함하지 않고 README에서 필요한 공식 원문을 인용합니다.

**검토한 선택지**

3. 루트 `AGENTS.md`에 항상 필요한 규칙을 두고 상세 내용은 문서로 연결합니다.

**결정**

- 프로젝트 루트 `AGENTS.md`를 Codex의 지속 개발 지침으로 사용합니다.

**영향 및 재검토 조건**

### D-014 · 확정 채택 사항은 기획서에 즉시 반영하고 구현 전 설계 동결 점검 수행

**상황**

기업 Techlog와 초급자 가이드를 계속 추가하면 새로운 기술과 분석 방식이 발견될 수 있습니다. 모든 자료 검토가 끝날 때까지 기획서 수정을 미루면 이미 확정한 요구사항이 구현·테스트에서 빠질 수 있고, 반대로 후보 아이디어까지 매번 기획서에 넣으면 기준선이 자주 흔들리고 범위가 커질 수 있습니다.

**검토한 선택지**

1. 앞으로 수집할 자료를 모두 검토한 뒤 마지막에 기획서를 한 번 수정합니다.
2. 게시물에서 발견한 모든 기술과 아이디어를 즉시 기획서에 추가합니다.
3. `채택`이 확정되어 코드·출력·테스트·제외 범위에 영향을 주는 항목만 즉시 반영하고, 구현 직전에 전체 설계 동결 점검을 수행합니다.

**결정**

- D-010 기준으로 `채택`이 확정된 항목은 다음 기획서 갱신에서 즉시 관련 절에 반영합니다.
- `참고만` 또는 `후순위` 항목은 검토 문서와 Decisionlog에 유지하며 핵심 기획서의 필수 범위에 넣지 않습니다.
- 이미 기획서에 같은 원칙이 있으면 새 기술 이름을 중복 추가하지 않고 기존 문장을 더 명확하게 보강합니다.
- Phase 4 MVP 구현을 시작하기 전에 리뷰 문서·Decisionlog·기획서·테스트 계획을 한 번 대조해 필수 범위를 동결합니다.
- 설계 동결 뒤에는 핵심 문제 해결력과 검증 가능성이 명확히 증가하는 저비용 변경만 추가합니다.

**근거**

- 기획서는 구현과 테스트의 기준선이므로 확정한 요구사항이 늦게 반영되면 코드·README·질문지와 어긋날 위험이 있습니다.
- 후보 기술을 모두 넣지 않으면 기업 기술을 억지로 연결하는 범위 확장을 막을 수 있습니다.
- 작은 누적 반영과 구현 전 종합 점검을 결합하면 최신성, 추적성, 일정 안정성을 함께 확보할 수 있습니다.

**영향 및 재검토 조건**

- 다음 기획서 갱신에서 관측 증상과 원인 가설 분리, 완화책·근본 조치·검증·롤백 조건, 오판 방지 테스트를 명시합니다.

- 추가 자료가 현재 핵심 문제나 아키텍처를 바꿀 정도로 중요하면 별도 결정 기록 후 설계 동결 시점을 재검토합니다.

### D-015 · Techlog 채택 후보는 사용자 승인 후 기획서에 반영

**상황**

D-014는 확정 채택 사항을 늦추지 않고 누적 반영하도록 정했지만, 무엇을 확정 채택으로 볼지에 대한 사용자 검토 단계가 명시되어 있지 않았습니다. 사용자는 새 Techlog를 검토할 때 Codex가 반영 후보를 먼저 제안하고, 사용자 승인 후 기획서를 수정한 다음 다음 자료 검토로 넘어가도록 요청했습니다.

**검토한 선택지**

1. Codex가 적합하다고 판단하면 승인 없이 기획서를 자동 수정합니다.
2. 모든 자료가 모일 때까지 기획서 수정을 보류합니다.

**결정**

- 각 Techlog의 출처 범위, 문제 적합성, 구현 비용, 검증 방법과 제외 범위를 먼저 문서화합니다.
- 기획서 반영 후보가 있으면 수정할 절, 기능·출력·테스트 영향과 예상 비용을 사용자에게 보고하고 명시적 승인을 요청합니다.

- D-014의 작은 누적 반영과 구현 전 설계 동결 원칙은 유지하되, 누적 반영 앞에 사용자 승인 관문을 추가합니다.

**근거**

- 사용자 승인 관문은 기술 범위가 의도와 다르게 커지는 것을 막습니다.
- 정확한 변경 위치와 검증 영향을 먼저 제시하면 채택 판단을 비교하고 되돌리기 쉽습니다.

**영향 및 재검토 조건**

### D-016 · Python 스트리밍 입력과 크기 보호 원칙을 채택 후보로 제안

**상황**

카카오페이 정산플랫폼팀의 공식 Techlog는 수억 건의 원장 통계를 처리하며 단순 병렬 Partitioning만으로 OOM이 발생했고, Cursor 스트리밍과 Bulk Write를 결합해 개선한 사례를 공개했습니다. OpenBell Guard도 로컬 로그 파일을 입력받지만 현재 기획서에는 큰 파일의 메모리 처리 전략과 입력 한도가 명시되어 있지 않습니다.

**검토한 선택지**

1. Spring Batch·MongoDB·멀티스레드 Partitioning을 그대로 구현합니다.
2. 작은 합성 fixture만 사용하므로 대용량 입력 원칙을 모두 제외합니다.
3. Python 표준 라이브러리의 반복자 입력과 크기 보호만 채택하고 병렬화·데이터베이스 기술은 제외합니다.

**제안 결정 — 사용자 승인 대기**

- 파일 크기, 처리·거부 레코드 수와 적용한 안전 한도를 검증 결과에 기록합니다.
- 설정 한도를 넘으면 명확한 파일·줄 위치와 이유를 보고하고 안전하게 중단합니다.
- 큰 JSON 배열은 표준 라이브러리로 완전 스트리밍하지 않고 JSONL 변환을 안내합니다.
- 논리 파티션·파티션 manifest·대규모 정확 백분위수는 후순위로 둡니다.
- Spring Batch·MongoDB·멀티스레드·Bulk Write는 MVP에서 제외합니다.
- 사용자 승인 전에는 기획서를 수정하지 않습니다.

**근거**

- 반복자 파싱은 외부 의존성 없이 OOM 위험을 낮추며 로컬 파일 분석 구조에 직접 맞습니다.
- 입력 크기와 처리 건수는 분석 결과의 재현성과 실패 원인 설명에도 도움이 됩니다.
- 현재 합성 데이터 규모에서는 병렬 처리의 성능 이득을 입증할 근거가 없고 동시성·결과 취합·테스트 복잡도만 증가합니다.
- 공식 Spring Batch 문서도 현실적인 단순 작업의 성능을 먼저 측정한 뒤 복잡한 병렬화를 고려하도록 안내합니다.
- 이 글은 카카오페이 정산플랫폼팀 사례이며 카카오페이증권의 기술 스택을 증명하지 않습니다.

**영향 및 재검토 조건**

### D-017 · 대용량 대응 후보를 최소 입력 안전계약으로 축소 채택

**상황**

사용자는 D-016의 네 후보가 OpenBell Guard에 꼭 필요한지, 초기 MVP에 과한 기능이라면 덜어낼 수 있는지 재검토하고 당장 필요한 부분만 기획서에 반영하도록 요청했습니다. 현재 플러그인은 아직 `src/`가 없는 로컬 Python MVP이므로 대규모 처리 성능보다 분석 누락 방지와 예측 가능한 실패가 우선입니다.

**검토한 선택지**

1. D-016의 네 후보를 각각 독립 기능과 설정으로 모두 구현합니다.
2. 작은 합성 fixture만 사용하므로 대용량 입력 관련 요구를 전부 후순위로 둡니다.
3. 분석 신뢰성과 메모리 안전에 직접 필요한 부분만 입력 계약으로 축소하고 일반 한도 설정·파티셔닝은 후순위로 둡니다.

**결정**

- JSONL·CSV·텍스트는 처음부터 행 단위로 순차 처리합니다. 이는 별도 대용량 기능이 아니라 Python 표준 라이브러리의 기본 구현 방식으로 취급합니다.
- 파일 바이트 크기, 처리·거부 레코드 수와 오류 위치를 입력 요약에 기록합니다. 별도 대시보드나 성능 보고서는 만들지 않습니다.
- JSON 배열에만 `max_json_bytes` 사전 검사를 적용합니다. 기본값은 구현 단계의 합성 성능 fixture 측정 후 고정합니다.
- 큰 JSON 배열은 자동 변환하지 않고 전체 파싱 전에 중단한 뒤 JSONL 사용 방법만 안내합니다.
- 일반 JSONL·CSV·텍스트의 사용자 조정형 파일·레코드 한도, 처리시간·메모리 benchmark, 논리 파티션과 정확한 대규모 백분위 최적화는 후순위로 둡니다.
- Spring Batch·MongoDB·병렬 Partitioning·Bulk Write는 계속 제외합니다.
- D-016의 승인 대기 제안은 이 결정으로 종결하고 축소 채택 범위를 기획서에 반영합니다.

**근거**

- 행 단위 입력은 파일 전체를 메모리에 올리는 구현보다 단순하며 이후 다시 작성할 가능성을 줄입니다.
- 처리·거부 건수와 오류 위치가 없으면 일부 입력이 누락되어도 보고서가 전체 자료를 분석한 것처럼 보일 수 있으므로 분석 신뢰성에 직접 필요합니다.
- 일반 행 단위 입력은 메모리 사용이 입력 전체 크기에 비례하지 않으므로 초기부터 임의의 레코드 상한 설정까지 만들 필요는 없습니다.
- JSON 배열은 Python `json.load`가 전체 문서를 메모리에 올리므로 이 형식에만 명시적 바이트 경계가 필요합니다.
- 자동 변환·병렬화·파티셔닝과 대용량 정확 백분위수는 핵심 장애 분석 흐름을 검증하기 전에 추가할 가치가 입증되지 않았습니다.

**영향 및 재검토 조건**

- 기획서 8.5·9.1·9.2·10.1·13·14절과 인접 Techlog 출처를 갱신합니다.
- Phase 4에서 `max_json_bytes` 기본값을 로컬 합성 fixture로 측정하고 실제 값과 실행 환경을 검증 문서에 기록합니다.
- 정상 행과 손상 행의 처리·거부 건수, 행 단위 처리와 큰 JSON 사전 중단을 자동 테스트합니다.
- 행 단위 처리 시간이 시연 요구를 충족하지 못하거나 실제 fixture에서 자원 문제가 재현될 때만 일반 한도·파티션·백분위 메모리 전략을 재검토합니다.

- 원본은 로컬 스크립트만 읽고 Codex는 마스킹된 제한 근거와 `analysis.json`만 읽습니다. 마스킹 실패 시 원본 직접 읽기로 우회하지 않습니다.
- `analysis.json`의 claim-evidence 참조와 보고서 claim ID를 기계 검증합니다.
- manifest 이름은 `openbell-guard`로 유지하고 `src` 디렉터리명과 일치할 필요가 없음을 명시합니다. 개발 Marketplace, Codex 재시작, 발견·자동 선택·명시 호출을 검증합니다.
- 합성 데모의 예측 가능한 실패를 위해 파일·레코드·번들 고정 지원 한도를 둡니다. 이 부분은 일반 한도를 후순위로 뒀던 D-017을 대체합니다.

**근거**

- 한 대표 흐름과 세 산출물은 심사자가 입력부터 결과 검증까지 재현하기 쉽고 미구현 기능을 과장할 위험을 줄입니다.
- 파일·필드·시간대·상태·종료 코드 계약이 있어야 Golden fixture와 오류 테스트의 정답을 만들 수 있습니다.
- 통계 계산법과 최소 표본을 고정해야 동일한 입력에 같은 결과가 나오고 작은 표본의 과도한 p99 해석을 피할 수 있습니다.
- Codex가 원본을 직접 읽지 않는 경계는 “로컬 파일”이라는 표현만으로는 해결되지 않는 데이터 노출 위험을 줄입니다.
- 구조화된 claim-evidence 관계는 사실 근거 연결률을 사람이 눈으로만 확인하는 대신 자동 검사할 수 있게 합니다.
- OpenAI 공식 플러그인 문서는 manifest `name`을 안정적인 kebab-case 식별자로 설명하며 플러그인 루트 디렉터리 이름과 같아야 한다고 요구하지 않습니다.

**영향 및 재검토 조건**

- `docs/openbell-guard-research-plan.md`, `AGENTS.md`, 플러그인 제작·개요 가이드와 QA·초급자 설명 문서를 같은 계약으로 갱신합니다.
- Phase 4에서는 입력·분석 계약과 Golden fixture를 코드보다 먼저 만들며 범위를 임의로 넓히지 않습니다.
- 초기 지원 한도를 로컬 환경에서 안정적으로 처리하지 못하면 부분 성공으로 숨기지 않고 benchmark와 새 결정 기록을 근거로 한도를 낮춥니다.
- 실제 조직 적용이나 실시간 연동 요구가 확인될 때만 MCP·관측 시스템 어댑터·대규모 근사 통계를 별도 범위로 재검토합니다.

### D-019 · 지표·수식·오류 판정 단일 계약 `1.0.0` 채택

**상황**

Phase 4 구현 전에 같은 지표가 문서·세션·함수마다 다른 분모, 시간 구간, 백분위수, 반올림 또는 오류 기준으로 작성될 위험을 제거해야 했습니다. D-018에서 핵심 통계 방향을 정했지만 입력 grain, 경계 포함 여부, `null`, 다중 임계치, 오류 우선순위와 성능 한도의 합격 기준까지 하나의 문서로 통제하지는 않았습니다.

**검토한 선택지**

1. 기획서·SKILL·코드 주석과 테스트마다 필요한 정의를 반복합니다.
2. 구현 코드의 동작을 사실상 기준으로 삼고 문서는 사후 갱신합니다.
3. 사람이 읽는 버전 계약을 단일 기준으로 만들고 모든 코드·fixture·산출물이 M-ID와 issue code를 참조하게 합니다.

**결정**

- `docs/openbell-guard-metrics-validation-contract.md`를 지표·수식·입력 검증·장애 상태·오류 판정의 유일한 기준으로 채택합니다.
- 계약 버전은 `1.0.0`, 입력·출력 스키마 버전은 `1.0`으로 시작합니다.
- 시간은 `[start, end)`와 UTC 60초 버킷, 수치는 `Decimal`과 `ROUND_HALF_UP`을 사용합니다.
- M-001~M-017에 요청·오류·처리량·백분위·관측 지연·기준 비교·레코드 건수·자원 맥락·실행시간·Python 추적 메모리 정의를 부여합니다.
- 버킷과 경로 상태, `complete`·`degraded`·`fatal`, 종료 코드와 29개 issue code의 필요조건과 우선순위를 고정합니다.
- 민감정보 탐지 정규식과 마스킹 성공·실패, evidence·claim과 가설 신뢰도를 계약에 포함합니다.
- 계약에 없는 지표는 계산된 사실로 출력하지 않습니다.
- 다른 현행 문서는 수식을 복제하지 않고 계약을 링크합니다. 교육·과거 검토 문서는 구현 기준이 아니라는 안내를 표시합니다.
- Phase 4에서 제출용 계약 복사본의 버전과 SHA-256을 개발 원본과 자동 대조합니다.

**근거**

- grain, 분모, 시간 구간, 반올림과 경계 부등호 중 하나만 달라도 같은 이름의 지표가 다른 값을 만들 수 있습니다.
- M-ID와 issue code는 요구사항·함수·테스트·보고서 사이의 추적성을 제공합니다.
- `null` reason code와 오류 우선순위를 고정하면 데이터 부족과 실제 0, 입력 오류와 보안 차단을 세션마다 다르게 해석하는 일을 줄입니다.
- 공식 SLO가 없는 합성 MVP에서 정의되지 않은 수치를 계산하지 않는 것이 임의 판단보다 안전합니다.
- 문서와 제출용 복사본의 SHA-256 검사는 사람이 눈으로만 동기화를 확인하는 것보다 재현성이 높습니다.

**영향 및 재검토 조건**

- Phase 4는 입력·출력 Schema와 Golden fixture를 계약 `1.0.0`에서 먼저 파생합니다.
- 지표 의미·수식·단위·경계·상태·오류 코드 변경은 사용자 승인과 새 D-ID 없이 코드에서 먼저 바꿀 수 없습니다.
- 지원 한도 benchmark가 M-016·M-017 기준을 넘으면 측정 환경과 결과를 기록하고 구현 개선 또는 계약·지원 한도 변경을 새 결정으로 남깁니다.
- 실제 운영 적용에서 새로운 지표가 필요해도 계약에 정의·테스트를 추가하기 전에는 보고서의 계산된 사실로 사용하지 않습니다.

### D-020 · 메인 에이전트 중심의 제한적 서브에이전트 운영 모델 채택

**상황**

Phase 4부터는 플러그인 스캐폴드, 입력 검증, 민감정보 마스킹, 지표 계산, evidence·claim 구성, 테스트, 문서와 제출 패키징이 동시에 얽힙니다. 사용자는 본 구현 전에 서브에이전트를 어떻게 분배하면 효율성과 품질을 동시에 높일 수 있는지 구상하도록 요청했습니다. 이 프로젝트는 공유 작업공간에서 진행되며 로그 무결성, 지표 계약, 보안 경계와 Notion 기록까지 지켜야 하므로 무제한 병렬화는 위험합니다.

**검토한 선택지**

1. 메인 에이전트가 모든 구현과 검증을 순차적으로 수행합니다.
2. 구현 단계마다 가능한 한 많은 서브에이전트를 일반 작업자로 투입합니다.
3. 메인 에이전트가 계약·파일 소유권·통합·기록을 책임지고, 서브에이전트는 허용 파일과 완료 조건이 분명한 작은 작업 패킷만 맡깁니다.

**결정**

- 선택지 3을 채택합니다.
- 동시 작업은 메인 에이전트 1명과 서브에이전트 최대 3명으로 제한합니다.
- 서브에이전트는 계약·스키마, 입력·보안, 분석 엔진, QA·검증, 문서·제출 같은 역할 풀에서 필요한 때만 사용합니다.
- 한 파일은 한 시점에 한 에이전트만 수정합니다.
- `AGENTS.md`, 기획서, 지표 계약, Worklog, Decisionlog, Notion과 최종 통합 판단은 메인 에이전트가 관리합니다.
- 작업 브리프에는 목표, 완료 조건, 허용 파일, 참고 문서, 금지 사항, 검증 명령, 중단 조건과 반환 형식을 명시합니다.
- 계약과 내부 인터페이스를 먼저 고정하고, 입력·보안, 분석 엔진, QA처럼 파일 소유권이 겹치지 않는 부분만 병렬화합니다.
- 구현 에이전트와 최종 검증 에이전트를 가능한 한 분리합니다.
- 서브에이전트 결과는 메인 에이전트가 실제 파일과 테스트 결과로 재확인한 뒤 통합합니다.

**근거**

- 공유 작업공간에서는 같은 파일을 여러 에이전트가 동시에 수정할 때 충돌과 의도치 않은 덮어쓰기 위험이 큽니다.
- 지표 계약과 보안 경계는 구현 편의로 달라지면 제출물 전체의 정합성이 무너질 수 있으므로 메인 에이전트가 통제해야 합니다.
- 구현과 독립 QA를 분리하면 잘못된 가정이 코드와 테스트에 동시에 복제되는 위험을 줄일 수 있습니다.
- 이번 검토에서 범위가 넓은 서브에이전트 요청은 중단·시간 초과가 발생했고, 특정 문서 절과 반환 형식을 좁힌 요청은 유효한 결과를 반환했습니다. 따라서 작은 작업 패킷과 명확한 반환 형식이 필요합니다.
- Worklog, Decisionlog와 Notion은 중복 기록과 불일치를 막기 위해 단일 책임자가 갱신하는 편이 안전합니다.

**영향 및 재검토 조건**

- `docs/subagent-operating-guide.md`를 Phase 4~6의 서브에이전트 운영 기준으로 사용합니다.
- `AGENTS.md`의 완료 기준에 서브에이전트 결과를 메인 에이전트가 재확인하고 통합했다는 조건을 추가합니다.
- Phase 4는 WP-01~WP-12 작업 패킷과 G0~G7 품질 게이트에 맞춰 시작합니다.
- 작은 단일 파일 수정, 사용자 승인 판단, 지표 계약 변경, 비밀정보 대응과 최종 패키징 판단은 메인 에이전트가 직접 수행합니다.
- 서브에이전트 사용이 오히려 충돌·지연·품질 저하를 만들면 역할 수를 줄이거나 순차 작업으로 되돌립니다.

### D-021 · Phase 4를 얇은 세로 조각 중심의 순차 구현으로 시작

**상황**

사용자는 실제 구현을 시작하되 가장 작은 단위의 기능부터 순차적으로 작업하고, 서브에이전트를 통해 기획 정합성·지표 오류·목적 적합성·코드 경량성을 다각도로 검증하도록 요청했습니다. 이미 WP-01~WP-12 운영 가이드는 있었지만 실제 코드 작성 순서, 각 단계의 검증 항목과 지표 연결은 더 작은 단위로 정리되지 않았습니다.

**검토한 선택지**

- 지표 계약과 다른 수식·경계·상태 판정이 필요해지면 구현을 멈추고 사용자 승인과 새 결정 기록을 먼저 남깁니다.
- 첫 세로 조각이 지나치게 무거워지거나 실행 검증이 어려워지면 P4 단계를 더 작게 쪼개되, 검증 없는 기능 확장은 하지 않습니다.

### D-022 · 단계 완료 보고에 전체 위치·남은 작업량·트러블슈팅 기록을 포함

**상황**

사용자는 Phase 4 구현을 순차 진행하되 각 단계 완료 시 현재 단계가 어떤 태스크를 수행했는지, 전체 작업에서 어느 위치인지, 남은 태스크와 예상 작업량이 어느 정도인지 함께 보고하도록 요청했습니다. 또한 지금부터 구현 과정에서 발생하는 트러블슈팅도 기록하라고 요청했습니다.

**검토한 선택지**

1. 기존 Worklog의 검증 결과와 남은 과제 항목에만 간단히 흡수합니다.
2. 모든 문제를 Decisionlog에 기록합니다.
3. Worklog는 작업 단위 기록으로 유지하고, 실제 발생한 오류·검증 실패·환경 문제는 별도 `Troubleshootinglog.md`에 T-ID로 기록하며, 완료 보고 형식을 확장합니다.

**결정**

- 선택지 3을 채택합니다.
- Phase 4 이후 단계 완료 보고에는 현재 단계, 전체 위치, 이번 단계 태스크, 완료 기능, 변경 파일, 실행 검증, 관련 M-ID·issue code, 트러블슈팅 T-ID, 서브에이전트 검토 결과, 남은 위험, 남은 태스크와 남은 예상 작업량을 포함합니다.
- 실제로 발생한 구현 문제는 `Troubleshootinglog.md`에 T-ID로 기록합니다.
- 예상 위험이나 일반 주의사항은 T-ID로 만들지 않고 Worklog의 남은 위험에 기록합니다.
- P4-01에서 이미 발생한 PyYAML 의존성 누락, 한글 검증 인코딩 문제, Notion 검색 검증 제한과 Git 상태 확인 실패를 초기 트러블슈팅 항목으로 남깁니다.

**근거**

- 완료 보고에 전체 위치와 남은 작업량을 포함하면 사용자가 구현 진행률과 남은 부담을 쉽게 판단할 수 있습니다.
- Worklog에 모든 오류 세부를 넣으면 작업 기록이 길어져 핵심 산출물과 검증 결과를 찾기 어려워집니다.
- Decisionlog는 선택지와 정책 결정 중심이므로, 도구 오류나 환경 문제까지 모두 넣으면 결정 기록의 의미가 흐려집니다.
- T-ID를 별도로 두면 같은 문제가 재발했을 때 원인과 기존 조치를 빠르게 추적할 수 있습니다.
- 공식 검증기 의존성, 인코딩, Notion 도구 제한처럼 작은 문제가 나중에 제출 검증에서 다시 나타날 수 있으므로 별도 기록이 유용합니다.

**영향 및 재검토 조건**

- `AGENTS.md`, `docs/phase4-implementation-sequence.md`, `Troubleshootinglog.md`를 이후 구현 보고와 문제 해결 기록의 기준으로 사용합니다.
- 단계 완료 보고는 더 길어지지만, 각 보고는 요약 중심으로 유지합니다.
- 트러블슈팅 항목이 너무 세분화되어 읽기 어려워지면 동일 원인 문제를 기존 T-ID 후속 메모로 묶는 방식으로 조정합니다.

### D-023 · 토큰 효율적 트러블슈팅 하네스 채택

**상황**

사용자는 `Troubleshootinglog.md`에 기록된 실수를 모든 대화에 적용하면 재발 방지에는 도움이 되지만, 매번 긴 로그를 읽어 토큰 소모가 커지지 않을지 우려했습니다. 이어서 안전하고 토큰 효율적인 구조를 먼저 구현하라고 요청했습니다.

**검토한 선택지**

1. 매 대화마다 `Troubleshootinglog.md` 전체를 읽습니다.
2. `AGENTS.md`에 모든 트러블슈팅 내용을 복사해 항상 컨텍스트에 넣습니다.
3. `Troubleshootinglog.md`에는 빠른 색인만 추가하고, 반복 가능한 점검은 `tools/preflight_check.py`로 자동화합니다.

**결정**

- 선택지 3을 채택합니다.
- `Troubleshootinglog.md` 상단에 상황별 T-ID 빠른 색인을 둡니다.
- `AGENTS.md`에는 전체 로그를 통째로 읽지 말고 관련 T-ID만 확인하라는 짧은 규칙을 둡니다.
- 반복 가능한 환경·구조 점검은 표준 라이브러리 기반 `tools/preflight_check.py`로 수행합니다.
- preflight는 `PyYAML` 누락, Git 저장소 인식 실패처럼 이미 알려진 환경 문제를 경고로 보고하고, 플러그인 구조·UTF-8·Markdown·핵심 ID 문제는 오류로 보고합니다.

**근거**

- 긴 트러블슈팅 로그를 매번 읽는 방식은 컨텍스트와 토큰을 크게 소모합니다.
- `AGENTS.md`가 비대해지면 매 작업마다 기본 컨텍스트가 무거워지고, 실제 구현 정보가 묻힐 수 있습니다.
- 빠른 색인은 사람이 관련 T-ID만 선택하게 하고, preflight는 반복 가능한 문제를 문서 재독해 없이 기계적으로 확인하게 합니다.
- 경고와 오류를 분리하면 현재 이미 알려진 PyYAML·Git 문제 때문에 모든 구현이 멈추는 일을 피하면서도, 제출 전 해결해야 할 한계를 추적할 수 있습니다.

**영향 및 재검토 조건**

- Phase 4 이후 구현·검증 단계 시작 전에는 필요에 따라 `python tools/preflight_check.py`를 실행합니다.
- preflight 경고는 자동 실패가 아니지만 단계 완료 보고의 트러블슈팅 또는 남은 위험에 연결합니다.
- 공식 검증기 실행이 필수인 단계에서는 `PyYAML` 경고를 해결하거나 미검증 범위를 명시합니다.
- preflight 출력이 너무 길어지면 기본 출력은 더 줄이고 `--quiet` 사용을 기본으로 조정합니다.

### D-024 · GitHub 원격 저장소를 단계 완료 동기화 대상으로 채택

**상황**

사용자는 새 GitHub 저장소 `gorhkdwj/AX-KakaopaySecurities-Qualifier.git`를 만들고, 현재 프로젝트를 해당 저장소에 push하며 앞으로 각 단계가 끝날 때마다 push까지 진행하라고 요청했습니다. 기존 작업 폴더에는 `.git` 디렉터리가 있었지만 Git 저장소로 인식되지 않아 T-004로 기록된 상태였습니다.

**검토한 선택지**

1. 전체 작업 폴더를 그대로 GitHub에 push합니다.
2. 코드·문서·설정은 push하되 `logs/`, 로컬 AI·IDE 상태와 비밀 가능 파일은 제외합니다.
3. GitHub push를 하지 않고 로컬 기록만 유지합니다.

**결정**

- 선택지 2를 채택합니다.
- 로컬 저장소를 `main` 브랜치로 초기화하고 GitHub 원격 `origin`에 연결합니다.
- 단계 완료 시 Worklog·Decisionlog·Troubleshootinglog·Notion 동기화를 마친 뒤 커밋하고 원격에 push합니다.
- `logs/`는 해커톤 제출 시 로컬 패키징에 포함하되 GitHub에는 기본 push하지 않습니다.
- `.agents/`, `.claude/`, `.codex/`, `.vscode/`, Python 캐시, `.env`, 키 파일과 제출 ZIP 산출물도 기본 제외합니다.

**근거**

- GitHub 원격 저장소는 단계별 변경사항을 되돌아보고 공유하기 위한 기준점이 됩니다.
- 원본 대화 로그는 과제 제출에 필요하지만 전체 대화가 들어 있어 공개 저장소에 올리기에는 정보 노출 위험이 큽니다.
- 로컬 AI·IDE 상태와 `.env`류 파일은 재현성보다 보안·개인 환경 노출 위험이 더 큽니다.
- 단계 완료 후 push를 완료 기준에 넣으면 로컬 변경과 Notion 기록, 원격 저장소 사이의 불일치를 줄일 수 있습니다.

**영향 및 재검토 조건**

- `AGENTS.md`의 완료 기준에 Git 커밋·push 여부 보고를 추가합니다.
- 이후 단계 완료 보고에는 push 완료 여부와 실패 시 사유를 포함합니다.
- 사용자가 원격 저장소가 비공개임을 확인하고 로그까지 버전 관리하라고 별도로 지시하지 않는 한 `logs/`는 계속 Git 추적에서 제외합니다.
- 제출 패키징 단계에서는 `.gitignore`와 무관하게 로컬 `logs/` 원본을 제출 ZIP에 포함하는지 별도 검사합니다.
