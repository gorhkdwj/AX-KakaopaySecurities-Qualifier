# OpenBell Guard P4-14~P4-18 제출 전 검토 보고서

## 1. 이 보고서의 목적

이 문서는 P4-14부터 P4-18까지 구현된 OpenBell Guard의 핵심 결과물을 비전공자·초급자도 이해할 수 있도록 설명하는 제출 전 검토 보고서입니다.

목표는 두 가지입니다.

1. 사용자가 “이 플러그인이 실제로 무엇을 만들고, 무엇을 검증했는지” 직관적으로 이해할 수 있게 합니다.
2. 심사자에게 설명할 때 너무 가볍지 않도록, 산출물·검증 기준·한계를 실무 보고서 수준으로 정리합니다.

이 문서는 카카오페이증권의 실제 내부 장애 원인을 단정하지 않습니다. 현재 검증은 합성 또는 익명화된 사고 분석 번들을 대상으로 합니다.

## 2. 한눈에 보는 결론

P4-14~P4-18까지의 구현은 “분석 결과를 만들고, 그 결과가 믿을 만한지 검증하고, 여러 상황과 성능 한도에서도 흔들리지 않는지 확인하는 구간”입니다.

| 단계 | 한 줄 설명 | 현재 상태 |
|---|---|---|
| P4-14 | 기계가 검증할 수 있는 최종 분석 원장 `analysis.json` 생성 | 완료 |
| P4-15 | 분석 산출물이 규칙을 지켰는지 검사하는 `output-validation.json` 생성 | 완료 |
| P4-16 | 사람이 읽을 수 있는 `openbell-report.md` 보고서 초안 생성 | 완료 |
| P4-17 | A~H 합성 상황별 시나리오 테스트 | 완료 |
| P4-18 | 지원 한도 입력에서 실행시간·메모리 benchmark | 완료 |

현재까지 확인된 핵심 성과는 다음과 같습니다.

- 대표 합성 사고 번들을 넣으면 `analysis.json`, `openbell-report.md`, `output-validation.json`이 생성됩니다.
- 보고서의 모든 핵심 문장은 claim ID로 추적됩니다.
- 확인된 사실, 원인 가설, 판단 불가 항목이 분리됩니다.
- 합성 비밀정보는 마스킹되어 최종 산출물에 남지 않는지 검증됩니다.
- 국내장 개장 피크, 외부 중개사 장애, 불완전 데이터, 관측 지연, 임계치 경계 등 A~H 상황별 검증을 통과했습니다.
- 지원 한도 합성 입력 기준 실행시간 중앙값 34.523738초, Python 추적 메모리 최고값 194.280592MiB로 기준을 통과했습니다.

남은 작업은 P4-19입니다. P4-19는 “이제 만든 플러그인을 제출 ZIP으로 포장했을 때도 똑같이 설치·실행·검증되는지” 확인하는 단계입니다.

## 3. 초급자를 위한 전체 흐름 설명

OpenBell Guard는 사고 분석을 다음 흐름으로 처리합니다.

```text
사고 분석 번들
  ↓
입력 검사와 민감정보 마스킹
  ↓
로그·메트릭을 60초 단위로 정리
  ↓
오류율·지연시간·관측 지연·CPU·메모리 맥락 지표 계산
  ↓
서비스 경로별 상태 판정
  ↓
근거 evidence와 주장 claim 연결
  ↓
analysis.json 생성
  ↓
output-validation.json으로 자체 검증
  ↓
openbell-report.md 보고서 초안 생성
  ↓
시나리오 테스트와 benchmark로 재검증
```

비유하면, OpenBell Guard는 “장애 원인을 마음대로 추측하는 AI”가 아니라 “사고 기록을 정리해 검사표와 근거표를 만드는 분석 보조 도구”에 가깝습니다.

- `analysis.json`은 감사 가능한 원장입니다.
- `output-validation.json`은 품질 검사표입니다.
- `openbell-report.md`는 사람이 읽는 보고서 초안입니다.
- P4-17 시나리오 테스트는 여러 상황별 모의고사입니다.
- P4-18 benchmark는 큰 입력에서도 제한 시간과 메모리 안에 들어오는지 보는 체력 테스트입니다.

## 4. 주요 용어 쉽게 설명

| 용어 | 쉬운 설명 | 이 프로젝트에서의 의미 |
|---|---|---|
| 사고 번들 | 분석에 필요한 파일 묶음 | `incident.json`, `logs.jsonl`, `metrics.csv`, `service-map.json` |
| 로그 | 사건이 발생한 기록 | 요청 시각, 서비스 경로, 성공·오류 상태, 지연시간 |
| 메트릭 | 숫자로 된 상태 측정값 | CPU, 메모리, 요청 수, 오류 수 등 |
| 버킷 | 시간을 일정 간격으로 자른 칸 | 이 프로젝트에서는 60초 단위 |
| 지표 | 판단에 쓰는 계산값 | 오류율, p95 지연시간, 관측 지연 등 |
| evidence | 근거 | “이 주장을 뒷받침하는 입력 위치와 계산 결과” |
| claim | 주장 | “확인된 사실”, “원인 가설”, “판단 불가” 문장 |
| validator | 검사기 | 결과물이 규칙을 지켰는지 자동 확인하는 코드 |
| benchmark | 성능 측정 | 큰 합성 입력에서 실행시간과 메모리를 재는 절차 |

## 5. P4-14: `analysis.json`과 `sanitization-report.md`

### 5.1 무엇을 만든 단계인가

P4-14는 OpenBell Guard의 핵심 분석 결과를 `analysis.json`으로 정리한 단계입니다.

`analysis.json`은 사람이 예쁘게 읽는 보고서가 아니라, 기계가 다시 검사할 수 있는 “분석 원장”입니다. 여기에는 다음 정보가 들어갑니다.

- 어떤 사고 번들을 분석했는지
- 어떤 계약 버전과 기준으로 분석했는지
- 몇 개의 로그·메트릭 행을 처리했는지
- 어떤 서비스 경로가 영향을 받았는지
- 60초 버킷별 오류율과 상태가 무엇인지
- 어떤 근거가 어떤 claim을 뒷받침하는지
- 원본 로그 전문이나 절대경로를 내보내지 않았는지

같은 단계에서 `sanitization-report.md`도 생성됩니다. 이 파일은 민감정보 마스킹 결과를 요약합니다.

### 5.2 실제 대표 결과

대표 합성 fixture `domestic-market-open-min` 기준으로 확인한 결과는 다음과 같습니다.

| 항목 | 결과 |
|---|---:|
| accepted record count | 9 |
| bucket count | 3 |
| service path count | 1 |
| evidence count | 5 |
| claim count | 5 |
| run status | complete |
| raw excerpts emitted | false |

서비스 경로는 `market_data` 하나이며, 상태는 `degradation_observed`입니다. 다만 중요한 점은 “근본 원인을 확정하지 않았다”는 것입니다.

즉, 이 결과는 “시세 경로에서 합성 사고 구간의 저하가 관측됐다”는 판단이지, “카카오페이증권의 실제 내부 원인은 이것이다”라는 판단이 아닙니다.

### 5.3 초급자 관점에서 왜 중요한가

AI 분석에서 위험한 부분은 “그럴듯한 말”을 만들어내는 것입니다. P4-14는 이 위험을 줄이기 위해 먼저 기계가 검사할 수 있는 원장을 만듭니다.

예를 들어 보고서에 “09:00에 오류율이 높았다”고 쓰려면, `analysis.json` 안에 실제로 다음 내용이 있어야 합니다.

- 몇 시 버킷인지
- 요청 수가 몇 개인지
- 오류 수가 몇 개인지
- 오류율이 몇 퍼센트인지
- 어떤 임계치를 넘었는지
- 어떤 evidence ID가 이 내용을 뒷받침하는지

이 구조 덕분에 보고서가 단순한 글이 아니라 검증 가능한 분석 결과가 됩니다.

### 5.4 사용자가 검토할 포인트

- `analysis.json`이 너무 기술적이더라도, “근거 있는 분석 원장”이라는 역할은 적절한지 확인하십시오.
- `market_data` 저하를 실제 카카오페이증권 내부 원인처럼 표현하지 않았는지 확인하십시오.
- `raw_excerpts_emitted=false`처럼 원본 로그 전문을 내보내지 않는 안전 경계가 지켜졌는지 확인하십시오.
- claim이 확인된 사실, 원인 가설, 판단 불가로 잘 나뉘어 있는지 확인하십시오.

## 6. P4-15: `output-validation.json`

### 6.1 무엇을 만든 단계인가

P4-15는 OpenBell Guard가 만든 결과물을 다시 검사하는 단계입니다.

여기서 생성되는 `output-validation.json`은 “분석 결과가 규칙을 지켰는지 확인하는 품질 검사표”입니다.

대표 검증 항목은 다음과 같습니다.

| 검사 항목 | 의미 |
|---|---|
| `analysis_schema` | `analysis.json` 구조가 맞는지 |
| `evidence_references` | claim이 존재하는 evidence를 참조하는지 |
| `confirmed_fact_evidence` | 확인된 사실에 근거가 붙어 있는지 |
| `report_claim_refs` | 보고서 문장에 claim ID가 제대로 붙었는지 |
| `secret_residue` | 민감정보가 산출물에 남아 있지 않은지 |

### 6.2 실제 대표 결과

대표 smoke run의 `output-validation.json` 결과는 다음과 같습니다.

| 검사 항목 | 결과 |
|---|---|
| `analysis_schema` | passed |
| `evidence_references` | passed |
| `confirmed_fact_evidence` | passed |
| `report_claim_refs` | passed |
| `secret_residue` | passed |
| fatal issue count | 0 |

### 6.3 초급자 관점에서 왜 중요한가

P4-15는 “AI가 만든 보고서를 사람이 눈으로만 믿는 구조”를 피하기 위한 단계입니다.

예를 들어 보고서에 `[C-001]`이라는 claim ID가 붙어 있는데, 실제 `analysis.json`에 `C-001`이 없다면 보고서는 겉보기에는 그럴듯해도 검증 가능한 문서가 아닙니다. P4-15는 이런 문제를 자동으로 잡습니다.

또 다른 예로, `confirmed_fact`는 말 그대로 확인된 사실입니다. 따라서 반드시 evidence가 있어야 합니다. evidence 없이 확인된 사실이라고 쓰면 위험한 단정이 됩니다.

### 6.4 사용자가 검토할 포인트

- 모든 검사 항목이 `passed`인지 확인하십시오.
- 특히 `secret_residue`가 passed인지 확인하십시오.
- `report_claim_refs`가 passed인지 확인하십시오. 이 항목은 보고서와 분석 원장이 서로 연결되어 있다는 뜻입니다.
- 검증 실패를 warning으로 낮추지 않고 fatal 오류로 막는 정책이 적절한지 확인하십시오.

## 7. P4-16: `openbell-report.md`

### 7.1 무엇을 만든 단계인가

P4-16은 사람이 읽을 수 있는 Markdown 보고서 초안인 `openbell-report.md`를 만든 단계입니다.

이 보고서는 원본 로그를 다시 읽지 않습니다. 이미 검증된 `analysis.json`만 읽어서 작성됩니다.

보고서의 기본 구성은 다음과 같습니다.

- 분석 기준
- 사고 구간
- 서비스 경로 영향
- 60초 버킷 요약
- 확인된 사실
- 원인 가설
- 추가 확인 필요
- 검토 메모

### 7.2 실제 대표 보고서의 핵심 내용

대표 smoke run에서 생성된 보고서는 다음 내용을 담고 있습니다.

- fixture: `domestic-market-open-min`
- stage: `P4-18`
- run status: `complete`
- primary telemetry: `logs.jsonl`
- accepted record count: `9`
- service path: `market_data`
- service path status: `degradation_observed`
- raw excerpts emitted: `false`

60초 버킷 요약은 다음과 같습니다.

| 시각 | 구간 | 상태 | 요청 | 오류 | 오류율 |
|---|---|---|---:|---:|---:|
| 2026-06-30 08:58 KST | baseline | healthy | 2 | 0 | 0.00% |
| 2026-06-30 09:00 KST | incident | breach | 3 | 2 | 66.67% |
| 2026-06-30 09:01 KST | incident | healthy | 3 | 1 | 33.33% |

확인된 사실과 가설은 claim ID로 끝납니다.

- `[C-001]`: 09:00 KST `market_data` 버킷이 오류율 66.67%로 임계치 50.00%를 초과했습니다.
- `[C-002]`: 09:01 KST 버킷은 오류율 33.33%로 임계치를 넘지 않았습니다.
- `[C-003]`: 09:00 KST의 CPU 맥락 지표가 기록됐지만, 메모리 값은 판단 불가입니다.
- `[C-004]`: `market_data` 저하는 사고 구간 영향 가설로는 타당하지만 근본 원인을 증명하지는 않습니다.
- `[C-005]`: 현재 번들만으로는 근본 원인을 확정할 수 없습니다.

### 7.3 초급자 관점에서 왜 중요한가

이 보고서는 사용자가 심사자에게 보여주기 쉬운 형태입니다. `analysis.json`은 기계가 읽기 좋은 구조이고, `openbell-report.md`는 사람이 읽기 좋은 구조입니다.

다만 이 보고서는 “완성된 공식 장애 보고서”가 아니라 “검토용 초안”입니다. 실제 고객 공지, 보상 여부, 법적 판단, 운영 조치는 사람이 검토해야 합니다.

이 경계는 매우 중요합니다. 금융 서비스 맥락에서는 자동 분석 결과가 곧바로 공식 입장처럼 보이면 위험할 수 있기 때문입니다.

### 7.4 사용자가 검토할 포인트

- 보고서가 너무 개발자 중심이어서 이해하기 어렵지는 않은지 확인하십시오.
- “원인 가설”이 “근본 원인 확정”처럼 읽히지 않는지 확인하십시오.
- `[C-001]` 같은 claim ID가 보고서 문장 끝에 자연스럽게 붙어 있는지 확인하십시오.
- `human review required` 성격이 명확히 드러나는지 확인하십시오.

## 8. P4-17: A~H 합성 시나리오 검증

### 8.1 무엇을 검증한 단계인가

P4-17은 “대표 예시 하나만 되는 플러그인”이 아니라 여러 상황에서도 안전하게 동작하는지 검증한 단계입니다.

여기서 A~H 시나리오는 일종의 모의고사입니다.

| 시나리오 | 검증 목적 |
|---|---|
| A | 국내장 개장 피크에서 일부 경로만 느린 상황 |
| B | 해외 중개사 같은 외부 의존성 장애 |
| C | 불완전 데이터 |
| D | 비밀정보 포함 입력 |
| E | 서비스는 정상인데 관측 지연만 큰 상황 |
| F | `DB connection timeout` 같은 메시지를 근본 원인으로 단정하지 않는지 |
| G | 임계치 경계값과 2개 연속 breach·healthy 규칙 |
| H | 손상 입력과 파일 한도 초과 |

### 8.2 초급자 관점에서 왜 중요한가

현실의 장애 분석은 항상 깔끔하지 않습니다.

- 어떤 경로는 느리지만 주문 경로는 정상일 수 있습니다.
- 외부 중개사 문제일 수도 있고 내부 문제일 수도 있습니다.
- 데이터가 불완전할 수 있습니다.
- 로그 메시지에 `DB timeout`이라고 쓰여 있어도 실제 원인이 DB라고 확정할 수는 없습니다.
- 관측 시스템이 늦게 수집했을 뿐 서비스 자체는 정상일 수 있습니다.

P4-17은 이런 상황에서 OpenBell Guard가 과장하거나 단정하지 않는지 확인합니다.

### 8.3 검증 결과의 의미

P4-17은 다음을 확인했습니다.

- 부분 장애를 전체 주문 장애로 과장하지 않습니다.
- 외부 의존성과 내부 정상 경로를 분리합니다.
- 데이터가 부족하면 판단 불가 또는 unknown으로 남깁니다.
- 합성 비밀정보 원값이 최종 산출물에 남지 않습니다.
- 관측 지연만으로 서비스 장애를 단정하지 않습니다.
- 로그 메시지 하나만 보고 DB·JVM 근본 원인을 확정하지 않습니다.
- 임계치 `>` 비교와 연속 bucket 규칙이 계약대로 동작합니다.
- 손상 입력과 한도 초과 입력을 서로 다른 방식으로 처리합니다.

### 8.4 사용자가 검토할 포인트

- 시나리오 A~H가 카카오페이증권 문제 배경과 충분히 연결되는지 확인하십시오.
- “합성 데이터”라는 표현이 명확한지 확인하십시오.
- 실제 운영 원인을 검증했다고 오해할 문장이 없는지 확인하십시오.
- 심사자에게 보여줄 때 가장 설득력 있는 시나리오가 무엇인지 표시해두는 것이 좋습니다.

## 9. P4-18: 성능·회귀 benchmark

### 9.1 무엇을 검증한 단계인가

P4-18은 OpenBell Guard가 지원 한도 크기의 합성 입력에서도 제한 시간과 메모리 기준 안에서 실행되는지 확인한 단계입니다.

검증한 지표는 두 가지입니다.

| 지표 | 의미 |
|---|---|
| M-016 | 결정론적 파이프라인 실행시간 중앙값 |
| M-017 | Python 추적 메모리 최고값 |

여기서 “결정론적”이라는 말은 같은 입력을 넣으면 같은 규칙으로 같은 결과를 내는 분석 과정을 뜻합니다. AI가 매번 다르게 추측하는 부분이 아니라, 코드로 고정된 계산 구간입니다.

### 9.2 실제 benchmark 결과

기본 benchmark 입력은 다음과 같습니다.

| 입력 | 크기 |
|---|---:|
| `logs.jsonl` | 100,000행 |
| `metrics.csv` | 50,000행 |

측정 방식은 1회 준비 실행 후 5회 측정입니다.

| 항목 | 결과 | 기준 |
|---|---:|---:|
| M-016 실행시간 중앙값 | 34.523738초 | 60초 이하 |
| M-017 Python 추적 메모리 최고값 | 194.280592MiB | 512MiB 이하 |
| 측정 run exit code | 모두 0 | 모두 0 |

따라서 P4-18 기준은 통과했습니다.

### 9.3 P4-18에서 발견한 개선점

첫 benchmark 시도는 도구 호출 제한 180초 안에 끝나지 않았습니다. 처음에는 성능이 부족한 것처럼 보일 수 있었지만, 확인 결과 단일 run 자체가 60초를 넘은 문제는 아니었습니다.

더 중요한 발견은 `source_locations`였습니다.

기존에는 각 bucket의 근거 행 위치를 전부 저장했습니다.

예를 들어 많은 행이 있으면 이런 식입니다.

```text
logs.jsonl:L1, logs.jsonl:L2, logs.jsonl:L3, ...
```

이 방식은 소규모 입력에서는 문제가 없지만, 100,000행 입력에서는 산출물 파일이 수 MB까지 커집니다.

그래서 P4-18에서 다음처럼 압축했습니다.

```text
logs.jsonl:L1-L99660
```

이 방식은 근거 위치를 잃지 않으면서도 파일 크기를 크게 줄입니다.

대표 출력 크기 변화는 다음과 같습니다.

| 파일 | 개선 전 관측 크기 | 개선 후 크기 |
|---|---:|---:|
| `bucket-summary.json` | 4,531,773 bytes | 4,689 bytes |
| `metric-summary.json` | 4,563,393 bytes | 36,309 bytes |
| `state-summary.json` | 2,992,629 bytes | 4,082 bytes |
| `analysis.json` | 2,943,358 bytes | 44,352 bytes |

### 9.4 초급자 관점에서 왜 중요한가

성능 검증은 단순히 “빠르다”를 말하기 위한 것이 아닙니다.

실무적으로는 다음을 확인하는 과정입니다.

- 입력이 커져도 프로그램이 멈추지 않는지
- 산출물이 불필요하게 커지지 않는지
- 메모리를 과하게 쓰지 않는지
- 성능 한계를 과장해서 말하지 않는지

P4-18은 이 네 가지를 확인했고, 그 과정에서 실제로 산출물 경량화 개선까지 수행했습니다.

### 9.5 사용자가 검토할 포인트

- benchmark 결과를 운영환경 성능 보장처럼 표현하지 않았는지 확인하십시오.
- M-017이 전체 프로세스 메모리가 아니라 Python `tracemalloc` 기준이라는 한계가 명확한지 확인하십시오.
- “지원 한도 합성 입력”이라는 표현이 충분히 명확한지 확인하십시오.
- 산출물 크기 개선이 기능 축소가 아니라 근거 위치 표현 방식 개선으로 설명되는지 확인하십시오.

## 10. P4-14~P4-18 전체 정합성 점검

P4-14~P4-18의 관계는 다음과 같습니다.

이 흐름은 제출물 관점에서 강점이 있습니다.

- 비밀정보와 원본 로그 전문을 직접 노출하지 않도록 설계했습니다.
- 상황별 검증과 성능 검증을 모두 수행했습니다.

P4-19는 새로운 분석 기능을 많이 추가하는 단계가 아니라 “제출 가능한 형태로 포장하고, 포장된 상태에서도 재현되는지 확인하는 단계”입니다.

## 12. 사용자가 먼저 검토하면 좋은 순서

초급자 관점에서는 다음 순서로 읽는 것이 좋습니다.

## 13. 최종 의견

P4-14~P4-18까지의 결과는 P4-19 패키징 전에 한 번 검토할 가치가 충분합니다.

특히 P4-16 보고서, P4-17 시나리오 검증, P4-18 benchmark는 심사자에게 직접 설명될 가능성이 높으므로 표현을 다듬는 것이 좋습니다.

다만 현재 구조는 방향이 크게 틀어진 상태가 아니라, “제출 전 독자가 이해하기 쉽게 정리하고 표현을 다듬을 단계”에 가깝습니다.

따라서 다음 순서는 다음과 같이 권장합니다.

1. 이 보고서를 사용자가 먼저 읽습니다.
2. 이해가 어렵거나 어색한 표현을 표시합니다.
3. 필요한 문구와 산출물 설명을 보완합니다.
4. P4-19 패키징 검증으로 넘어갑니다.
