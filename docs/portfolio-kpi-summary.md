# OpenBell Guard 포트폴리오 KPI 요약

> 목적: 이 문서는 OpenBell Guard 프로젝트를 포트폴리오로 설명할 때 사용할 핵심성과와 KPI를 한곳에 모은 요약본입니다.
>
> 주의: 아래 KPI는 실제 카카오페이증권 운영 환경 성과가 아니라, 공개 자료 기반 문제 정의와 로컬 합성 데이터·제출 패키지 검증 기준의 프로젝트 성과입니다.

## 1. 포트폴리오 핵심 한 줄

공개 자료로 확인 가능한 금융 서비스 지연 문제를 정의하고, 국내장 개장 직후 장애 상황을 가정한 합성 로그·메트릭 번들을 분석해 서비스 경로별 상태, 근거, 원인 가설, 추가 확인 필요 사항을 보고하는 Codex 플러그인 MVP를 구현했습니다.

## 2. 핵심성과 요약

| 성과 영역 | 핵심성과 | 포트폴리오에서 보여줄 의미 |
|---|---|---|

| 범위 통제 | 대규모 백엔드 인프라 대신 Skill·로컬 Python·합성 데이터 중심 MVP로 축소 | 제한된 시간과 역량 안에서 완성 가능한 현실적 범위를 설계했음을 보여줍니다. |

| 안전성 | 원본 입력 수정 금지, 민감정보 탐지·마스킹, 원인 단정 방지 | 금융 도메인에서 중요한 보안·책임 경계를 고려했음을 보여줍니다. |
| 프로젝트 운영 | Worklog, Decisionlog, Troubleshootinglog로 작업·판단·문제 해결 기록 | AI 협업 과정을 통제하고 재현 가능한 방식으로 관리했음을 보여줍니다. |

## 4. 포트폴리오에서 강조할 대표 KPI 5개

### 4.1 자동 테스트 68개 통과

- 의미: 분석 로직, 입력 검증, 출력 검증, 패키징 관련 테스트가 한 번에 통과했다는 뜻입니다.
- 포트폴리오 해석: “아이디어 구현”이 아니라 “검증 가능한 소프트웨어 산출물”로 관리했다는 근거가 됩니다.
- 표현 예시: “자동 테스트 68개를 통과시켜 분석 로직과 제출 구조의 회귀 위험을 줄였습니다.”
- 근거: [P4-19 패키징 보고서](./p4-19-packaging-report.md)

### 4.2 80,000행 합성 데이터 분석 완료

- 의미: 작은 예제뿐 아니라 로그 60,000행, 메트릭 20,000행을 합친 80,000행 규모의 현실형 합성 시나리오를 처리했습니다.
- 포트폴리오 해석: 데이터 분석가 관점에서 입력 규모, 노이즈, 정상·장애·회복 구간을 고려해 검증했다는 점을 보여줍니다.
- 표현 예시: “80,000행 합성 로그·메트릭으로 개장 직후 장애·정상·회복 구간을 검증했습니다.”
- 근거: [case-002 보고서](./manual-test-reports/case-002-large-scenario.md)

### 4.3 실행시간 기준 통과

- 결과: 실행시간 중앙값 34.523738초
- 기준: M-016 기준 60초 이하
- 의미: 대규모 합성 입력을 분석해도 사전에 정의한 실행시간 기준 안에 들어왔습니다.
- 포트폴리오 해석: “성능이 좋다”라고 막연히 말하지 않고, 측정 기준과 결과값을 함께 제시할 수 있습니다.
- 주의: 이 값은 로컬 합성 데이터와 현재 개발 환경 기준입니다. 실제 운영 인프라 성능 보장을 뜻하지 않습니다.
- 근거: [P4-18 benchmark 보고서](./p4-18-benchmark-report.md)

### 4.4 메모리 기준 통과

- 결과: Python 추적 메모리 최고값 194.280592MiB
- 기준: M-017 기준 512MiB 이하
- 의미: 대규모 입력에서도 Python이 추적한 메모리 사용량이 기준 이하로 유지되었습니다.
- 포트폴리오 해석: 결과 파일과 근거 위치를 압축해 불필요하게 무거운 출력 구조를 피했다는 점을 설명할 수 있습니다.
- 주의: M-017은 Python 표준 라이브러리 `tracemalloc` 기준이며, OS 전체 메모리나 네이티브 메모리를 모두 보장하지 않습니다.
- 근거: [P4-18 benchmark 보고서](./p4-18-benchmark-report.md)

### 4.5 판단·작업·문제 해결 기록 체계

- 결과: Worklog W-001~W-092, Decisionlog D-001~D-040, Troubleshootinglog T-000~T-027
- 의미: 단순 산출물만 만든 것이 아니라, 왜 그렇게 만들었는지와 어떤 문제가 있었는지를 추적 가능하게 남겼습니다.
- 포트폴리오 해석: AI와 협업하더라도 판단을 위임하지 않고, 범위 통제·검증·수정 이력을 관리했다는 점을 보여줍니다.
- 근거: [Worklog](../Worklog.md), [Decisionlog](../Decisionlog.md), [Troubleshootinglog](../Troubleshootinglog.md)

## 5. KPI별 포트폴리오 해석

| KPI 유형 | 보여주는 역량 | 설명할 때의 핵심 |
|---|---|---|
| 문제 정의 KPI | 문제 발견력 | 공개 자료로 확인 가능한 기업 문제를 선정했고, 내부 정보를 안다고 단정하지 않았습니다. |
| 구현 KPI | 실행력 | Codex 플러그인 manifest, Skill, Python CLI, 테스트 구조를 실제 파일로 구성했습니다. |
| 검증 KPI | 데이터 분석 품질 관리 | 지표·수식·오류 기준을 먼저 정의하고, 합성 데이터로 정상·장애·예외 상황을 확인했습니다. |
| 안전 KPI | 금융 도메인 리스크 감각 | 민감정보, 고객 데이터, 실제 주문 조작, 근본 원인 단정의 위험을 경계했습니다. |
| 운영 KPI | 협업·프로젝트 관리 | Worklog, Decisionlog, Troubleshootinglog, Notion 기록으로 프로젝트 맥락을 유지했습니다. |

## 6. 포트폴리오 문장 예시

### 짧은 버전

OpenBell Guard는 카카오페이증권의 개장 시점 서비스 지연 문제를 공개 자료로 정의하고, 80,000행 합성 로그·메트릭을 분석해 서비스 경로별 장애 여부, 근거, 원인 가설과 추가 확인 필요 사항을 보고하는 Codex 플러그인 MVP입니다.

### 자세한 버전

이 프로젝트에서는 카카오페이증권의 개장 시점 서비스 지연과 외부 의존성 분석 문제를 공개 자료 기반으로 정의했습니다. 구현은 백엔드 인프라 전체를 재현하는 대신, Codex Skill과 로컬 Python 분석기로 범위를 축소했습니다. 분석기는 합성 로그·메트릭을 입력받아 서비스 경로별 상태를 판정하고, 확인된 사실·원인 가설·추가 확인 필요 사항을 분리해 보고합니다. 최종 산출물은 자동 테스트 68개, 80,000행 합성 데이터 시나리오, 실행시간·메모리 benchmark, 제출 ZIP 구조 검증으로 확인했습니다.

### 데이터 분석가 포트폴리오 관점

이 프로젝트의 핵심은 “모델을 복잡하게 만든 것”이 아니라, 문제 정의-지표 계약-데이터 생성-검증-보고서화의 흐름을 끝까지 닫은 점입니다. 특히 원인 단정 금지, 민감정보 마스킹, 서비스 경로 분리, 추가 확인 데이터 명시 같은 규칙은 금융 데이터 분석에서 필요한 책임 있는 분석 태도를 보여줍니다.

## 8. 피해야 할 표현

| 피해야 할 표현 | 이유 | 대체 표현 |
|---|---|---|
| 카카오페이증권 실제 장애를 해결했습니다 | 실제 운영 적용·검증이 아닙니다. | 공개 자료 기반 문제를 합성 데이터로 재현해 분석 MVP를 구현했습니다. |
| 실제 운영 성능을 보장합니다 | benchmark는 로컬 합성 데이터 기준입니다. | 로컬 합성 benchmark에서 사전 기준을 통과했습니다. |
| 근본 원인을 밝혀냈습니다 | 내부 trace, 외부 상태 feed, 의존성 메트릭이 없으면 단정할 수 없습니다. | 가능한 원인 가설과 추가 확인 필요 데이터를 분리해 제시했습니다. |
| 고객 데이터로 검증했습니다 | 실제 고객 데이터는 사용하지 않았습니다. | 합성 로그·메트릭으로 정상·장애·예외 상황을 검증했습니다. |
| 실제 금융 주문 흐름을 제어합니다 | 플러그인은 읽기 전용 분석 도구입니다. | 주문·계좌 변경 없이 로그와 메트릭을 분석하는 보조 도구입니다. |

## 10. 한계와 정직한 설명

- 실제 카카오페이증권 내부 로그, trace, 인프라 메트릭에는 접근하지 않았습니다.
- 모든 검증은 공개 자료 기반 문제 정의와 로컬 합성 데이터 기준입니다.
- benchmark는 현재 개발 환경과 Python `tracemalloc` 기준입니다.
- 실제 운영 적용을 위해서는 내부 로그 스키마, trace-level causal chain, 외부 의존성 상태 feed, 실제 SLO 기준이 추가로 필요합니다.
- 따라서 이 프로젝트의 성과는 “실제 운영 장애를 해결했다”가 아니라, “운영 장애 분석을 보조할 수 있는 Codex 플러그인 MVP와 검증 가능한 분석 체계를 구축했다”로 설명하는 것이 정확합니다.
