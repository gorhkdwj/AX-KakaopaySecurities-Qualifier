# 수동 테스트 보고서: case-002-large-scenario

## 1. 결론

OpenBell Guard는 80,000행 규모의 현실형 합성 데이터에서 기대한 판정과 일치했습니다.

- 입력 규모: `logs.jsonl` 60,000행, `metrics.csv` 20,000행
- 분석 결과: `market_data`, `watchlist_info`는 `outage_detected`
- 분석 결과: `order_execution`은 `healthy`
- ground truth 대조: 3개 서비스 경로 모두 기대 상태, 장애 시작, 회복 시각 일치
- 출력 validator: `passed`
- 민감정보 원문 잔존 검색: 결과 폴더 기준 히트 없음

이 테스트는 실제 카카오페이증권 운영 로그가 아니라, OpenBell Guard의 로컬 분석 파이프라인이 복잡한 합성 상황에서도 안정적으로 동작하는지 확인하기 위한 통제된 더미데이터 검증입니다.

## 2. 이 테스트가 필요한 이유

`case-001`은 사람이 흐름을 이해하기 쉬운 작은 예제였습니다. 하지만 작은 데이터만으로는 다음 질문에 충분히 답하기 어렵습니다.

- 정상 경로에 소량의 오류가 있어도 장애로 과장하지 않는가?
- 장애 경로 안에도 성공 요청이 섞여 있을 때 장애를 놓치지 않는가?
- 임계값 근처의 경계 상황을 안정적으로 처리하는가?
- 시세·관심종목 경로의 장애와 주문 경로의 정상을 계속 분리하는가?
- 대량 로그 안에 가짜 민감정보 패턴이 드물게 섞여도 결과물에 원문이 남지 않는가?

그래서 `case-002`는 행 수만 늘리지 않고, 정상·장애·회복·경계·노이즈를 함께 넣은 대규모 합성 시나리오로 구성했습니다.

## 8. 처음 보는 사용자를 위한 해석 가이드

case-002는 “큰 데이터에서도 OpenBell Guard가 흔들리지 않는가”를 확인하기 위한 대표 예제입니다. case-001이 작은 장난감 예제라면, case-002는 실제 운영 로그에 가까운 복잡성을 넣은 통제된 합성 시나리오입니다.

이 케이스의 상황을 말로 풀면 다음과 같습니다.

> 국내장 개장 직후 10분 동안 시세와 관심종목·정보 경로에 오류와 지연이 집중적으로 나타났습니다. 주문·체결 경로에도 약간의 오류와 느린 응답은 있었지만, 장애 기준을 넘지는 않았습니다. OpenBell Guard는 이 차이를 구분해 시세·관심종목 경로만 장애로 감지하고 주문 경로는 정상으로 남겼습니다.

이 테스트에서 보는 핵심 질문은 다섯 가지입니다.

1. 로그가 80,000행으로 커져도 실행이 완료되는가?
2. 정상 경로에 소량의 오류가 있어도 장애로 과장하지 않는가?
3. 장애 경로에 성공 요청이 섞여 있어도 장애를 놓치지 않는가?
4. 임계값과 정확히 같은 경계 상황을 올바르게 처리하는가?
5. 원인 가설을 제시하되 근본 원인을 성급하게 단정하지 않는가?

이번 결과는 다섯 질문 모두에 대해 긍정적인 검증 근거를 제공합니다.

## 9. 대표 출력 인용과 읽는 법

아래는 실제 실행 요약에서 확인한 핵심 값입니다.

```text
run_status = report_validated
accepted logs.jsonl rows = 60,000
accepted metrics.csv rows = 20,000
total accepted rows = 80,000
bucket_count = 60
bucket_state_counts = breach 7, healthy 53
path_status_counts = outage_detected 2, healthy 1
claim_count = 63
evidence_count = 72
total_redactions = 20
raw_excerpts_emitted = false
```

이 값을 처음 보는 사용자는 다음처럼 해석하면 됩니다.

- `report_validated`: 보고서 초안까지 만들어졌고, 출력 검증도 통과했습니다.
- `total accepted rows=80,000`: 입력 데이터 80,000건이 분석 구간 안에서 정상 수용됐습니다.
- `bucket_count=60`: 3개 서비스 경로 × 20개 60초 bucket입니다.
- `breach 7`: 전체 60개 bucket 중 7개가 기준 초과였습니다.
- `outage_detected 2`: 3개 서비스 경로 중 2개에서 장애가 감지됐습니다.
- `healthy 1`: 나머지 1개 서비스 경로, 즉 주문·체결 경로는 정상입니다.
- `total_redactions=20`: 가짜 민감정보 패턴 20건이 마스킹됐습니다.
- `raw_excerpts_emitted=false`: 원본 로그 전문을 보고서에 그대로 싣지 않았습니다.

즉, 이 출력은 “대규모 입력을 처리했고, 일부 경로의 장애를 감지했고, 주문 경로는 정상으로 분리했고, 민감정보 보호와 출력 검증도 통과했다”는 의미입니다.

## 10. 시간대별 결과를 어떻게 읽어야 하는가

### 10.1 `market_data`: 시세 경로

`market_data`는 시세 조회와 관련된 경로입니다. 기준은 오류율 8% 초과, p95 지연 900ms 초과, p95 수집 지연 1500ms 초과입니다.

| 시간 KST | 요청 수 | 오류 수 | 오류율 | p95 지연 | p95 수집 지연 | 판정 |
| --- | ---: | ---: | ---: | ---: | ---: | --- |
| 09:00 | 1,000 | 80 | 8.0% | 258ms | 277ms | 정상 |
| 09:01 | 1,000 | 120 | 12.0% | 1468ms | 2570ms | 기준 초과 |
| 09:02 | 1,000 | 150 | 15.0% | 1488ms | 2649ms | 기준 초과 |
| 09:03 | 1,000 | 120 | 12.0% | 1392ms | 2592ms | 기준 초과 |
| 09:04 | 1,000 | 100 | 10.0% | 1330ms | 2536ms | 기준 초과 |
| 09:05 | 1,000 | 50 | 5.0% | 260ms | 276ms | 회복 |

여기서 09:00이 중요합니다. 오류율이 정확히 8.0%인데도 정상입니다. 현재 판정 기준은 `8% 이상`이 아니라 `8% 초과`입니다. 따라서 8.0%는 정상이고, 12.0%부터 기준 초과입니다.

이 경계값 처리는 플러그인의 안정성을 보여줍니다. 임계값을 `>`로 볼지 `>=`로 볼지에 따라 장애 시작 시각이 달라질 수 있기 때문입니다.

### 10.2 `watchlist_info`: 관심종목·정보 경로

`watchlist_info`는 관심종목, 종목 정보, 포트폴리오 스냅샷처럼 사용자가 화면에서 정보를 확인하는 경로입니다. 기준은 오류율 8% 초과, p95 지연 1000ms 초과, p95 수집 지연 1600ms 초과입니다.

| 시간 KST | 요청 수 | 오류 수 | 오류율 | p95 지연 | p95 수집 지연 | 판정 |
| --- | ---: | ---: | ---: | ---: | ---: | --- |
| 09:00 | 1,000 | 50 | 5.0% | 316ms | 349ms | 정상 |
| 09:01 | 1,000 | 110 | 11.0% | 1553ms | 2525ms | 기준 초과 |
| 09:02 | 1,000 | 120 | 12.0% | 1590ms | 2605ms | 기준 초과 |
| 09:03 | 1,000 | 90 | 9.0% | 1513ms | 2227ms | 기준 초과 |
| 09:04 | 1,000 | 60 | 6.0% | 318ms | 350ms | 회복 |

`watchlist_info`는 09:01~09:03에 장애 신호가 나타났고, 09:04에 회복했습니다. `market_data`보다 한 분 빠르게 회복한 점도 중요합니다. 두 경로가 같은 시간에 나빠졌더라도 회복 시점이 다르므로, 내부적으로는 별도 경로로 추적하는 것이 맞습니다.

### 10.3 `order_execution`: 주문·체결 경로

`order_execution`은 주문 요청과 체결 관련 경로입니다. 이 경로는 일부 오류가 있었지만 기준을 넘지 않았습니다.

| 시간 KST | 요청 수 | 오류 수 | 오류율 | p95 지연 | 판정 |
| --- | ---: | ---: | ---: | ---: | --- |
| 09:00 | 1,000 | 8 | 0.8% | 407ms | 정상 |
| 09:01 | 1,000 | 12 | 1.2% | 407ms | 정상 |
| 09:02 | 1,000 | 25 | 2.5% | 414ms | 정상 |
| 09:03 | 1,000 | 15 | 1.5% | 411ms | 정상 |
| 09:04 | 1,000 | 12 | 1.2% | 403ms | 정상 |

여기서 핵심은 “오류가 0건이라서 정상”이 아니라는 점입니다. 정상 경로에도 소량의 오류와 지연은 있습니다. 하지만 기준을 넘지 않았기 때문에 장애로 과장하지 않습니다. 이 점이 실제 운영 로그에 가까운 검증입니다.

## 11. 원인 가설과 추가 확인 필요를 읽는 법

자동 생성 보고서의 원인 가설은 다음 취지입니다.

```text
market_data와 watchlist_info의 성능 저하는 사고 구간에 실제 영향을 준 것으로 볼 만한 타당한 가설이다.
하지만 현재 bundle만으로는 근본 원인을 증명하지 못한다.
```

이 문장은 다음처럼 나누어 읽어야 합니다.

| 표현 | 의미 |
| --- | --- |
| 성능 저하가 타당한 가설 | 09:01 이후 오류율·지연·수집 지연이 실제로 기준을 넘었으므로 영향 발생은 말할 수 있음 |
| 신뢰도 medium | 장애 신호는 분명하지만, 원인 확정에는 증거가 부족함 |
| 근본 원인은 증명하지 못함 | 내부 서버, 외부 거래소, 시세 공급자, 네트워크 중 무엇이 원인인지는 아직 모름 |
| 추가 확인 필요 | downstream 지표, trace, 외부 상태 데이터가 더 필요함 |

사용자가 이 결과를 보고 바로 하면 안 되는 말은 다음과 같습니다.

- “외부 거래소 장애가 원인입니다.”
- “주문 장애가 발생했습니다.”
- “내부 서버 과부하로 확정됐습니다.”
- “보상 대상 여부가 확정됐습니다.”

반대로 안전한 표현은 다음과 같습니다.

- “09:01~09:04 KST 시세 경로에서 오류율과 지연이 기준을 초과했습니다.”
- “09:01~09:03 KST 관심종목·정보 경로에서 오류율과 지연이 기준을 초과했습니다.”
- “주문·체결 경로는 현재 기준상 장애로 판정되지 않았습니다.”
- “근본 원인 확정을 위해 하위 의존성 지표, trace, 외부 상태 데이터 확인이 필요합니다.”

## 12. 플러그인 사용자가 취해야 할 조치

case-002와 비슷한 결과가 실제 운영 분석에서 나왔다면 사용자는 다음 순서로 움직이면 됩니다.

1. 영향 범위 보고
   - 시세 경로와 관심종목·정보 경로를 우선 영향 범위로 표시합니다.
   - 주문·체결 경로는 정상으로 분리합니다.

2. 시간대 고정
   - `market_data`: 09:01~09:04 KST 집중 조사
   - `watchlist_info`: 09:01~09:03 KST 집중 조사
   - 회복 시각은 각각 09:05, 09:04 KST로 기록합니다.

3. 원인 조사 요청
   - quote API, quote cache, provider adapter 같은 시세 관련 하위 서비스의 오류율과 지연을 확인합니다.
   - watchlist API, instrument info, portfolio snapshot 관련 지표를 확인합니다.
   - 요청 trace로 어느 구간에서 시간이 늘어났는지 확인합니다.
   - 거래소나 외부 시세 공급 상태 페이지·공지·연동 로그를 별도로 확인합니다.

4. 고객 공지 초안 작성
   - “일부 시세 및 관심종목 정보 조회 지연이 감지되었습니다”처럼 영향 범위를 좁혀 표현합니다.
   - 주문 장애, 외부 장애, 보상 여부는 사람 검토 전 확정하지 않습니다.

5. 사후 보고서 작성
   - OpenBell Guard 보고서는 초안으로 사용합니다.
   - 최종 사후 보고서에는 사람이 확인한 추가 근거와 재발 방지 조치를 별도로 추가합니다.

## 13. 이 케이스가 보여주는 OpenBell Guard의 핵심 가치

case-002는 OpenBell Guard가 단순히 “오류가 많다”만 말하는 도구가 아니라는 점을 보여줍니다.

| 가치 | case-002에서 확인한 내용 |
| --- | --- |
| 대규모 입력 처리 | 80,000행 분석 완료 |
| 서비스 경로 분리 | 시세·관심종목 장애와 주문 정상 분리 |
| 경계값 안정성 | 오류율 8.0%는 정상, 8% 초과부터 장애 |
| 과잉 탐지 방지 | 주문 경로의 소량 오류를 장애로 과장하지 않음 |
| 회복 시점 분리 | `market_data`와 `watchlist_info`의 회복 시각을 별도 산출 |
| 근거 추적 | claim 63개, evidence 72개 생성 |
| 안전한 출력 | 원본 로그 전문 미출력, 민감정보 20건 마스킹 |

심사자나 포트폴리오 독자에게는 이 케이스가 가장 설득력 있는 대표 예제입니다. 작은 case-001로 개념을 이해하고, case-002로 “복잡한 현실형 데이터에서도 같은 원칙이 유지된다”는 점을 확인할 수 있습니다.

## 16. 실행 중 발생한 문제와 조치

첫 실행에서는 `service-map.json` 검증 오류가 발생했습니다.

- 오류: `INP009_SERVICE_MAP`
- 원인: 외부 피드처럼 보이게 하려고 `synthetic-exchange-feed`를 dependency에 넣었으나, 현재 service map 계약은 dependencies가 같은 파일 안의 서비스만 참조해야 합니다.
- 조치: 해당 외부 dependency 참조를 제거하고, 외부 의존성 성격은 `dependency_type=exchange`와 로그 메시지 패턴으로 표현했습니다.
- 결과: 재생성 후 CLI 실행과 validator가 통과했습니다.

또한 ground truth 대조용 임시 Python 명령을 처음에 Bash heredoc 형태로 실행해 PowerShell 문법 오류가 발생했습니다. 같은 내용을 PowerShell here-string 방식으로 재실행해 정상 대조했습니다.
