## 1. 요약

본 프로젝트는 카카오페이증권이 2026년 2~3월 실제로 겪은 반복적인 거래 서비스 지연 문제를 대상으로 합니다. 공개 보도에 따르면 국내 정규장 개장 직후 접속·시세·관심종목·커뮤니티 서비스가 세 차례 지연됐으며, 회사는 개장 시간대의 트래픽 집중과 초과를 원인으로 설명했습니다. 이어 외부 중개사 장애로 미국주식 모으기 주문 일부가 미체결된 사례도 발생했습니다.

동시에 카카오페이증권의 예탁자산은 2026년 5월 20조 원을 넘어서는 등 매우 빠르게 성장했습니다. 사용자와 거래량이 빠르게 증가하는 상황에서는 국내장 개장 피크, 외부 중개사 장애, 시세·주문 경로 간 부분 장애를 반복적으로 점검할 수 있는 표준화된 복원력 분석 절차가 중요합니다.

이를 위해 `openbell-guard`를 제안합니다. 이 플러그인은 장애 종료 후 내보낸 익명·합성 사고 번들을 로컬에서 검사하고, 다음 세 산출물을 생성하는 Codex 플러그인입니다.

- `analysis.json`: 통계, 서비스 경로별 영향, 사실·가설·미확인 판단과 근거 연결을 담은 기계 검증용 원본
- `openbell-report.md`: 타임라인, 고객 영향, 경쟁 원인 가설과 재발 방지 계획을 사람이 읽기 쉽게 합친 보고서
- `sanitization-report.md`: 민감정보 탐지·마스킹 결과와 분석 가능 여부를 값 노출 없이 요약한 보고서

대표 사용 장면은 SRE·백엔드·플랫폼 운영자가 장애 종료 후 사고 번들을 분석하는 경우입니다. 장애 중 사용은 사용자가 직접 내보낸 고정 파일을 분석하는 보조 용도로만 다루며, 실시간 모니터링이나 운영 시스템 자동 변경을 약속하지 않습니다.

플러그인은 실제 운영 시스템이나 주문 시스템을 변경하지 않습니다. 또한 공개 자료 또는 제공된 로그만으로 확인할 수 없는 내부 근본 원인을 단정하지 않습니다.

## 2. AX 해커톤 과제 배경

예선 과제는 선택한 기업, 해당 산업 또는 고객이 실제 겪는 문제를 공개·검증 가능한 자료로 입증하고, 이를 해결하는 Codex 플러그인을 제출하는 것입니다.

제출물은 다음 기준을 충족해야 합니다.

- 전체 플러그인 루트가 `submission.zip/src/` 안에 있어야 합니다.
- `src/.codex-plugin/plugin.json`이 반드시 존재해야 합니다.
- 스킬, MCP 서버 또는 실행 코드 중 하나 이상의 실제 동작 요소가 있어야 합니다.
- `README.md`에는 문제, 사용법, 작동 방식과 검증 결과가 설명되어야 합니다.
- 플러그인 개발 과정의 AI 대화 원본 전체를 `logs/`에 편집 없이 제출해야 합니다.
- 질문 5문항과 실제 플러그인, README, 로그의 설명이 서로 일치해야 합니다.

OpenAI 공식 문서에 따르면 플러그인은 스킬, 앱 연동, MCP 서버 등을 하나의 설치 가능한 단위로 묶을 수 있습니다. 스킬은 지침, 참고자료와 실행 스크립트를 포함할 수 있으므로, 본 프로젝트는 스킬과 로컬 분석 스크립트를 중심으로 구성합니다.

### 5.2 고객에게 미치는 영향

회사는 국내장 지연 당시 매수·매도 주문은 정상적으로 접수됐다고 설명했습니다. 그러나 시세, 관심종목, 커뮤니티 등 투자 판단에 사용되는 정보 경로가 느려지면 사용자는 현재 가격과 종목 정보를 제때 확인하기 어렵습니다. 따라서 고객 영향은 다음 두 경로로 구분해야 합니다.

- 거래 실행 경로: 로그인, 주문 접수, 주문 체결, 잔고 반영
- 투자 판단 지원 경로: 시세, 관심종목, 종목 정보, 커뮤니티, 알림

두 번째 경로의 지연이 첫 번째 경로의 완전 중단과 동일하다고 단정할 수는 없습니다. 다만 변동성이 높은 개장 직후 투자 판단을 방해할 수 있으므로 별도의 고객 영향으로 측정해야 합니다.

카카오페이증권의 공식 주문장애 안내에 따르면, 회사 주문 시스템 때문에 MTS·고객센터·ARS 등 모든 방법으로 주문할 수 없는 경우를 주문장애로 정의합니다. 단순 시세지연이나 외부 유관기관 장애 등은 원칙적으로 주문장애에서 제외합니다. 이 기준은 운영 장애 분석과 고객 보상 판단이 동일하지 않을 수 있음을 보여줍니다.

이는 카카오페이증권이 대규모 로그 수집 기반을 실제로 운영하며 장 시작 피크를 중요한 설계 조건으로 다룬다는 직접 근거입니다. 따라서 OpenBell Guard는 새로운 로그 저장 플랫폼을 제안하기보다, 사용자가 내보낸 로그·메트릭을 서비스 경로별 사고 판단과 재발 방지 계획으로 변환하는 분석 계층에 집중합니다.

이 자료만으로 공개 보도된 서비스 지연의 근본 원인이나 기존 로그 플랫폼과의 인과관계를 알 수는 없습니다. 또한 플러그인이 실제 사내 플랫폼에 연결된다고 주장하지 않습니다.

## 7. 최종 문제 정의

### 7.1 한 문장 정의

> 급격히 성장하는 카카오페이증권의 엔지니어와 운영 담당자가 국내장 개장 피크 및 외부 중개사 장애 상황에서 주문 경로와 투자 판단 지원 경로의 이상을 근거 기반으로 신속히 분리·분석하고, 재발 방지 테스트까지 일관되게 연결할 수 있도록 지원합니다.

### 7.2 공개 자료로 확인된 사실

- 2026년 2~3월 국내장 개장 직후 일부 서비스 지연이 세 차례 발생했습니다.
- 회사는 해당 사례들에 대해 개장 시간대 트래픽 집중 문제라고 설명했습니다.
- 외부 중개사 장애로 미국주식 모으기 주문 일부가 미체결된 사례가 발생했습니다.
- 2026년 5월 카카오페이증권의 예탁자산은 20조 원을 넘어섰습니다.
- 회사는 로그·메트릭 분석, 장애 원인 분석과 자동화 역량을 공개 채용 수요로 제시했습니다.

### 7.3 확인할 수 없는 사항

- 실제 내부 아키텍처와 서비스별 용량
- 개별 사건의 상세 근본 원인
- 내부 모니터링·알람·사후 분석 절차의 수준
- 각 사건의 정확한 고객 피해와 보상 규모
- 실제 운영 데이터와 사내 SLO

본 프로젝트는 위 사항을 사실처럼 추정하거나 카카오페이증권의 내부 시스템에 접근했다고 표현하지 않습니다.

## 8. 제안 플러그인 개요

### 8.1 이름

- 표시 이름: OpenBell Guard
- 플러그인 ID: `openbell-guard`
- 핵심 설명: 개장 피크와 외부 연계 장애가 기록된 사고 번들을 분석하고 근거가 연결된 사후 분석 보고서를 만드는 Codex 워크플로

### 8.2 주요 사용자

- 1차 사용자: 사고 종료 후 원인과 영향을 정리하는 SRE, 백엔드 개발자, 플랫폼 운영자
- 2차 사용자: 분석 결과를 재현하고 회귀 테스트로 바꾸는 QA·성능 테스트 담당자
- 파생 결과 사용자: 사람이 검토한 결과를 활용하는 고객지원·소비자보호·서비스 운영 담당자

### 8.3 대표 사용자 흐름

MVP의 대표 흐름은 다음 하나로 고정합니다.

1. 사용자가 합성 또는 사전 익명화된 사고 번들 폴더 경로를 지정합니다.
2. 로컬 스크립트가 파일 크기·형식·시간대와 민감정보를 검사하고 안전한 작업본을 만듭니다.
3. 결정론적 분석기가 60초 단위 통계와 서비스 경로별 영향을 계산하고 근거 ID를 부여합니다.
4. Codex는 원본 파일이 아니라 검증된 `analysis.json`과 마스킹된 제한 근거만 읽어 `openbell-report.md`를 작성합니다.
5. 검증기가 사실-근거 연결, 보고서의 claim ID, 비밀정보 잔존과 산출물 구조를 검사합니다.

대표 사용 시점은 **장애 종료 후**입니다. 장애 중에도 사용자가 고정 파일로 직접 내보낸 번들은 분석할 수 있지만 실시간 스트림 수집·모니터링·자동 조치는 제공하지 않습니다. 장 시작 전 준비 점검, 고객 공지문, 별도 부하 테스트 계획은 기본 산출물이 아니라 후속 선택 기능입니다.

### 8.4 해결 범위

- 사고 번들의 형식·시간대·민감정보와 지원 한도 검사
- 로그·메트릭을 공통 서비스 경로와 60초 구간으로 정규화
- 처리량·오류율·지연시간·관측 지연과 임계치 기반 장애 구간 계산
- 주문·시세·관심종목·외부 연계 등 경로별 영향 분리
- 확인된 사실·원인 가설·판단 불가를 기계 검증 가능한 형태로 기록
- 재발 방지 후보를 근거·검증 조건·롤백 조건과 함께 보고서에 제안

다음 항목은 해결 범위에서 제외합니다.

- 실시간 모니터링·알림과 운영 인프라 자동 증설·배포·설정 변경
- 실제 주문 실행·취소와 고객·계좌 데이터 변경
- 보상 여부, 법적 책임 또는 규제 준수 여부의 자동 판정
- 임계치가 없을 때 장애 시작·회복 시각이나 SLO 위반을 임의 판정
- 내부 데이터 없이 실제 카카오페이증권 사건의 근본 원인 단정
- 모든 개인정보를 완벽하게 탐지·제거한다는 보장

### 8.5 공개 기술 자료의 채택 경계

기업 Techlog에 등장했다는 이유만으로 기술 스택을 복제하지 않습니다. 핵심 문제를 측정 가능하게 개선하고 로컬 합성 데이터로 검증할 수 있는 원칙만 MVP에 반영합니다.

| 구분 | 내용 | 현재 처리 |
|---|---|---|
| MVP 설계에 확정 반영 | 공통 필드 정규화, 관측 지연 계산, 시간·서비스 경로 중심 분석, 증상과 원인 가설 분리, 경쟁 가설·반대 근거, 완화책·근본 조치 구분, 행 단위 입력 처리, 고정 지원 한도 | 본 기획서의 입력·통계·출력·보안·검증 계약으로 고정, Phase 4에서 구현 |
| 개념만 참고 | OpenTelemetry의 정규화, Kafka의 생산·처리 분리, ClickHouse의 시간·필드 집계, HyperDX·Grafana의 역할 분리, JVM 웜업 사례의 조사 방식 | 설계 이유 설명에만 사용 |
| 후순위 | 배포·설정 변경 이벤트 입력, 승인된 읽기 전용 관측 시스템 어댑터, 사용자 조정형 처리 한도, 논리 파티션과 대규모 백분위 최적화 | 핵심 MVP와 필수 검증 후 재검토 |

카카오페이 굿딜 서비스의 JVM 웜업 글은 카카오페이증권의 기술 스택이나 실제 장애 원인을 입증하는 자료로 사용하지 않습니다. 오류 이름을 근본 원인으로 단정하지 않는 조사 절차만 참고합니다.

카카오페이 정산플랫폼팀의 Spring Batch Partitioning 글도 카카오페이증권의 기술 스택을 입증하는 자료로 사용하지 않습니다. 전체 백엔드 구성을 재현하지 않고 입력을 한꺼번에 메모리에 올리지 않는 원칙만 Python 입력 계약으로 축소 적용합니다.

## 9. 입력 계약

### 9.1 사고 번들 구성

MVP는 하나의 로컬 폴더를 **사고 번들**로 입력받습니다. 실제 카카오페이증권 내부 데이터는 사용하지 않으며 개발·심사 시 공개 사건의 양상만 참고한 합성 데이터를 사용합니다.

| 파일 | 필수 여부 | 역할 |
|---|---|---|
| `incident.json` | 필수 | 스키마 버전, 사고 ID, 시간대, 기준 구간, 사고 구간과 선택 임계치 |
| `logs.jsonl` | 둘 중 하나 이상 | 요청·응답 또는 오류 관측 레코드 |
| `metrics.csv` | 둘 중 하나 이상 | 서비스·시간별 수치형 메트릭 |
| `service-map.json` | 권장 | 서비스 이름을 서비스 경로와 내부·외부 의존성으로 매핑 |

지원하지 않는 자유 형식 텍스트·Markdown·YAML·대형 JSON 배열은 MVP 핵심 입력에서 제외합니다. 이를 통해 “어떤 파일이 들어와도 분석한다”는 모호한 약속을 피하고 테스트 가능한 계약을 만듭니다.

### 9.2 파일별 필드 계약

필수·선택 필드, 자료형, 허용 코드, grain, 숫자 범위와 잘못된 필드의 처리 방식은 [지표·수식·판정 기준 계약 3~4절](./openbell-guard-metrics-validation-contract.md#3-분석-단위와-표준-코드)을 유일한 기준으로 사용합니다. 기획서에는 같은 목록을 복제하지 않습니다.

### 9.3 시간대·누락·오류 처리

시간 구간은 `[start, end)`, 내부 계산은 UTC를 사용합니다. 시간대, 누락, 필수·선택 필드 오류, 구간 밖 레코드와 SLO·임계치 부재의 정확한 판정은 [계약 2·4절](./openbell-guard-metrics-validation-contract.md#2-공통-계산-규칙)을 따릅니다.

### 9.4 실행 상태와 종료 코드

`complete`, `degraded`, `fatal`의 필요충분조건, 종료 코드와 여러 오류가 동시에 발견될 때의 우선순위는 [계약 8~9절](./openbell-guard-metrics-validation-contract.md#8-실행-상태와-종료-코드)을 따릅니다.

### 9.5 MVP 지원 한도

파일·레코드·행·번들의 포함 경계와 초과 판정은 [계약 4.2절](./openbell-guard-metrics-validation-contract.md#42-고정-지원-한도)의 바이트 기준을 사용합니다. Phase 4 측정으로 한도를 변경하려면 계약 버전과 Decisionlog를 먼저 갱신합니다.

## 10. 결정론적 분석 및 근거 계약

### 10.1 통계 계산

M-ID별 grain, 단위, 포함·제외 기준, 수식, 최소 표본, `null` 이유, Decimal 반올림과 데이터 소스 우선순위는 [지표 사전 5~6절](./openbell-guard-metrics-validation-contract.md#5-데이터-소스-우선순위)을 유일한 기준으로 사용합니다. 지원 한도 안에서는 계약에 정의된 정확 계산만 사용하며 근사·병렬·분산 집계는 하지 않습니다.

### 10.2 장애 시작·회복 판정

`breach`, `healthy`, `unknown`, 다중 임계치, 연속 버킷, 시작·회복·지속시간과 서비스 경로 상태는 [계약 7절](./openbell-guard-metrics-validation-contract.md#7-임계치와-장애-상태-판정)을 따릅니다. 이는 합성 입력의 사용자 정의 임계치에 대한 판정이며 카카오페이증권의 실제 내부 SLO가 아닙니다.

### 10.3 서비스 경로 분리

표준 코드와 허용 의존성 유형은 [계약 3절](./openbell-guard-metrics-validation-contract.md#3-분석-단위와-표준-코드)을 따릅니다. 서비스 이름은 합성 데이터와 `service-map.json`으로만 매핑하며 카카오페이증권의 실제 내부 서비스명이나 아키텍처를 추정하지 않습니다.

### 10.4 `analysis.json`과 근거 계약

`analysis.json`은 보고서보다 먼저 생성되는 기계 검증용 기준 산출물입니다. contract·schema 버전, M-ID 결과, evidence·claim 필드, 독립 근거 유형, 가설 신뢰도와 실패 issue code는 [계약 8~11절](./openbell-guard-metrics-validation-contract.md#8-실행-상태와-종료-코드)을 따릅니다.

### 10.5 코드와 Codex의 역할

결정론적 Python 코드는 입력·민감정보·한도 검증, 시간 정규화, 통계 계산, evidence·claim 구조 생성과 최종 참조 검사를 담당합니다. 동일한 입력에는 동일한 수치와 구조가 나와야 합니다.

Codex는 검증된 `analysis.json`을 바탕으로 시간 관계와 고객 영향을 설명하고, 증상·가설·판단 불가를 구분하며, 재발 방지 후보를 사람이 검토할 수 있는 `openbell-report.md`로 작성합니다. 오류 이름을 근본 원인으로 승격하거나 단일 상관관계를 인과관계로 표현하지 않으며, 실제 사건과 합성 fixture의 원인을 혼동하지 않습니다.

## 11. 보안 및 개인정보 보호

금융 서비스 로그에는 토큰, 계좌 식별자, 전화번호나 사용자 식별값이 포함될 수 있으므로 원본 데이터와 Codex 사이에 명확한 경계를 둡니다.

```text
사용자가 사고 번들 경로 지정
   ↓
로컬 스크립트가 원본을 행 단위로 읽고 탐지·마스킹
   ↓
임시 마스킹 작업본과 sanitization-report.md 생성
   ↓
결정론적 분석기가 analysis.json 생성
   ↓
Codex는 analysis.json과 제한된 마스킹 근거만 읽음
   ↓
openbell-report.md 생성 후 잔존 비밀정보 검사
```

- 사용자는 대화에 원본 내용을 붙여넣지 않고 사고 번들 경로만 지정합니다.
- `SKILL.md`는 Codex가 원본 `logs.jsonl`·`metrics.csv`를 직접 열거나 인용하지 못하게 하고 반드시 로컬 마스킹 단계를 먼저 실행합니다.
- 마스킹이 실패하면 원본 직접 읽기로 우회하지 않고 종료 코드 `3`으로 중단합니다.
- 원본은 수정하지 않으며 임시 마스킹 작업본을 별도로 만듭니다.
- `sanitization-report.md`에는 유형·개수·논리적 위치만 기록하고 원래 비밀값은 기록하지 않습니다.
- 테스트용 API 키·JWT·쿠키·세션 토큰·계좌번호·전화번호·이메일 패턴이 임시 작업본, `analysis.json`과 Markdown 산출물에 남지 않아야 합니다.
- 정규식 기반 탐지는 알려진 패턴에 대한 방어선이며 모든 개인정보 제거를 보장하지 않습니다. 실제 운영 데이터 적용 전 별도 보안 검토가 필요합니다.
- 번들 스크립트 자체는 외부 API나 네트워크를 호출하지 않습니다. 다만 Codex 서비스 전체가 네트워크와 무관하다고 약속하지 않으며, 원본을 Codex 컨텍스트에 넣지 않는 데이터 흐름으로 위험을 줄입니다.
- 운영 시스템 부하 테스트·주문·배포·설정 변경은 수행하지 않습니다.

## 12. 출력 및 플러그인 구조

## 13. 구현·실행 단계

### 1단계: 스캐폴드와 계약 파일

- 공식 plugin creator로 `src/.codex-plugin/plugin.json`과 `src/skills/openbell-guard/SKILL.md`를 생성합니다.
- 입력·분석·근거·보고서 계약을 references에 문서화합니다.
- 대표 합성 fixture와 기대 `analysis.json`을 코드보다 먼저 고정합니다.

### 2단계: 단일 실행 명령

`run_openbell.py`가 마스킹과 결정론적 분석을 순서대로 조정해 `analysis.json`과 `sanitization-report.md`를 만듭니다.

```powershell
python src/skills/openbell-guard/scripts/run_openbell.py --bundle tests/fixtures/domestic-market-open --output out/domestic-market-open
```

대표 사용자 요청은 “OpenBell Guard로 `domestic-market-open` 사고 번들을 분석하고 기본 보고서를 생성해 주십시오”입니다. Skill이 위 스크립트를 실행하고, 성공 후 Codex가 `analysis.json`만 읽어 `openbell-report.md`를 완성한 다음 `validate_bundle.py`를 실행합니다. 따라서 Python 명령 하나가 AI 보고서까지 단독 생성한다고 표현하지 않습니다.

### 3단계: 안전한 입력 처리

- 사고 번들 파일·스키마·UTF-8·시간대·분석 구간과 고정 지원 한도를 검사합니다.
- JSONL·CSV를 행 단위로 처리하고 파일 크기·처리·거부 건수와 오류 위치를 기록합니다.
- 민감정보를 마스킹한 임시 작업본을 만들고 실패하면 원본 읽기로 우회하지 않습니다.

### 4단계: 텔레메트리 분석기

- 60초 버킷의 처리량·오류율·nearest-rank p50·p95·p99·관측 지연을 계산합니다.
- 기준 구간 중앙값과 사고 구간을 비교합니다.
- 계약의 버킷 상태와 M-ID 규칙으로 장애 시작·회복을 판정합니다.
- `analysis.json`의 evidence·claim 참조를 생성합니다.

### 5단계: Codex 보고서 워크플로

- `analysis.json`에서 확인된 사실·가설·판단 불가를 분리합니다.
- 정상 주문 경로와 느린 정보 경로, 내부와 외부 의존성을 구분합니다.
- 가설마다 지지·반대 근거, 경쟁 가설, 누락 데이터와 질적 신뢰도를 제시합니다.
- 즉시 완화와 근본 조치 후보에 검증·롤백 조건을 붙이되 실제 변경은 실행하지 않습니다.

### 6단계: 설치·호출·패키징 검증

- 공식 manifest·스킬 구조 검증을 통과시킵니다.
- 개발 Marketplace로 `./src`를 설치하고 Codex를 재시작합니다.
- 새 스레드에서 플러그인 발견, 설명에 따른 자동 선택, 명시적 `openbell-guard` 호출을 각각 확인합니다.
- README·질문 5문항·코드·테스트 결과·로그가 같은 대표 흐름과 산출물을 설명하는지 대조합니다.
- 최종 ZIP에서 `.agents`·`docs`·임시 출력·캐시를 제외하고 필수 경로를 검사합니다.

## 14. 검증 계획

### 14.1 합성 시나리오

#### A. 국내장 개장 피크와 경로별 부분 장애

- 08:55부터 요청량이 증가하고 09:00에 시세·관심종목 경로의 p99·오류율이 임계치를 넘지만 주문 접수 경로는 정상으로 유지됩니다.
- 기대 결과: 전체 주문장애로 표현하지 않고 투자 판단 지원 경로의 부분 장애로 분류합니다.

#### B. 외부 중개사 장애

- 내부 API·데이터베이스 지표는 정상이고 해외 브로커 어댑터 타임아웃으로 주식 모으기 경로만 영향을 받습니다.
- 기대 결과: 외부 의존성 장애를 내부 용량 문제와 분리합니다.

#### C. 불완전 데이터

- 메트릭 또는 로그 하나가 없거나 일부 행이 손상되고 서비스 구성도·임계치가 빠집니다.
- 기대 결과: 가능한 통계는 계산하되 `degraded`, 거부 위치, 판단할 수 없는 항목과 필요한 데이터를 기록합니다.

#### D. 비밀정보 포함 입력

- 테스트용 API 키·JWT·계좌번호 패턴을 삽입합니다.
- 기대 결과: 원본을 수정하지 않고 마스킹하며 모든 파생 산출물의 원값 잔존이 0건이어야 합니다. 안전하게 처리할 수 없으면 종료 코드 `3`으로 중단합니다.

#### E. 서비스 정상·로그 유입 지연

- 서비스 지연·오류율은 정상이고 `observed_time - event_time`만 급증합니다.
- 기대 결과: 고객 서비스 장애로 단정하지 않고 관측 파이프라인 지연을 확인된 현상 또는 가설로 분리합니다.

#### F. 타임아웃 증상과 경쟁 가설

- `DB connection timeout`이 있지만 쿼리 지연·커넥션 풀·CPU·GC·배포 시각은 없거나 서로 다른 결론을 지지합니다.
- 기대 결과: 타임아웃을 증상으로 기록하고 DB 용량 부족이나 JVM 웜업을 근본 원인으로 확정하지 않습니다.

#### G. 통계 경계와 임계치 누락

- M-ID별 최소 표본·분모·임계치·연속 버킷의 경계값과 바로 아래·위 값을 각각 만듭니다.
- 기대 결과: nearest-rank 기대값, `null`·reason code, 경로 상태와 시작·회복 시각이 단일 계약과 정확히 일치합니다.

#### H. 지원 한도와 손상 입력

- 정상·손상 행이 섞인 JSONL과 각 파일·레코드·전체 번들 한도를 경계값 및 1단위 초과로 만듭니다.
- 기대 결과: 경계값은 처리하고 초과 입력은 전체 분석 전에 종료 코드 `4`로 중단하며 임의 절단하지 않습니다.

### 14.3 핵심 합격 기준

- 대표 사용자 요청 한 번으로 Skill이 기본 산출물 세 개를 재현합니다.
- 동일한 입력에서 동일한 `analysis.json` 통계와 claim/evidence 관계가 나옵니다.
- 주문 경로가 정상일 때 전체 주문장애라고 표현하지 않습니다.
- 내부 장애와 외부 중개사 장애, 서비스 지연과 관측 지연을 구분합니다.
- 임계치나 표본이 부족하면 장애·백분위수를 만들어내지 않고 `판단 불가` 또는 `null`로 기록합니다.
- 오류 이름을 근본 원인으로 곧바로 승격하지 않고 가설별 지지·반대 근거와 누락 데이터를 표시합니다.
- 모든 확인된 사실에 유효한 evidence가 있고 보고서의 claim ID를 기계가 검증합니다.
- 준비된 비밀정보 원값이 마스킹 작업본과 최종 산출물에 남지 않습니다.
- 지원 한도를 넘는 입력을 부분 분석하지 않고 명시적으로 중단합니다.
- 기본 동작은 읽기 전용이며 운영 시스템을 변경하지 않습니다.

## 15. 과제 질문 5문항과의 정합성

### 15.1 무엇을, 누가, 어떤 상황에서 사용합니까?

`openbell-guard`는 증권 서비스 SRE·백엔드·플랫폼 운영자가 장애 종료 후 내보낸 익명·합성 사고 번들을 분석하고, 서비스 경로별 영향·원인 가설·재발 방지 후보를 근거와 함께 정리하는 플러그인입니다.

### 15.2 왜 이 문제를 선택했습니까?

카카오페이증권에서 2026년 2~3월 개장 직후 서비스 지연이 반복됐고, 외부 중개사 장애도 발생했습니다. 동시에 예탁자산이 빠르게 증가하고 있어 서비스 복원력이 고객 신뢰와 직결된다고 판단했습니다.

### 15.3 플러그인은 어떻게 작동합니까?

사용자가 사고 번들 경로를 지정하면 로컬 스크립트가 먼저 입력·시간대·민감정보·지원 한도를 검사합니다. 이어 60초 통계와 evidence·claim을 담은 `analysis.json`을 만들고, Codex는 이 검증된 결과만 사용해 `openbell-report.md`를 작성합니다.

### 15.4 AI를 어떻게 활용했습니까?

AI는 검증된 분석 결과의 시간 관계와 고객 영향을 설명하고, 확인된 사실·경쟁 가설·판단 불가를 분리하며 재발 방지 후보를 사람이 검토할 수 있는 보고서로 변환합니다. 수치 계산, 민감정보 검사, 근거 참조 검증은 재현 가능한 코드가 담당합니다.

### 15.5 어떻게 검증했습니까?

국내장 개장 피크, 외부 중개사 장애, 불완전 데이터, 비밀정보, 통계 경계와 지원 한도를 합성 fixture로 만듭니다. Golden `analysis.json` 비교, claim-evidence 참조 검사, 비밀정보 잔존 검색과 새 Codex 세션의 발견·자동 선택·명시 호출을 각각 검증합니다.

최종 질문지에는 구현 후 실제 테스트 결과만 기록하며, 아직 수행하지 않은 테스트를 완료된 것처럼 표현하지 않습니다.

| 금융 로그에 민감정보가 포함될 수 있음 | 원본을 Codex가 직접 읽지 않고 로컬 마스킹·분석 결과만 전달, 실패 시 중단 |
| 정규식 마스킹이 모든 개인정보를 식별하지 못할 수 있음 | 합성 데이터 검증으로 범위를 제한하고 실제 운영 적용 전 별도 보안 검토 필요성을 명시 |
| 입력이 너무 크거나 통계 표본이 부족할 수 있음 | 고정 지원 한도에서 명시적으로 중단하고 최소 표본 미달은 null·판단 불가로 기록 |
| 장애 분석 도구가 실제 장애 방지를 보장할 수 없음 | 분석 품질과 준비 절차 개선을 목표로 명시하고 무장애를 약속하지 않음 |
| 기존 사내 도구와 기능이 중복될 수 있음 | 공개 정보상 확인되지 않은 부분이며 보완 도구로 제안한다고 명시 |
