# OpenBell Guard 최종 QA P0 여섯 항목 초급자 설명

> 설명 대상: 백엔드·플러그인 개발 입문자  
> 작성일: 2026-06-30  
> 기준 문서: [`analysis-qa-checklist_report.md`](../reviews/analysis-qa-checklist_report.md)  
> 상태: 2026-06-30 사용자 승인 후 여섯 항목을 기획 기준선과 관련 개발 가이드에 모두 반영했습니다.
> 구현 주의: 이 문서의 수치와 예시는 설명용입니다. 실제 지표·수식·오류 판정은 [단일 기준 계약](../openbell-guard-metrics-validation-contract.md)만 사용합니다.

## 1. P0는 무엇을 의미합니까?

`P0`는 Priority 0, 즉 **코드를 본격적으로 만들기 전에 먼저 결정해야 하는 가장 높은 우선순위**를 뜻합니다.

P0라고 해서 반드시 크고 화려한 기능은 아닙니다. 오히려 이번 여섯 항목은 기능을 더 붙이기보다 다음 질문에 명확히 답해 구현 범위를 줄이는 작업입니다.

1. 정확히 누가, 언제, 무엇을 요청합니까?
2. 어떤 파일을 어떤 형식으로 받습니까?
3. 같은 입력에서 어떻게 같은 숫자를 계산합니까?
4. 민감한 원본 데이터는 누가 읽습니까?
5. 보고서의 문장이 어느 데이터에서 나왔는지 어떻게 증명합니까?
6. 만든 결과가 실제 Codex 플러그인으로 설치·호출됩니까?

여섯 항목의 관계는 다음과 같습니다.

```text
대표 사용자 요청
   │  P0-1: 무엇을 끝까지 해결할지 결정
   ▼
incident bundle
   │  P0-2: 받아들일 파일과 필드 결정
   ▼
로컬 검증·마스킹
   │  P0-4: 원본과 Codex 사이의 보안 경계 결정
   ▼
결정론적 통계 계산
   │  P0-3: 계산식과 지원 범위 결정
   ▼
analysis.json
   │  P0-5: 주장과 근거 연결 및 자동 검사
   ▼
openbell-report.md
   │
   ▼
Codex 플러그인 설치·새 세션 검증
      P0-6: 포장과 실제 호출 검증
```

앞 단계가 정해지지 않으면 뒤 단계도 계속 바뀝니다. 예를 들어 입력 파일 형식이 확정되지 않으면 통계 코드와 테스트 fixture를 만들 수 없고, 통계 규칙이 없으면 보고서의 숫자가 맞는지 검사할 수 없습니다.

## 2. P0-1: 대표 사용자 흐름을 하나로 동결합니다

### 2.1 현재 어떤 문제가 있습니까?

P0 반영 전 기획서는 OpenBell Guard를 다음 여러 상황에서 사용한다고 설명했습니다.

- 장 시작 전 준비 점검
- 장애 발생 중 영향 분석
- 장애 종료 후 사후 분석
- 외부 중개사 장애 분석
- 부하 테스트 계획 작성
- 고객 안내 초안 작성

출력 파일도 타임라인, 가설, 고객 영향, 개선 계획, 부하 테스트 계획, 사후 보고서 등 일곱 개로 나뉘어 있습니다.

각 기능은 개별적으로는 의미가 있지만, 첫 번째 MVP에서 모두 똑같이 중요하다고 두면 무엇이 실제 제품의 중심인지 흐려집니다.

### 2.2 쉬운 비유

처음 식당을 여는 사람이 다음을 한꺼번에 하려는 상황과 비슷합니다.

- 아침에는 빵집
- 점심에는 한식당
- 저녁에는 고깃집
- 밤에는 배달 전문점

각 메뉴를 조금씩 만들 수는 있지만, 손님에게 “이 식당에서 가장 잘하는 음식이 무엇입니까?”라고 물으면 답하기 어렵습니다. 조리법, 재료, 주방 동선과 품질 검사도 전부 달라집니다.

해커톤 심사에서도 마찬가지입니다. 심사자는 짧은 시간 안에 다음을 이해해야 합니다.

> “이 파일 묶음을 주면 OpenBell Guard가 이 문제를 이렇게 해결하고, 이 결과가 나옵니다.”

### 2.3 왜 실시간 장애 대응을 중심에서 내립니까?

현재 MVP에는 Prometheus·Grafana·ClickHouse 같은 실제 관측 시스템과 연결하는 MCP가 없습니다. 사용자가 로그와 메트릭 파일을 직접 내보내서 제공해야 합니다.

따라서 장애가 발생한 바로 그 순간에 시스템을 자동 감시하는 실시간 도구라고 설명하면 구현과 기대가 어긋납니다. 최신 export 파일을 사람이 제공하면 장애 중에도 보조적으로 사용할 수는 있지만, 그것을 핵심 성능처럼 약속해서는 안 됩니다.

### 2.4 해결 방안은 무엇입니까?

대표 흐름을 다음 하나로 고정합니다.

```text
익명화된 incident bundle 경로 제공
→ 로컬 검증·마스킹·통계 계산
→ 근거가 연결된 장애 분석 보고서 생성
```

여기서 `incident bundle`은 한 사건을 분석하는 데 필요한 파일을 한 폴더에 모은 묶음입니다. 예를 들면 다음과 같습니다.

```text
sample-incident/
├── incident.json
├── logs.jsonl
├── metrics.csv
└── service-map.json
```

기본 출력도 다음 세 개로 줄입니다.

| 출력 | 쉬운 역할 |
|---|---|
| `analysis.json` | 컴퓨터가 다시 검사할 수 있는 계산 원장 |
| `openbell-report.md` | 사람이 읽는 종합 보고서 |
| `sanitization-report.md` | 어떤 종류의 민감정보를 처리했는지 보여주는 안전 검사표 |

기존 일곱 문서의 내용이 사라지는 것은 아닙니다. 타임라인·고객 영향·원인 가설·재발 방지는 `openbell-report.md`의 절로 합칩니다. 고객 공지나 부하 테스트 계획은 사용자가 요청할 때만 선택적으로 생성합니다.

### 2.5 왜 이 조치로 구성했습니까?

- **사용자에게 설명하기 쉽습니다.** 한 문장으로 입력과 결과를 말할 수 있습니다.
- **구현량이 줄어듭니다.** 일곱 파일의 템플릿과 상호 정합성을 따로 관리하지 않아도 됩니다.
- **테스트가 쉬워집니다.** 대표 fixture 하나로 처음부터 끝까지 검증할 수 있습니다.
- **AI 출력 불일치가 줄어듭니다.** 여러 문서에서 같은 사건 시각이나 원인을 서로 다르게 쓰는 위험이 줄어듭니다.
- **향후 확장이 가능합니다.** 종합 보고서가 안정된 뒤 필요한 절만 별도 파일로 분리할 수 있습니다.

### 2.6 예시 요청

```text
OpenBell Guard로 tests/fixtures/domestic-market-open 사건 묶음을 분석하고
기본 보고서를 생성해주십시오.
```

이 한 요청으로 입력 검사부터 세 출력까지 끝나야 대표 흐름이 동결됐다고 볼 수 있습니다.

## 3. P0-2: 입력 계약을 파일 단위로 고정합니다

### 3.1 입력 계약은 무엇입니까?

입력 계약은 플러그인과 사용자 사이의 **제출 양식**입니다.

예를 들어 택배를 보낼 때 주소, 수령인과 전화번호를 정해진 칸에 적어야 합니다. 주소를 사진 설명에 쓰거나 전화번호 자리에 메모를 적으면 배송 시스템이 자동으로 이해하기 어렵습니다.

OpenBell Guard도 로그와 메트릭에 어떤 필드가 있어야 하는지 알아야 정확히 계산할 수 있습니다.

### 3.2 현재 어떤 문제가 있습니까?

P0 반영 전 기획서는 JSONL·JSON·텍스트·CSV·YAML·Markdown을 모두 입력 후보로 뒀지만 다음이 완전히 정해지지 않았습니다.

- 어떤 파일이 반드시 있어야 합니까?
- 각 파일에는 어떤 필드가 있어야 합니까?
- 필드의 자료형은 무엇입니까?
- 시각에 시간대가 없으면 어떻게 합니까?
- 로그는 있지만 메트릭이 없으면 분석합니까, 중단합니까?
- SLO가 없으면 어떤 기준으로 장애라고 판단합니까?
- 손상된 레코드 한 건을 건너뛸지 전체 분석을 중단할지 어떻게 정합니까?

이 상태에서 코드를 만들면 개발 중에 매번 새로운 예외 규칙을 추가하게 됩니다.

### 3.3 왜 입력 형식을 줄여야 합니까?

형식마다 읽는 방법이 다릅니다.

- JSONL: 한 줄에 JSON 객체 하나가 있습니다.
- JSON: 배열이나 객체 전체를 한꺼번에 읽을 수 있습니다.
- CSV: 첫 줄의 열 이름에 따라 각 행을 읽습니다.
- YAML: Python 표준 라이브러리에 전용 파서가 없어 보통 외부 패키지가 필요합니다.
- 자유 텍스트·Markdown: 같은 뜻을 여러 방식으로 쓸 수 있어 자동 계산이 어렵습니다.

입력 형식을 많이 지원하는 것은 사용자에게 친절해 보이지만, 실제로는 파서·오류 메시지·테스트 조합을 크게 늘립니다. MVP에서는 적게 지원하되 정확히 지원하는 편이 낫습니다.

### 3.4 해결 방안은 무엇입니까?

한 사건 묶음의 역할을 다음처럼 분리합니다.

| 파일 | 역할 | 예시 필드 |
|---|---|---|
| `incident.json` | 이번 분석의 범위와 규칙 | `incident_id`, `baseline_window`, `incident_window`, `timezone` |
| `logs.jsonl` | 요청·오류·외부 연계 사건 | `event_time`, `service_name`, `service_path`, `status`, `latency_ms` |
| `metrics.csv` | 시간에 따른 수치 변화 | `timestamp`, `service_name`, `metric_name`, `value`, `unit` |
| `service-map.json` | 서비스와 외부 의존성 관계 | `service_name`, `service_path`, `dependency_type` |

이 표는 최종 필드 결정을 보여주는 예시이며, 실제 필수·선택 여부는 기획서 수정 단계에서 고정해야 합니다.

### 3.5 `degraded` 상태는 무엇입니까?

`degraded`는 분석이 완전히 실패하지는 않았지만 **제한된 자료만으로 수행됐다**는 상태입니다.

예를 들어 로그는 있는데 CPU·메모리 메트릭이 없다고 가정합니다.

- 확인 가능: 특정 서비스에서 타임아웃이 늘었습니다.
- 확인 불가: 서버 자원 부족이 원인이었는지는 알 수 없습니다.

이때 플러그인은 그럴듯한 CPU 부족 원인을 만들어내면 안 됩니다. 대신 다음처럼 표시해야 합니다.

```text
분석 상태: degraded
누락 입력: metrics.csv
영향: 자원 포화 여부 판단 불가
```

### 3.6 SLO가 없을 때 왜 임의 기준을 만들면 안 됩니까?

SLO는 서비스가 목표로 하는 성능 기준입니다. 예를 들어 “주문 API 요청의 99%는 500ms 안에 응답한다”와 같은 약속입니다.

SLO가 없는데 OpenBell Guard가 임의로 500ms를 기준으로 정하면 실제 조직의 기준과 다를 수 있습니다. 따라서 다음처럼 처리해야 합니다.

- 입력된 SLO가 있으면 기준 초과 여부를 계산합니다.
- SLO가 없으면 통계 변화는 보여주되 `SLO 위반 여부 판단 불가`라고 표시합니다.

### 3.7 왜 이 조치로 구성했습니까?

- **오류를 일찍 발견합니다.** 분석 중간이 아니라 입력 단계에서 누락을 알려줍니다.
- **테스트 정답을 만들 수 있습니다.** 필드 뜻이 고정돼야 기대값을 계산할 수 있습니다.
- **환각을 줄입니다.** 없는 데이터는 `판단 불가`로 연결됩니다.
- **외부 의존성을 줄입니다.** JSONL·JSON·CSV는 Python 표준 라이브러리로 처리할 수 있습니다.
- **향후 형식 확장이 쉽습니다.** 공통 내부 스키마가 정해진 뒤 YAML 어댑터 등을 추가할 수 있습니다.

## 4. P0-3: 통계와 장애 구간 계산을 재현 가능하게 정의합니다

### 4.1 현재 어떤 문제가 있습니까?

P0 반영 전 기획서는 다음을 계산한다고만 적었습니다.

- 처리량
- 오류율
- p50·p95·p99 지연시간
- 기준 구간과 사고 구간의 차이
- 장애 시작 시각과 회복 시각

하지만 정확한 계산 규칙이 없으면 같은 입력을 두 프로그램이 서로 다르게 계산할 수 있습니다.

### 4.2 쉬운 비유

“라면을 적당히 끓이십시오”라는 조리법을 생각해보십시오.

- 물은 몇 ml입니까?
- 몇 분 끓입니까?
- 센 불입니까, 약한 불입니까?
- 계란은 언제 넣습니까?

사람마다 결과가 달라집니다. 데이터 분석에서 계산식이 없는 상태도 같습니다. `p95를 계산한다`는 말만으로는 충분하지 않습니다.

### 4.3 p50·p95·p99는 무엇입니까?

요청 지연시간을 빠른 순서로 정렬했다고 가정합니다.

- p50: 가운데에 있는 요청 수준입니다.
- p95: 요청의 약 95%가 이 값 이하에 끝납니다.
- p99: 가장 느린 1%에 가까운 꼬리 지연을 보여줍니다.

문제는 백분위수를 구하는 방법이 하나만 있는 것이 아니라는 점입니다. 표본이 적을 때 중간 값을 보간하는 방법에 따라 결과가 달라질 수 있습니다. 그래서 `nearest-rank` 같은 한 방법을 프로젝트 규칙으로 고정해야 합니다.

### 4.4 시간 버킷은 무엇입니까?

로그 한 건씩만 보면 전체 흐름을 알기 어렵습니다. 예를 들어 09:00:00부터 09:00:59까지의 요청을 `1분 버킷` 하나로 묶어 다음을 계산할 수 있습니다.

```text
09:00 버킷
- 요청 수: 12,000건
- 오류 수: 360건
- 오류율: 3%
- p99 지연: 2,400ms
```

버킷 크기가 10초인지 1분인지에 따라 장애 시작 시각과 수치가 달라지므로 미리 고정해야 합니다.

### 4.5 기준 구간과 사고 구간은 왜 입력받습니까?

지연시간 800ms만 보고 느리다고 단정할 수는 없습니다. 평소 100ms였는데 800ms가 됐다면 큰 변화지만, 평소에도 750ms였다면 해석이 달라집니다.

따라서 `incident.json`이 다음 두 구간을 명시하도록 합니다.

- 기준 구간: 평소 상태를 보여주는 시간 범위
- 사고 구간: 문제가 의심되는 시간 범위

시간 범위를 코드가 임의 추정하지 않으면 테스트 결과가 안정되고 사용자의 사건 정의도 보존됩니다.

### 4.6 장애 시작·회복 규칙은 왜 필요합니까?

한 번의 느린 요청만으로 장애 시작을 선언하면 오탐이 많아집니다. 반대로 아주 긴 평균 구간을 사용하면 실제 장애를 늦게 발견합니다.

예를 들어 다음과 같은 규칙을 고려할 수 있습니다.

```text
장애 시작 예시:
오류율 또는 p99가 기준을 3개 버킷 연속 초과

회복 예시:
같은 지표가 기준 이하로 3개 버킷 연속 유지
```

이 설명에서 제시한 방향은 2026-06-30 승인 후 [단일 기준 계약](../openbell-guard-metrics-validation-contract.md)의 M-ID와 경계값으로 최종 확정됐습니다.

### 4.7 행 단위 읽기와 정확한 백분위수는 왜 충돌할 수 있습니까?

행 단위 읽기는 컨베이어 벨트에서 물건을 하나씩 확인하는 것과 같습니다. 파일 전체를 한꺼번에 메모리에 올리지 않으므로 안전합니다.

하지만 정확한 p99를 계산하려고 모든 지연값을 나중에 정렬하기 위해 모아두면, 컨베이어에서 본 물건을 결국 창고에 전부 쌓는 것과 같습니다. 파일은 한 줄씩 읽었지만 메모리에는 지연값 전체가 남습니다.

따라서 다음 중 하나가 필요합니다.

1. 검증한 최대 입력 규모 안에서만 정확한 백분위수를 지원합니다.
2. 디스크 기반 정렬이나 SQLite를 사용합니다.
3. 검증된 근사 백분위 알고리즘을 사용합니다.

MVP에는 첫 번째가 가장 단순합니다. 사용자에게 복잡한 한도 설정 화면을 주는 대신, Phase 4 측정으로 검증된 **고정 지원 범위**를 정하고 초과 시 명확히 거부합니다.

### 4.8 왜 이 조치로 구성했습니까?

- **같은 입력에 같은 숫자가 나옵니다.** 결정론적 코드라는 약속을 지킬 수 있습니다.
- **Golden test를 만들 수 있습니다.** 계산법이 고정돼야 정답 JSON이 생깁니다.
- **성능을 과장하지 않습니다.** 검증한 입력 규모까지만 지원한다고 말할 수 있습니다.
- **복잡한 대규모 기술을 피합니다.** 첫 MVP부터 분산 처리나 근사 알고리즘을 넣지 않습니다.
- **장애 판정의 근거를 설명할 수 있습니다.** 왜 09:02를 시작 시각으로 봤는지 규칙을 보여줄 수 있습니다.

## 5. P0-4: 원본 데이터와 Codex 사이의 보안 경계를 바로잡습니다

### 5.1 현재 어떤 문제가 있습니까?

P0 반영 전 기획서에는 `로컬 파일만 처리하며 기본 설정에서 네트워크 전송을 하지 않는다`는 표현이 있었습니다.

Python 스크립트가 사용자 컴퓨터에서 실행되고 외부 API를 호출하지 않는다는 뜻이라면 맞을 수 있습니다. 그러나 Codex가 원본 파일을 직접 읽어 모델 문맥에서 분석하면, 단순히 `로컬 파일`이라는 이유만으로 원본이 로컬에서만 처리된다고 단정할 수 없습니다.

### 5.2 쉬운 비유

민감한 서류가 든 봉투를 생각해보십시오.

- 로컬 스크립트: 봉투를 안전 구역에서 열어 개인정보를 검게 가리는 담당자
- Codex: 검게 가려진 복사본을 읽고 보고서를 쓰는 분석가

분석가가 원본 봉투까지 직접 받으면 앞의 가림 작업이 의미가 약해집니다. 따라서 원본을 읽는 주체와 마스킹 결과를 읽는 주체를 분리해야 합니다.

### 5.3 해결 방안은 무엇입니까?

```text
사용자
  └─ 원본 내용이 아니라 incident bundle의 경로를 제공
       │
       ▼
로컬 Python 스크립트
  ├─ 파일 형식 검사
  ├─ 비밀정보·개인정보 패턴 탐지
  ├─ 마스킹된 파생본 생성
  └─ 결정론적 analysis.json 생성
       │
       ▼
Codex
  ├─ 마스킹된 analysis.json 읽기
  ├─ 허용된 근거 조각만 종합
  └─ 사람용 보고서 작성
```

`SKILL.md`에는 다음 원칙을 명시해야 합니다.

- Codex는 마스킹 전 원본 파일을 직접 열거나 인용하지 않습니다.
- 스크립트에 원본 파일 경로를 전달합니다.
- 스크립트가 실패하면 원본을 대신 읽어서 분석하지 않고 중단합니다.
- `sanitization-report.md`에는 발견한 원래 값을 쓰지 않고 유형·건수·위치만 기록합니다.

### 5.4 왜 `네트워크 전송 없음`이라는 문장을 바꿉니까?

우리가 직접 검증할 수 있는 사실은 다음과 같습니다.

- 번들 Python 스크립트가 외부 API나 데이터베이스를 호출하지 않습니다.
- 원본 파일을 Codex 입력으로 사용하지 않도록 워크플로를 설계합니다.
- 마스킹된 결과에서 Seeded secret이 남지 않는지 테스트합니다.

반면 Codex 제품 전체의 데이터 처리 방식은 실행 환경·계정·설정에 따라 달라질 수 있으므로 플러그인 문서가 포괄적으로 `절대 네트워크 전송되지 않는다`고 보장해서는 안 됩니다.

### 5.5 정규식 마스킹은 완벽합니까?

아닙니다. API 키처럼 일정한 모양이 있는 값은 탐지하기 쉽지만, 자유 텍스트에 적힌 이름이나 회사 내부 식별자처럼 패턴이 불명확한 정보는 놓칠 수 있습니다.

따라서 다음 경계를 함께 둡니다.

- 테스트로 준비한 비밀정보 패턴은 100% 제거합니다.
- 탐지할 수 없는 모든 개인정보까지 완벽히 제거한다고 주장하지 않습니다.
- 실제 운영 데이터 적용 전에는 조직의 보안 검토와 추가 탐지 규칙이 필요하다고 표시합니다.

### 5.6 왜 이 조치로 구성했습니까?

- **원본 노출 면적을 줄입니다.** AI는 필요한 최소 정보만 봅니다.
- **설명과 실제 동작이 일치합니다.** 로컬 스크립트와 Codex의 역할을 구분합니다.
- **실패를 안전하게 처리합니다.** 마스킹 실패 시 원본을 대신 읽는 우회로를 막습니다.
- **테스트할 수 있습니다.** 원본을 금지하고 파생 결과의 비밀정보 잔존을 검사할 수 있습니다.
- **과도한 보안 보장을 피합니다.** 정규식의 한계를 정직하게 표시합니다.

## 6. P0-5: 사실과 근거를 기계가 검사할 수 있게 만듭니다

### 6.1 현재 어떤 문제가 있습니까?

기획서는 모든 사실 문장에 파일 위치나 타임스탬프 근거를 붙이도록 요구합니다. 좋은 원칙이지만, 사람에게 “근거를 꼭 쓰십시오”라고 지시하는 것만으로는 누락을 자동으로 잡기 어렵습니다.

다음과 같은 문장이 보고서에 생겼다고 가정합니다.

> 시세 서비스의 p99 지연이 2.4초로 증가했습니다.

이 문장이 어느 파일의 어느 구간에서 계산됐는지 표시되지 않으면 심사자는 사실을 다시 확인하기 어렵습니다.

### 6.2 쉬운 비유

법정에서 다음 두 문장의 차이를 생각해보십시오.

1. “피고인이 현장에 있었습니다.”
2. “CCTV E-014와 출입기록 E-015가 피고인의 현장 출입을 보여줍니다.”

두 번째 문장은 주장과 증거가 연결돼 있습니다. OpenBell Guard에서도 보고서 문장을 `claim`, 원본 위치·시간·계산 결과를 `evidence`로 나눕니다.

### 6.3 `claim`과 `evidence`는 무엇입니까?

- claim: 보고서가 주장하는 내용
- evidence: 그 주장을 지지하거나 반박하는 확인 가능한 근거

예시는 다음과 같습니다.

```json
{
  "claims": [
    {
      "claim_id": "C-001",
      "claim_type": "confirmed_fact",
      "text": "시세 경로의 p99 지연이 사고 구간에 증가했습니다.",
      "evidence_refs": ["E-014", "E-015"]
    }
  ],
  "evidence": [
    {
      "evidence_id": "E-014",
      "source_file": "metrics.csv",
      "source_location": "rows 120-180",
      "time_window": "09:00-09:05"
    }
  ]
}
```

### 6.4 왜 Markdown 보고서만으로는 부족합니까?

Markdown은 사람이 읽기 좋지만 문장 구조가 자유롭습니다. 근거 표기가 빠졌는지, 존재하지 않는 ID를 썼는지 프로그램이 검사하기 어렵습니다.

반면 `analysis.json`은 다음을 자동 검사할 수 있습니다.

- `confirmed_fact` claim마다 근거가 하나 이상 있습니까?
- `evidence_refs`가 실제 evidence ID를 가리킵니까?
- 근거 파일과 위치가 존재합니까?
- 가설에 지지 근거와 반대 근거가 있습니까?
- 결과에 비밀정보가 남아 있습니까?

Markdown은 이 검증을 통과한 `analysis.json`을 바탕으로 작성합니다. 즉, JSON은 원장이고 Markdown은 읽기 쉬운 설명서입니다.

### 6.5 원인 가설의 신뢰도는 왜 숫자로 쓰지 않습니까?

근거가 불완전한 상태에서 `원인일 확률 83%`처럼 쓰면 실제 통계 모델이 계산한 숫자로 오해될 수 있습니다.

따라서 MVP에서는 다음과 같은 질적 등급이 더 적합합니다.

- 높음: 독립된 여러 근거가 지지하고 중요한 반대 근거가 없습니다.
- 중간: 일부 근거가 지지하지만 반대 근거나 누락 데이터가 있습니다.
- 낮음: 간접 근거만 있거나 경쟁 가설을 충분히 배제하지 못했습니다.
- 판단 불가: 필요한 데이터가 없습니다.

구체적인 근거 개수와 규칙은 기획서 수정 단계에서 고정해야 합니다.

### 6.6 왜 이 조치로 구성했습니까?

- **심사자가 결과를 재검증할 수 있습니다.** 문장부터 원본 위치까지 추적됩니다.
- **환각을 자동으로 잡습니다.** 근거 없는 사실 claim을 검증기가 실패시킵니다.
- **AI와 코드의 역할이 분명해집니다.** 코드는 근거 원장을 만들고 AI는 설명을 구성합니다.
- **보고서 형식이 바뀌어도 근거가 유지됩니다.** Markdown 템플릿과 JSON 원장을 분리합니다.
- **과도한 확신을 줄입니다.** 가설과 사실, 지지와 반대 근거가 구조적으로 분리됩니다.

## 7. P0-6: 플러그인 이름과 설치 검증 규칙을 바로잡습니다

### 7.1 현재 어떤 문제가 있습니까?

AX 해커톤 제출 구조는 플러그인 루트를 `submission.zip/src/`에 두도록 요구합니다. 따라서 실제 루트 폴더 이름은 `src`입니다.

P0 반영 전 `AGENTS.md`와 제작 가이드는 `plugin.json`의 `name`이 플러그인 디렉터리 이름과 같아야 한다고 적고 있었습니다. 이를 그대로 따르면 다음 충돌이 생겼습니다.

```text
플러그인 루트 폴더: src
원하는 플러그인 ID: openbell-guard
```

디렉터리 이름과 같게 만들면 manifest 이름이 `src`가 되어 프로젝트 ID와 어긋납니다.

### 7.2 쉬운 비유

택배 상자와 상품명을 생각해보십시오.

- `src`: 해커톤이 지정한 택배 상자의 위치
- `openbell-guard`: 상자 안 제품의 고유 이름

상자가 배송 트럭의 세 번째 칸에 있다고 해서 상품 이름까지 `세 번째 칸`이 될 필요는 없습니다.

### 7.3 공식 규칙은 무엇입니까?

[OpenAI Build plugins](https://developers.openai.com/codex/plugins/build) 문서는 manifest `name`을 안정적인 kebab-case 식별자로 사용하도록 설명합니다. 루트 디렉터리 이름과 반드시 같아야 한다는 요구는 현재 공식문서에서 확인되지 않았습니다.

따라서 다음 구성이 가능합니다.

```text
src/.codex-plugin/plugin.json
```

```json
{
  "name": "openbell-guard",
  "version": "0.1.0",
  "description": "...",
  "skills": "./skills/"
}
```

### 7.4 파일 구조가 맞으면 왜 설치 테스트가 또 필요합니까?

폴더와 JSON이 보기 좋게 배치됐다고 해서 Codex가 실제 플러그인으로 발견한다는 보장은 없습니다.

다음 문제가 있을 수 있습니다.

- manifest JSON 문법 오류
- `skills` 상대경로 오류
- `SKILL.md` 메타데이터 오류
- Marketplace가 잘못된 경로를 가리킴
- 수정 전 캐시가 남음
- 기존 스레드가 새 플러그인을 다시 로드하지 않음

따라서 개발용 Marketplace로 플러그인을 설치한 뒤 Codex를 재시작하고 새 스레드에서 확인해야 합니다.

### 7.5 세 가지 호출 검증

1. 발견: 플러그인 목록에 OpenBell Guard가 보입니까?
2. 자동 선택: “개장 직후 지연 로그를 분석해주십시오”라고 했을 때 관련 Skill을 선택합니까?
3. 명시 호출: 사용자가 플러그인·Skill을 직접 지정했을 때 실행됩니까?

파일 구조 검사는 상자가 제대로 포장됐는지 보는 일이고, 새 세션 호출 검증은 실제 배송 후 제품이 작동하는지 확인하는 일입니다.

### 7.6 Marketplace는 제출물입니까?

아닙니다. Marketplace는 개발 중 설치와 발견을 시험하기 위한 도구입니다. 해커톤 제출 ZIP의 필수 구조와 구분해야 합니다.

- 개발 환경: Marketplace가 `src/` 플러그인 경로를 가리킵니다.
- 제출 환경: `submission.zip/src/.codex-plugin/plugin.json`과 실제 Skill·코드가 들어갑니다.
