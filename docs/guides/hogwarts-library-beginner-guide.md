## 1. 1분 요약

카카오페이증권에는 로그인, 시세 조회, 관심종목, 주문, 외부 브로커 연계 등 많은 서비스가 있습니다. 각 서비스는 실행 중에 자신에게 무슨 일이 있었는지를 `로그`라는 기록으로 남깁니다.

서비스가 성장하면서 카카오페이증권이 공개한 로그량은 하루 약 100GB에서 41TB·200억 건 이상으로 증가했습니다. 기존 시스템에서는 로그가 늦게 도착하거나 검색이 오래 걸리고 비용이 빠르게 늘어나는 문제가 생겼습니다.

호그와트 도서관 프로젝트는 이 문제를 해결하기 위해 다음 전 과정을 다시 설계한 작업입니다.

```text
로그 생성 → 수집 → 임시 보관 → 분류·처리 → 저장 → 검색 → 장기 보관
```

OpenBell Guard는 이 전체 시스템을 다시 만들지 않습니다. 사용자가 파일로 내보낸 로그와 메트릭을 받아 장애 시간대, 영향 경로, 원인 가설과 추가 확인할 증거를 정리하는 분석 도구입니다.

```text
호그와트 도서관: 로그를 빠르고 안전하게 모으고 찾게 해주는 시스템
OpenBell Guard: 그 로그를 읽고 사고를 설명하고 재발 방지 작업을 만드는 도구
```

이 차이를 이해하면 플러그인 구축에 필요한 핵심의 절반을 이해한 것입니다.

## 2. 로그란 무엇입니까?

로그는 프로그램이 남기는 업무 일지입니다. 예를 들어 사용자가 시세를 조회하면 서버가 다음과 비슷한 기록을 남길 수 있습니다.

```json
{
  "event_time": "2026-03-03T09:00:02+09:00",
  "service_name": "synthetic-market-data-api",
  "service_path": "market_data",
  "status": "timeout",
  "latency_ms": 1800
}
```

이 예시는 프로젝트용 합성 데이터이며 실제 카카오페이증권 서비스명이 아닙니다.

각 필드는 다음 뜻입니다.

- `event_time`: 사건이 실제로 발생한 시각
- `service_name`: 기록을 남긴 서비스
- `service_path`: 시세·주문·외부연계 등 고객 기능 경로
- `status`: 성공, 오류 또는 시간 초과 여부
- `latency_ms`: 요청을 처리하는 데 걸린 시간

로그 한 건은 작은 메모에 불과합니다. 하지만 수십억 건을 시간순으로 모으면 “09:00에 어느 서비스부터 느려졌는가”, “내부 서비스는 정상이었지만 외부 브로커만 응답하지 않았는가” 같은 질문에 답할 수 있습니다.

## 3. 도서관 비유로 전체 구조 이해하기

거대한 학교에서 매초 수많은 쪽지가 만들어진다고 생각해 보십시오.

| 기술 구성요소 | 도서관 비유 | 실제 역할 |
|---|---|---|
| 애플리케이션 | 쪽지를 쓰는 학생 | 서비스가 로그를 생성합니다. |
| OpenTelemetry Collector | 쪽지를 모으고 표준 라벨을 붙이는 사서 | 여러 로그를 수집·정규화하고 묶어서 전송합니다. |
| Kafka | 쪽지가 잠시 기다리는 컨베이어 벨트 | 로그 생산과 처리를 분리하고 밀린 데이터를 보관합니다. |
| Routing·처리 Collector | 종류별 분류대 | 로그 유형과 우선순위에 따라 처리 경로를 나눕니다. |
| ClickHouse | 분류가 잘된 대형 서고 | 대량 로그를 저장하고 시간·서비스 조건으로 빠르게 집계합니다. |
| HyperDX | 도서 검색대 | 사람이 로그를 검색하고 사건 흐름을 봅니다. |
| Grafana | 건물 계기판 | CPU, 처리량, 오류율과 같은 메트릭을 대시보드로 봅니다. |
| ssak3 | 오래된 책을 옮기는 기록 담당자 | 오래된 로그를 S3로 내보내고 진행 상태를 관리합니다. |
| Amazon S3 | 값싼 외부 보관 창고 | 오래된 로그를 저렴하게 장기 보관합니다. |
| AWS ClickHouse | 필요할 때만 여는 임시 열람실 | 장기 로그 조사가 필요할 때 S3 데이터를 조회합니다. |

기술 이름을 전부 외울 필요는 없습니다. `수집 → 대기 → 분류 → 저장 → 검색 → 보관`의 흐름과 각 구성요소가 담당하는 역할만 이해하면 됩니다.

## 4. 기존 시스템은 왜 힘들어졌습니까?

### 4.1 로그가 410배 이상 증가했습니다

원문에 따르면 2022년 약 100GB였던 하루 로그량이 3년여 만에 41TB·200억 건 이상으로 늘었습니다. 작은 도서관에서 쓰던 방식으로 초대형 도서관을 운영하게 된 셈입니다.

회사가 공개한 기존 문제는 다음과 같습니다.

- 로그가 검색 시스템에 들어오기까지 수 분에서 수 시간이 걸렸습니다.
- 로그 검색이 5분 이상 걸리는 경우가 있었습니다.
- OpenSearch 관련 비용이 로그 증가와 함께 크게 늘었습니다.
- 서비스별 Kafka Topic과 Fluentd가 지나치게 많이 늘어 운영 복잡도가 커졌습니다.
- 오래된 로그를 조회하기 위한 스키마 관리가 어려워졌습니다.

이 수치와 성과는 카카오페이증권이 공식 글에서 공개한 값이며 이 프로젝트가 독립적으로 측정한 결과는 아닙니다.

### 4.2 로그 한 건씩 보내는 비용이 컸습니다

기존 방식은 로그 한 줄마다 JSON 필드 이름을 반복해 Kafka로 보냈습니다.

```text
한 건 전송 × 1,000건 = 네트워크 요청과 반복 문자열도 1,000번
```

새 방식은 여러 로그를 한 묶음으로 보내고, 필드명을 짧은 숫자 태그로 표현하는 OTLP Protocol Buffers를 활용했습니다.

```text
여러 건을 한 상자에 포장 → 운송 횟수와 포장재 감소
```

우리 플러그인은 OTLP를 구현하지 않습니다. 다만 입력마다 제각각인 필드명을 공통 이름으로 바꾸는 `정규화` 원칙을 사용합니다.

### 4.3 서비스마다 별도 처리 시설이 늘었습니다

한 서비스의 로그 폭증이 다른 서비스에 영향을 주는 문제를 막기 위해 서비스별 Kafka Topic과 Fluentd를 늘렸지만, 서비스가 많아지면서 Topic은 300개 이상, Fluentd Pod는 1,000개 이상으로 증가했다고 설명합니다.

새 구조에서는 성능 테스트를 거친 뒤 Topic을 다시 통합하고, 중요도와 로그 유형을 기준으로 처리 자원을 나눴습니다.

여기서 배울 점은 “항상 합치는 것이 좋다”가 아닙니다.

- 분리하면 장애 영향은 줄지만 운영 대상이 많아집니다.
- 통합하면 운영은 단순해지지만 하나의 폭증이 전체에 영향을 줄 수 있습니다.
- 따라서 실제 부하 테스트 결과와 우선순위 기준이 필요합니다.

OpenBell Guard에서는 Kafka Topic을 다루지 않습니다. 대신 주문, 시세, 정보, 외부 브로커와 관측 계층을 분리해 어느 고객 경로가 영향을 받았는지 분석합니다.

### 4.4 저장 기술과 실제 검색 방식이 맞지 않았습니다

OpenSearch는 문장 속 특정 단어를 찾는 전문 검색에 강합니다. 그러나 원문에 따르면 실제 로그 조회의 90% 이상은 다음과 같은 형태였습니다.

```text
특정 시간대 + 특정 서비스 + 특정 필드 조건
```

예를 들면 다음과 같습니다.

```text
09:00~09:05 사이
market-data 서비스에서
status가 timeout인 로그
```

ClickHouse는 열 단위 저장과 대량 집계에 강하므로 이런 조회 패턴에 더 잘 맞았습니다. 중요한 교훈은 유명한 기술을 선택하는 것이 아니라, 실제 질문과 데이터 접근 방식에 맞는 기술을 선택해야 한다는 것입니다.

OpenBell Guard도 같은 순서로 분석합니다.

1. 사고 시간대를 좁힙니다.
2. 서비스와 고객 경로를 좁힙니다.
3. 오류·지연·외부 의존성 필드를 비교합니다.
4. 원본 파일과 행 번호를 근거로 남깁니다.

### 4.5 스키마란 무엇이며 왜 장기 로그에서 어려웠습니까?

스키마는 데이터에 어떤 필드가 있고, 각 필드가 어떤 의미와 자료형을 가지는지 정한 `데이터 설계도`입니다.

예를 들어 다음 로그가 있다고 가정하겠습니다.

```json
{
  "event_time": "2026-03-03T09:00:02+09:00",
  "service_name": "synthetic-order-api",
  "status": "timeout",
  "latency_ms": 1800
}
```

#### 필드란 무엇입니까?

필드는 로그 한 건 안에서 특정 사실 하나를 담는 `이름표가 붙은 칸`입니다.

```json
"latency_ms": 1800
```

이 한 줄은 다음 두 부분으로 나뉩니다.

- `latency_ms`: 필드 이름입니다. 어떤 정보를 담는 칸인지 알려줍니다.
- `1800`: 필드 값입니다. 해당 사건에서 실제로 기록된 내용입니다.

회원가입 신청서로 비유하면 `이름`, `생년월일`, `전화번호`가 각각 필드이고 사용자가 적은 `홍길동`, `2000-01-01`, `010-0000-0000`이 값입니다.

Python의 딕셔너리로 이해하면 거의 정확합니다.

```python
log = {
    "event_time": "2026-03-03T09:00:02+09:00",
    "service_name": "synthetic-order-api",
    "status": "timeout",
    "latency_ms": 1800,
}
```

```text
Python dict의 key   = 로그의 필드 이름
Python dict의 value = 해당 필드의 값
Python dict 한 개   = 로그 한 건 또는 레코드
dict 여러 개의 list = 여러 건의 로그 데이터
```

```python
log["status"]       # "timeout"
log["latency_ms"]  # 1800
```

다만 `필드`와 `key`가 항상 완전히 같은 말은 아닙니다.

- JSON 로그를 Python `dict`로 읽었을 때는 필드 이름이 `key`가 됩니다.
- 같은 데이터를 Pandas DataFrame으로 만들면 필드는 `column`이 됩니다.
- 데이터베이스 테이블에서는 필드가 컬럼에 해당합니다.
- Python `dict`의 key는 숫자·튜플도 가능하지만 JSON 로그의 필드 이름은 문자열이어야 합니다.
- 스키마는 단순한 딕셔너리 한 건이 아니라 각 key에 허용할 자료형·필수 여부·값의 범위를 정한 규칙입니다.

예를 들어 다음은 스키마를 단순화해 Python 형태로 표현한 것입니다.

```python
schema = {
    "event_time": str,
    "service_name": str,
    "status": str,
    "latency_ms": int,
}
```

실제 스키마는 `status`에 어떤 문자열을 허용할지, `latency_ms`를 반드시 입력해야 하는지 같은 조건도 포함할 수 있습니다.

로그를 표로 펼치면 더 쉽게 이해할 수 있습니다.

| event_time | service_name | status | latency_ms |
|---|---|---|---:|
| 09:00:02 | synthetic-order-api | timeout | 1800 |
| 09:00:03 | synthetic-order-api | success | 120 |
| 09:00:04 | synthetic-market-api | error | 950 |

이 표에서는 다음처럼 부릅니다.

- 표 전체: 로그 데이터셋
- 한 행: 로그 한 건 또는 레코드
- 한 열: 필드 또는 컬럼
- 한 칸: 특정 로그의 필드 값
- 열 이름과 자료형 규칙 전체: 스키마

```text
데이터셋
 ├─ 레코드 1
 │   ├─ event_time 필드 = 09:00:02
 │   ├─ service_name 필드 = synthetic-order-api
 │   ├─ status 필드 = timeout
 │   └─ latency_ms 필드 = 1800
 ├─ 레코드 2
 └─ 레코드 3
```

필드가 있으면 로그를 통째로 읽지 않고 필요한 질문을 정확히 표현할 수 있습니다.

- 필터링: `status`가 `timeout`인 로그만 찾습니다.
- 그룹화: `service_name`별 오류 건수를 계산합니다.
- 계산: `latency_ms`의 평균·p95·p99를 계산합니다.
- 시간 정렬: `event_time` 순서로 사고 타임라인을 만듭니다.
- 경로 분류: `service_path`가 주문인지 시세인지 구분합니다.
- 외부 장애 구분: `dependency_type`이 내부·거래소·브로커 중 무엇인지 확인합니다.

값이 없다는 표현도 구분해야 합니다.

| 표현 | 의미 |
|---|---|
| 필드 자체가 없음 | 해당 서비스가 그 정보를 기록하지 않았습니다. |
| `"latency_ms": null` | 필드는 있지만 값을 알 수 없거나 측정하지 못했습니다. |
| `"latency_ms": 0` | 실제 기록된 값이 0입니다. |

`없음`, `알 수 없음`, `실제 0`을 같은 것으로 처리하면 평균과 오류율이 왜곡될 수 있습니다. OpenBell Guard는 없는 필드를 임의의 값으로 채우지 않고 누락 상태로 보고해야 합니다.

이 로그의 스키마는 다음처럼 표현할 수 있습니다.

| 필드 | 자료형 | 의미 |
|---|---|---|
| `event_time` | 날짜·시간 | 사건이 발생한 시각 |
| `service_name` | 문자열 | 로그를 만든 서비스 |
| `status` | 문자열 | 성공·오류·시간 초과 상태 |
| `latency_ms` | 정수 | 처리 지연시간(밀리초) |

도서관으로 비유하면 스키마는 모든 책에 공통으로 붙이는 도서 목록 양식입니다.

```text
제목은 문자열
출판일은 날짜
페이지 수는 숫자
분류 번호는 정해진 코드
```

스키마는 다음 기능을 수행합니다.

1. **해석:** 각 값이 무엇을 뜻하는지 알려줍니다.
2. **검증:** 숫자 필드에 문자가 들어오는 등 잘못된 데이터를 찾습니다.
3. **검색:** `latency_ms > 1000`처럼 필드와 자료형을 이용한 조건 검색을 가능하게 합니다.
4. **집계:** 평균 지연시간, 오류 건수와 서비스별 통계를 계산하게 합니다.
5. **일관성:** 서로 다른 서비스의 로그를 같은 기준으로 비교하게 합니다.
6. **저장 최적화:** 날짜·숫자·문자 특성에 맞는 압축과 인덱스를 선택하게 합니다.

S3는 파일을 저장하는 창고이므로 파일 안의 각 값이 무엇을 의미하는지 자체적으로 관리하지 않습니다. Athena가 S3의 오래된 로그를 표처럼 조회하려면 `어떤 파일에 어떤 컬럼이 있고 자료형이 무엇인지`를 테이블 스키마로 정의해야 합니다.

카카오페이증권은 애플리케이션·Nginx·DB·Kafka·Redis·FEP 등 다양한 로그를 다루었고, 오래된 로그에는 2,000개 이상의 컬럼이 생길 수 있었다고 설명합니다. 이때 다음 문제가 발생합니다.

- 새로운 로그 필드가 추가될 때마다 테이블 정의를 갱신해야 합니다.
- 같은 이름의 필드가 서비스마다 다른 자료형을 가질 수 있습니다.
- 일부 파일에만 존재하는 필드와 중첩 JSON을 관리해야 합니다.
- 오래된 파일과 최신 파일의 구조가 다를 수 있습니다.
- 수천 개 컬럼 중 실제 조회에 필요한 컬럼을 찾고 문서화하기 어렵습니다.

예를 들어 과거에는 `latency_ms`가 숫자였는데 어떤 서비스가 이를 `"1.8s"`라는 문자열로 저장하면 같은 컬럼을 숫자로 집계하기 어려워집니다. 이름을 `response_time`으로 바꾸면 과거와 현재 로그를 하나의 조건으로 검색하기도 어렵습니다. 이를 `스키마 변경` 또는 `스키마 불일치` 문제로 볼 수 있습니다.

호그와트 도서관 프로젝트는 오래된 로그를 Parquet 형식으로 S3에 보관하고 ClickHouse의 S3 기능으로 조회합니다. Parquet 파일에는 컬럼 이름과 자료형 정보가 함께 들어 있어 ClickHouse가 이를 읽을 수 있고, 필요한 경우 View에서 서로 다른 로그 유형을 공통 형태로 맞춥니다.

이는 스키마가 사라진다는 뜻이 아닙니다. 사람이 수천 개 컬럼을 하나의 Athena 테이블에 계속 수동 정의하는 부담을 줄이고, 파일에 포함된 스키마와 조회용 View를 활용한다는 뜻입니다.

OpenBell Guard에서는 수천 개 필드를 다루지 않습니다. 분석에 필요한 최소 공통 스키마만 정합니다.

```text
event_time, observed_time, service_name, service_path,
log_type, severity, status, latency_ms, dependency_type
```

Python 분석기는 이 필드의 존재 여부와 자료형을 검사하고, 없는 값을 임의로 만들어내지 않습니다. Athena·S3·ClickHouse 스키마를 직접 관리하는 기능은 MVP에 포함하지 않습니다.

## 5. 호그와트 도서관이 바꾼 일곱 단계

### 5.1 Filebeat에서 OpenTelemetry로 수집기 전환

여러 로그를 표준 형태로 정리하고 배치로 전송해 네트워크와 Kafka 처리 부담을 줄였습니다.

우리에게 필요한 이해:

- 로그마다 공통 필드가 있어야 비교하기 쉽습니다.
- 한 건씩 처리하는 것보다 먼저 묶고 요약하는 편이 효율적일 수 있습니다.

우리 프로젝트에서 하지 않는 일:

- OpenTelemetry Collector 설치
- OTLP 바이너리 전송

### 5.2 Kafka와 Topic 구조 통합

Kafka는 대량의 메시지를 순서대로 받아 잠시 보관하고, 필요한 처리 프로그램에 전달하는 시스템입니다. 로그 처리에서는 서비스가 만든 로그와 ClickHouse에 저장하는 처리기 사이의 `대기·전달 계층` 역할을 합니다.

```text
서비스가 로그 생성
      │
      ▼
Kafka가 순서대로 보관
      │
      ▼
처리기가 가져가 정리
      │
      ▼
ClickHouse에 저장
```

도서관으로 비유하면 학생들이 반납한 책을 사서가 즉시 서가에 꽂지 못할 때 사용하는 `반납 책 대기 컨베이어`와 같습니다.

- 학생이 책을 계속 반납해도 컨베이어가 잠시 받아둡니다.
- 사서는 자신이 처리할 수 있는 속도로 책을 가져갑니다.
- 사서가 잠시 멈춰도 대기 중인 책이 바로 사라지지 않습니다.
- 사서를 더 투입하면 밀린 책을 더 빠르게 처리할 수 있습니다.

Kafka가 필요한 이유는 로그를 만드는 서비스와 로그를 처리하는 프로그램의 속도가 다를 수 있기 때문입니다. 09:00 장 시작에 로그가 평소보다 훨씬 빠르게 생성되더라도 처리기가 같은 속도로 바로 처리할 수 있다는 보장은 없습니다. Kafka가 없다면 처리기가 감당하지 못한 로그가 유실되거나, 로그를 보내느라 실제 고객 요청을 처리하는 서비스까지 느려질 수 있습니다.

Kafka의 기본 용어는 다음 정도만 이해하면 충분합니다.

| 용어 | 쉬운 뜻 | 도서관 비유 |
|---|---|---|
| Producer | Kafka에 메시지를 보내는 프로그램 | 책을 반납하는 학생 |
| Broker | 메시지를 보관하고 전달하는 Kafka 서버 | 반납 책을 받아두는 컨베이어 시설 |
| Topic | 메시지를 용도별로 나눈 통로 | 책 종류별 반납 레일 |
| Partition | 하나의 Topic을 병렬 처리하도록 나눈 줄 | 같은 종류의 책을 나눈 여러 줄 |
| Consumer | Kafka에서 메시지를 가져가 처리하는 프로그램 | 반납 책을 가져가는 사서 |
| Consumer Group | 작업을 나누어 처리하는 Consumer 묶음 | 같은 업무를 나누어 맡은 사서 팀 |

Kafka는 최종 분석용 데이터베이스와 다릅니다.

- Kafka는 메시지가 처리될 때까지 기다리는 중간 구간에 가깝습니다.
- ClickHouse는 정리된 로그를 장기간 저장하고 검색·집계하는 서고에 가깝습니다.
- HyperDX는 ClickHouse에 저장된 로그를 사람이 검색하는 화면입니다.

호그와트 도서관 프로젝트에서는 서비스마다 지나치게 잘게 나뉜 Topic을 로그 유형과 처리 목적에 맞게 다시 구성했습니다. 파티션과 Consumer 수를 조절해 장 시작 트래픽에 대응할 여유도 마련했습니다.

우리에게 필요한 이해:

- 09:00 국내장과 해외장 개장 시점은 일반 시간대와 다른 피크 구간입니다.
- 중요한 로그가 덜 중요한 로그 때문에 늦어지지 않도록 우선순위가 필요합니다.

OpenBell Guard에는 Kafka를 설치하거나 Topic·Partition·Consumer를 설정하지 않습니다. 합성 JSONL·CSV 파일은 Kafka와 로그 플랫폼에서 안전하게 내보낸 분석 대상이라고 가정합니다. 따라서 이 프로젝트에서 필요한 이해는 `피크 시간에는 로그가 몰릴 수 있고, 로그 생성과 분석 처리는 분리되어야 한다`는 원칙까지입니다.

간단히만 알아도 되는 내용:

- Topic, Partition, Consumer Group의 세부 설정

### 5.3 Fluentd에서 OpenTelemetry로 처리기 전환

Fluentd는 여러 곳에서 발생한 로그를 읽고, 필요한 형태로 가공한 뒤, 저장소나 다른 시스템으로 보내는 오픈소스 로그 처리 프로그램입니다.

```text
Kafka에서 로그 읽기
       │
       ▼
로그 형식 파싱·필드 추가·불필요한 내용 제거
       │
       ▼
OpenSearch·S3 같은 목적지로 전달
```

도서관으로 비유하면 Kafka라는 반납 컨베이어에서 책을 가져와 종류와 라벨을 확인하고 올바른 서고로 옮기는 사서입니다.

Fluentd의 기본 동작은 다음 네 단계로 이해하면 됩니다.

| 단계 | 역할 | 도서관 비유 |
|---|---|---|
| Input | 파일·Kafka·애플리케이션 등에서 로그를 받습니다. | 컨베이어에서 책을 가져옵니다. |
| Parse·Filter | 로그를 읽을 수 있는 구조로 바꾸고 필드를 추가·제거합니다. | 책 정보와 라벨을 확인·수정합니다. |
| Buffer·Chunk | 여러 로그를 일정 크기의 묶음으로 모아 잠시 보관합니다. | 책을 운반 상자에 모읍니다. |
| Output | OpenSearch·S3·다른 시스템으로 보냅니다. | 상자를 올바른 서고나 창고로 옮깁니다. |

Kafka와 Fluentd의 차이는 `수동적인 대기`와 `능동적인 처리`입니다.

- Kafka는 로그를 순서대로 보관하고 Consumer가 가져가기를 기다립니다.
- Fluentd는 로그를 직접 가져와 파싱·필터링·묶음 처리한 뒤 목적지로 보냅니다.
- ClickHouse와 OpenSearch는 전달된 로그를 저장하고 검색하는 시스템입니다.

호그와트 도서관의 기존 구조에서는 서비스별 Kafka Topic에 맞추어 Fluentd도 여러 개로 나누어 운영했습니다. 한 서비스의 로그 폭증을 격리하는 데는 도움이 됐지만, 서비스가 늘면서 Fluentd Pod가 1,000개 이상으로 증가했고 어떤 Fluentd는 쉬는 반면 어떤 Fluentd는 과부하되는 자원 불균형이 발생했다고 설명합니다. Chunk를 모으는 과정에서 메모리 사용과 로그 처리 지연도 문제가 됐습니다.

새 구조에서는 많은 Fluentd 배포를 OpenTelemetry Collector 중심 구조로 바꾸고, 로그 유형에 따라 적절한 처리 경로로 라우팅했습니다. 피크 시간에는 중요 로그 처리 자원을 늘리고 디버그 로그 자원을 줄이는 방식도 사용했습니다.

OpenTelemetry Collector와 Fluentd는 일부 역할이 겹치지만 완전히 같은 제품은 아닙니다.

- Fluentd는 전통적으로 로그 수집·가공·전달과 다양한 플러그인 생태계에 강합니다.
- OpenTelemetry Collector는 로그뿐 아니라 메트릭·트레이스도 공통 관측 규격으로 처리하는 데 초점을 둡니다.
- 어느 쪽이 항상 우수한 것이 아니라 데이터 종류, 처리량, 운영 방식과 기존 환경에 따라 선택합니다.

우리 프로젝트에서는 이 아이디어를 다음처럼 번역합니다.

- 주문 경로는 고객 거래에 직접 영향을 주므로 가장 먼저 확인합니다.
- 시세·관심종목 경로는 투자 판단 영향으로 별도 분류합니다.
- 외부 브로커 경로는 내부 장애와 분리합니다.
- 디버그 로그가 많아도 고객 영향이 큰 오류를 우선 분석합니다.

OpenBell Guard에는 Fluentd나 OpenTelemetry Collector를 설치하지 않습니다. 사용자가 내보낸 JSONL·CSV를 Python이 직접 읽으므로, `로그를 입력받아 공통 필드로 정리하고 서비스 경로별로 분류한다`는 처리 원칙만 활용합니다.

### 5.4 OpenSearch에서 ClickHouse로 저장소 전환

시간·서비스·필드 중심 조회와 대규모 집계에 맞춰 컬럼형 데이터베이스인 ClickHouse를 선택했습니다. 자주 조회하는 필드는 미리 별도 컬럼으로 추출하고, 시간과 서비스 중심으로 정렬했습니다.

우리에게 필요한 이해:

- 자주 묻는 질문에 맞춰 데이터 구조를 정해야 합니다.
- 시간, 서비스, 고객 경로와 상태는 핵심 분석 필드입니다.

간단히만 알아도 되는 내용:

- Shard, Replica, Partition, Part
- Buffer·MergeTree 테이블 엔진
- 압축 Codec과 Bloom Filter 세부 설정

### 5.5 ssak3를 이용한 S3 아카이빙

최근 9일의 로그는 빠른 저장소에 두고 오래된 로그는 S3로 옮기는 구조입니다. 직접 만든 Python 도구 `ssak3`가 작업 상태와 실패 위치를 기록하고 재시도합니다.

우리에게 필요한 이해:

- 최근 사고 데이터와 장기 기준선 데이터는 사용 목적이 다릅니다.
- 실패한 작업은 처음부터 다시 하지 않고 중단 지점부터 이어갈 수 있어야 합니다.

우리 프로젝트에서 하지 않는 일:

- S3 업로드
- 아카이빙 스케줄러
- 운영 데이터 보관 정책 관리

### 5.6 Athena에서 ClickHouse S3 조회로 전환

장기 로그가 필요할 때만 AWS의 ClickHouse를 실행해 S3의 Parquet 파일을 조회합니다. 평소 사용하지 않는 고비용 자원을 항상 켜두지 않는 비용 최적화입니다.

우리 프로젝트에는 포함되지 않습니다. 알아둘 교훈은 `항상 켜두는 편의성`과 `필요할 때만 쓰는 비용 절감` 사이에 선택이 있다는 정도입니다.

### 5.7 HyperDX와 Grafana로 조회 역할 분리

- HyperDX: 개별 로그 검색과 사건 추적
- Grafana: CPU, 처리량, 오류율과 같은 메트릭 확인

OpenBell Guard도 로그와 메트릭을 구분합니다.

- 로그는 “무슨 일이 있었는가”를 보여줍니다.
- 메트릭은 “얼마나 많이, 얼마나 느리게, 언제부터 변했는가”를 보여줍니다.
- Codex는 두 증거를 결합해 가설을 작성합니다.

## 6. 플러그인에서 가장 중요한 개념

### 6.1 서비스 지연과 로그 유입 지연은 다릅니다

이 차이는 반드시 이해해야 합니다.

```text
사용자 요청 발생
    │
    ├─ 서비스가 응답하는 데 걸린 시간 = service latency
    │
    └─ 생성된 로그가 분석 플랫폼에 도착하는 데 걸린 시간 = ingestion lag
```

예를 들어 서비스가 0.2초 만에 정상 응답했지만 로그가 5분 뒤에 도착할 수 있습니다. 이 경우 고객 서비스가 느린 것이 아니라 관측 시스템이 늦은 것입니다.

반대로 로그는 즉시 도착했지만 서비스 응답이 3초 걸릴 수 있습니다. 이 경우 관측 시스템은 정상이고 서비스 경로가 느린 것입니다.

OpenBell Guard는 다음 두 시각을 사용합니다.

```json
{
  "event_time": "2026-03-03T09:00:02+09:00",
  "observed_time": "2026-03-03T09:05:02+09:00"
}
```

```text
ingestion_lag = observed_time - event_time = 5분
```

플러그인은 이 값을 계산해 서비스 장애와 로그 파이프라인 장애를 혼동하지 않아야 합니다.

### 6.2 처리량, 오류율과 지연시간

- 처리량: 초당 또는 분당 처리한 요청 수
- 오류율: 전체 요청 중 오류가 발생한 비율
- 지연시간: 요청 하나를 처리하는 데 걸린 시간
- 자원 포화: CPU·메모리·연결 수 등이 한계에 가까운 상태

09:00에 처리량이 증가하는 것 자체는 장애가 아닙니다. 처리량 증가와 함께 오류율이나 지연시간이 비정상적으로 상승하고 특정 서비스 경로에 고객 영향이 나타나는지 확인해야 합니다.

### 6.3 p50, p95와 p99

평균만 보면 일부 사용자가 겪는 심한 지연이 숨겨질 수 있습니다.

- p50: 전체 요청의 절반이 이 시간 안에 끝납니다.
- p95: 전체 요청의 95%가 이 시간 안에 끝납니다.
- p99: 전체 요청의 99%가 이 시간 안에 끝납니다.

예를 들어 요청 100개 중 99개가 1초 안에 끝나고 가장 느린 요청들이 5초에 가깝다면 p99는 고객이 겪는 꼬리 지연을 보여주는 지표가 됩니다.

플러그인은 Python으로 이 수치를 계산하고 Codex가 임의의 숫자를 만들지 못하게 해야 합니다.

### 6.4 로그 유형과 고객 서비스 경로

`log_type`은 로그가 어떤 기술에서 왔는지 나타냅니다.

```text
application, nginx, database, kafka, dependency
```

`service_path`는 고객이 어떤 기능에서 영향을 받는지 나타냅니다.

```text
authentication, market_data, watchlist, order, recurring_investment, external_broker
```

기술 로그 유형과 고객 경로를 함께 봐야 “DB 로그 오류가 주문 경로에 영향을 주었는가” 같은 질문을 다룰 수 있습니다.

## 7. OpenBell Guard는 정확히 어디까지 합니까?

```text
┌──────────────────────────────────────────────────────┐
│ 호그와트 도서관과 같은 관측 플랫폼                  │
│ 로그 생성·수집·대기·처리·저장·검색·장기 보관        │
└──────────────────────────┬───────────────────────────┘
                           │ 사용자가 안전하게 내보낸 파일
                           ▼
┌──────────────────────────────────────────────────────┐
│ OpenBell Guard                                       │
│ 정규화 → 마스킹 → 지표 계산 → 경로 분류 → 근거 종합 │
└──────────────────────────┬───────────────────────────┘
                           ▼
  타임라인·원인 가설·고객 영향·개선안·부하 테스트 계획
```

### 반드시 구현할 범위

- JSONL·JSON·CSV 입력
- 공통 필드 정규화
- 비밀정보 탐지와 마스킹
- 처리량·오류율·p50·p95·p99 계산
- `ingestion_lag_ms` 계산
- 서비스 경로와 외부 의존성 분류
- 사실·가설·미확인 사항 분리
- 합성 데이터와 기대 결과 자동 검증

### 개념만 이해하면 되는 범위

- OpenTelemetry가 로그를 표준화한다는 점
- Kafka가 로그를 잠시 보관하고 생산자와 처리자를 분리한다는 점
- ClickHouse가 시간·필드 집계에 적합하다는 점
- HyperDX는 로그, Grafana는 메트릭 확인에 사용된다는 점
- 최근 데이터와 장기 보관 데이터의 비용 구조가 다르다는 점

### 구현하지 않을 범위

- Kafka, ClickHouse와 Kubernetes 설치
- OpenTelemetry Collector 배포
- HyperDX·Grafana 대시보드 구축
- S3 아카이빙과 AWS 인프라
- 실제 카카오페이증권 시스템 연결
- 운영 환경 자동 증설 또는 설정 변경

## 8. Codex와 Python의 역할

### Python이 담당할 일

숫자와 규칙처럼 결과가 항상 같아야 하는 작업을 담당합니다.

- 파일 형식과 필수 필드 검사
- 타임스탬프 변환
- 관측 지연 계산
- 오류율과 지연시간 백분위수 계산
- 비밀정보 패턴 탐지
- 사고 구간과 기준 구간 비교

### Codex가 담당할 일

여러 증거의 의미를 연결하고 사람이 읽을 문서를 만드는 작업을 담당합니다.

- 시간순 사건 정리
- 서비스 경로별 고객 영향 설명
- 서로 경쟁하는 원인 가설 작성
- 가설을 지지하거나 반박하는 증거 연결
- 추가로 필요한 데이터 제안
- 재발 방지와 부하 테스트 계획 작성

Codex가 계산까지 임의로 수행하게 두면 같은 입력에서 다른 숫자가 나오거나 근거 없는 원인을 만들 수 있습니다. 따라서 `계산은 Python, 해석은 Codex`로 역할을 나눕니다.

## 9. 심사자에게 설명할 수 있어야 하는 문장

다음 정도를 자신의 말로 설명할 수 있으면 충분합니다.

> 카카오페이증권은 OpenTelemetry와 ClickHouse를 활용한 대규모 로그 플랫폼을 공개했습니다. OpenBell Guard는 그 인프라를 다시 만드는 플러그인이 아닙니다. 내보낸 로그와 메트릭을 시간·서비스 경로별로 분석하고, 서비스 지연과 로그 유입 지연을 구분해 근거가 연결된 사고 보고서와 재발 방지 계획을 생성하는 분석 계층입니다. 숫자는 Python이 계산하고 Codex는 사실과 가설을 분리해 해석합니다.

주의할 표현은 다음과 같습니다.

- 실제 사내 플랫폼과 직접 연동된다고 말하지 않습니다.
- 공개 보도된 장애가 로그 플랫폼 때문에 발생했다고 말하지 않습니다.
- 카카오페이증권에 AI 사고 분석 도구가 없다고 단정하지 않습니다.
- 회사가 공개한 성과 수치를 우리가 재검증했다고 말하지 않습니다.

## 10. 초급자 용어집

| 용어 | 쉬운 뜻 | 우리 프로젝트 중요도 |
|---|---|---|
| Backend | 사용자의 화면 뒤에서 요청과 데이터를 처리하는 프로그램 | 개념 이해 |
| Log | 프로그램이 남긴 사건 기록 | 필수 |
| Metric | 처리량·오류율처럼 시간에 따라 측정한 숫자 | 필수 |
| Telemetry | 로그·메트릭·트레이스 같은 관측 데이터 전체 | 필수 |
| Trace | 한 요청이 여러 서비스를 거친 경로를 연결한 기록 | 있으면 활용 |
| OpenTelemetry | 관측 데이터를 표준 방식으로 수집·전달하는 규격과 도구 모음 | 원칙만 활용 |
| Fluentd | 여러 곳의 로그를 읽고 가공해 저장소로 전달하는 로그 처리 프로그램 | 개념만 이해 |
| Kafka | 대량 메시지를 순서대로 보관하고 전달하는 시스템 | 개념만 이해 |
| Topic | Kafka 메시지를 용도별로 나누는 통로 | 간단히만 이해 |
| Consumer | Kafka에서 메시지를 가져가 처리하는 프로그램 | 간단히만 이해 |
| ClickHouse | 대량 데이터를 열 단위로 저장·집계하는 데이터베이스 | 선택 이유만 이해 |
| OpenSearch | 문장 속 단어 검색에 강한 검색·분석 시스템 | 비교 개념 |
| HyperDX | 관측 데이터를 검색하고 살펴보는 UI | 구현하지 않음 |
| Grafana | 메트릭을 차트와 대시보드로 보는 도구 | 구현하지 않음 |
| S3 | 파일을 저렴하게 장기 보관하는 클라우드 저장소 | 구현하지 않음 |
| Kubernetes | 여러 서버 프로그램을 배포·확장·복구하는 플랫폼 | 구현하지 않음 |
| Schema | 데이터에 어떤 필드가 있고 어떤 형식인지 정한 규칙 | 필수 |
| Normalization | 서로 다른 데이터 형식을 공통 형태로 맞추는 작업 | 필수 |
| Ingestion lag | 사건 발생 후 로그가 분석 시스템에 도착할 때까지의 지연 | 필수 |
| SLO | 서비스가 달성하기로 정한 신뢰성 목표 | 입력되면 활용 |
