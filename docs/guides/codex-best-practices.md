# OpenBell Guard를 위한 Codex 활용 모범 사례

> 문서 유형: OpenAI 공식문서 기반 프로젝트 특화 해설본  
> 대상 프로젝트: OpenBell Guard  
> 공식 원문: [Codex Best practices](https://developers.openai.com/codex/learn/best-practices)  
> 최종 확인일: 2026-06-28  
> 주의: Codex의 최신 명령과 설정은 공식 원문을 우선합니다.
> 프로젝트 지표·수식·오류 판정은 [단일 기준 계약](../openbell-guard-metrics-validation-contract.md)을 우선합니다.

## 1. 핵심 원칙

Codex를 한 번 질문하고 답을 받는 챗봇보다, 프로젝트 규칙과 검증 방법을 공유하는 개발 팀원처럼 사용합니다.

```text
명확한 작업 요청
→ 복잡하면 먼저 계획
→ 반복 규칙은 AGENTS.md
→ 반복 작업은 Skill
→ 외부 시스템은 필요한 경우에만 MCP
→ 테스트와 검토로 결과 확인
→ 충분히 안정된 작업만 자동화
```

## 2. 좋은 작업 요청의 네 요소

Codex에게 일을 맡길 때 다음 네 가지를 알려주면 범위 이탈과 불필요한 추측을 줄일 수 있습니다.

| 요소 | 질문 | OpenBell Guard 예시 |
|---|---|---|
| 목표 | 무엇을 만들거나 바꿉니까? | JSONL 텔레메트리 분석기 구현 |
| 맥락 | 어떤 파일과 자료가 중요합니까? | 기획서와 합성 fixture |
| 제약 | 무엇을 반드시 지켜야 합니까? | 원본 로그 불변, 실제 고객 데이터 금지 |
| 완료 조건 | 무엇이 확인되면 끝입니까? | 기대 통계 일치와 테스트 통과 |

권장 요청 예시는 다음과 같습니다.

```text
목표:
OpenBell Guard의 JSONL 텔레메트리 분석기를 구현해주십시오.

맥락:
docs/openbell-guard-research-plan.md와 합성 fixture를 기준으로 하십시오.

제약:
원본 입력은 수정하지 말고 Python 표준 라이브러리를 우선하십시오.
실제 고객 정보와 운영 주소를 사용하지 마십시오.

완료 조건:
서비스별 오류율과 장애 구간을 계산하고 관련 테스트가 통과해야 합니다.
실행한 검증 명령과 실제 결과를 보고하십시오.
```

## 3. 복잡한 작업은 계획부터 시작합니다

플러그인 전체 구현처럼 여러 파일과 설계 판단이 필요한 작업은 바로 코딩하지 않습니다.

OpenBell Guard에서는 다음 순서가 적절합니다.

1. 입력·출력 계약과 합성 데이터의 기대 결과를 정의합니다.
2. 플러그인 manifest와 Skill 골격을 만듭니다.
3. 입력 검증과 민감정보 마스킹을 구현합니다.
4. 결정론적 통계 분석기를 구현합니다.
5. Codex가 사실·가설·미확인을 나누도록 Skill을 작성합니다.
6. Golden fixture, 보안, 설치와 패키징을 검증합니다.
7. 실제 결과를 README와 질문 5문항에 반영합니다.

계획은 작업 도중 발견한 사실에 따라 갱신하되, 문제의 핵심 범위를 조용히 바꾸지 않습니다.

## 4. 반복 규칙은 `AGENTS.md`로 관리합니다

다음처럼 매번 반복하는 지시는 루트 `AGENTS.md`에 둡니다.

- 격식 있는 한국어로 보고합니다.
- `logs/` 원본을 수동 편집하지 않습니다.
- 실제 금융 주문이나 운영 인프라를 변경하지 않습니다.
- 공개 자료로 확인할 수 없는 내부 사실을 단정하지 않습니다.
- 실행하지 않은 테스트를 통과했다고 표현하지 않습니다.

현재 프로젝트에는 실제 적용 파일인 루트 [`AGENTS.md`](../../AGENTS.md)가 있습니다. 상세한 조사 내용은 그 파일에 모두 복사하지 않고 기획서를 참조하도록 유지합니다.

## 5. 권한과 설정은 필요한 만큼만 사용합니다

품질 문제처럼 보이는 현상이 잘못된 작업 폴더, 누락된 도구, 쓰기 권한 또는 지나치게 넓은 권한에서 발생할 수 있습니다.

- 작업 전에 현재 폴더와 대상 파일을 확인합니다.
- 실제 필요한 범위보다 넓은 권한을 상시 부여하지 않습니다.
- 운영 시스템이나 외부 서비스 변경은 별도의 명시적 승인이 없는 한 수행하지 않습니다.
- 비밀정보는 설정 파일, 문서, 대화와 테스트 데이터에 넣지 않습니다.

## 6. 구현만 하지 말고 검증까지 수행합니다

완료 흐름은 다음과 같습니다.

```text
구현 → 테스트 작성·수정 → 테스트 실행 → 결과 확인 → 변경사항 검토 → 문서 반영
```

OpenBell Guard의 핵심 검증은 다음과 같습니다.

- 고정 입력에서 오류율과 지연 통계가 기대값과 일치합니다.
- 시세 경로만 느릴 때 전체 주문장애로 오판하지 않습니다.
- 내부 문제와 외부 중개사 타임아웃을 구분합니다.
- 불완전한 데이터에서는 근본 원인을 확정하지 않습니다.
- 준비된 비밀정보가 마스킹된 출력에 남지 않습니다.
- 플러그인이 새 Codex 세션에서 발견되고 대표 시나리오가 실행됩니다.

## 7. MCP는 외부 연결이 실제로 필요할 때 사용합니다

MCP는 저장소 밖의 시스템과 Codex를 연결하는 규격입니다. 데이터가 자주 바뀌거나 반복해서 외부 도구를 호출해야 할 때 유용합니다.

OpenBell Guard MVP는 다음 이유로 MCP를 사용하지 않습니다.

- 별도 계정이나 API 키 없이 심사자가 재현할 수 있어야 합니다.
- 실제 사내 모니터링 시스템에 접근할 권한이 없습니다.
- 합성 로그와 로컬 스크립트만으로 핵심 가치를 검증할 수 있습니다.

실제 조직 적용 단계에서는 읽기 전용 Prometheus·Grafana 연동을 검토할 수 있지만, 필요성과 보안 모델이 확인된 후 추가합니다.

## 8. 반복 업무는 Skill로 만듭니다

Skill은 특정 업무의 표준 작업 절차입니다. `openbell-guard` Skill은 다음 한 가지 목표에 집중합니다.

> 익명화된 증권 서비스 텔레메트리를 분석해 장애 영향, 근거 기반 가설과 재발 방지 계획을 생성합니다.

Skill에는 다음을 명확히 적습니다.

- 언제 호출해야 하는지
- 필요한 입력 파일
- 실행할 검증·분석 스크립트
- 생성할 결과물
- 사실과 가설을 구분하는 방법
- 수행하면 안 되는 작업

처음부터 모든 예외를 처리하지 않고 대표 시나리오를 정확히 처리한 뒤 점진적으로 확장합니다.

## 9. 자동화는 Skill이 안정된 후에 검토합니다

- Skill은 작업 방법을 정의합니다.
- Automation은 작업 시점과 주기를 정의합니다.

해커톤 단계에서는 사람이 합성 데이터를 제공하고 Skill을 호출하는 흐름을 검증합니다. 반복 실행 결과가 충분히 예측 가능해진 후에만 정기 분석 자동화를 고려합니다.

## 10. 세션은 작업 단위로 관리합니다

한 세션에 프로젝트의 모든 일을 계속 쌓지 않습니다. 다음처럼 일관된 목표 단위로 나누는 편이 좋습니다.

- 문제 조사와 주제 선정
- 플러그인 스캐폴드와 Skill 설계
- 분석·마스킹 스크립트 구현
- 테스트와 보안 검증
- README·질문지·패키징 점검

같은 목표를 해결하는 중에는 문맥을 보존하고, 목표가 실제로 갈라질 때 새 세션을 사용합니다. 여러 세션이 같은 파일을 동시에 수정하게 하지 않습니다.

## 11. 피해야 할 실수

- 지속 규칙을 매번 긴 프롬프트로 반복합니다.
- 복잡한 작업에서 계획을 건너뜁니다.
- Codex가 실행할 테스트와 완료 기준을 알려주지 않습니다.
- 검증되지 않은 워크플로를 먼저 자동화합니다.
- 실제로 필요하지 않은 MCP와 의존성을 한꺼번에 추가합니다.
- 한 프로젝트 전체를 하나의 끝없는 세션으로 관리합니다.
- 테스트가 실패했는데 완료했다고 보고합니다.
- 합성 테스트 결과를 실제 운영 환경 검증이라고 표현합니다.
