# OpenBell Guard의 `AGENTS.md` 사용 가이드

> 문서 유형: OpenAI 공식문서 기반 프로젝트 특화 해설본  
> 대상 프로젝트: OpenBell Guard  
> 공식 원문: [Custom instructions with AGENTS.md](https://developers.openai.com/codex/guides/agents-md)  
> 최종 확인일: 2026-06-28  
> 주의: 정확한 탐색 순서와 설정 키는 공식 원문을 우선합니다.

## 1. `AGENTS.md`는 무엇입니까?

`AGENTS.md`는 Codex가 작업을 시작하기 전에 읽는 프로젝트 업무 규정집입니다.

사람이 프로젝트에 합류할 때 코딩 규칙, 테스트 방법, 수정 금지 영역과 완료 조건을 전달받는 것처럼 Codex에도 같은 정보를 제공합니다.

현재 프로젝트의 실제 지침 파일은 루트 [`AGENTS.md`](../../AGENTS.md)입니다.

## 2. 무엇을 할 수 있습니까?

- 응답 언어와 보고 방식을 정합니다.
- 프로젝트 목표와 디렉터리 구조를 알려줍니다.
- 빌드·테스트·검증 절차를 지정합니다.
- 로그, 개인정보와 운영 시스템에 대한 금지 규칙을 지정합니다.
- 무엇이 완료인지 정의합니다.
- 프로젝트 전반에서 반복되는 실수를 줄입니다.

`AGENTS.md` 자체가 테스트나 로그 수집을 실행하지는 않습니다. 지침을 실제로 수행하는 주체는 Codex이며, 자동 동작은 별도 스크립트나 훅이 담당합니다.

## 3. 이 프로젝트에서 필요한 이유

OpenBell Guard에는 일반 프로젝트보다 엄격한 조건이 있습니다.

- AX 해커톤 로그는 편집하거나 발췌하면 안 됩니다.
- 비밀정보가 대화와 로그에 남아서는 안 됩니다.
- 공개 근거로 확인되지 않는 기업 내부 사실을 단정하면 안 됩니다.
- 합성 장애와 실제 카카오페이증권 장애를 혼동하면 안 됩니다.
- 실제 주문과 운영 시스템에는 접근하면 안 됩니다.
- 플러그인·README·질문지·로그의 설명이 일치해야 합니다.

이 규칙을 매 세션마다 다시 입력하지 않도록 루트 `AGENTS.md`에 고정했습니다.

## 4. Codex가 파일을 찾는 방식

Codex는 대체로 다음 계층의 지침을 결합합니다.

```text
사용자 전역 지침
→ 프로젝트 루트 AGENTS.md
→ 현재 작업 디렉터리에 더 가까운 AGENTS.md 또는 AGENTS.override.md
```

- 전역 기본값은 Codex 홈 디렉터리에 둡니다.
- 저장소 공통 규칙은 프로젝트 루트에 둡니다.
- 특정 하위 작업 규칙은 필요할 때 하위 디렉터리에 둡니다.
- 같은 위치에 `AGENTS.override.md`가 있으면 일반 파일보다 우선합니다.
- 세션 시작 시 지침을 읽으므로 변경 후에는 새 세션에서 확인하는 것이 안전합니다.

실격 방지와 보안처럼 항상 필요한 규칙은 하위 파일에만 두지 않고 루트에 유지합니다.

## 5. 파일별 역할을 구분합니다

| 파일 | 대상 | 역할 |
|---|---|---|
| `AGENTS.md` | 프로젝트를 개발하는 Codex | 개발·검증·안전 규칙 |
| `SKILL.md` | 설치된 플러그인을 사용하는 Codex | 실제 로그 분석 워크플로 |
| `README.md` | 사용자와 심사자 | 문제, 설치, 사용법, 검증 결과 |
| `plugin.json` | Codex 플러그인 시스템 | 플러그인 식별 및 구성요소 선언 |
| `docs/openbell-guard-metrics-validation-contract.md` | 개발자·Codex·테스트 | 모든 지표·수식·오류 판정의 단일 기준 |
| 로그 훅·수집기 | 실행 환경 | 대화 원본 자동 저장 |

예를 들어 “코드를 바꾸면 테스트하십시오”는 `AGENTS.md`에 적합합니다. “로그를 마스킹하고 장애 구간을 계산하십시오”는 플러그인 `SKILL.md`에 적합합니다.

## 6. 현재 루트 지침의 핵심

현재 [`AGENTS.md`](../../AGENTS.md)는 다음을 규정합니다.

- 모든 답변을 격식 있는 한국어로 작성합니다.
- 카카오페이증권의 개장 시점 서비스 복원력 문제에 집중합니다.
- 공개 근거와 추론을 구분합니다.
- `logs/`의 기존 파일을 수동 편집하지 않습니다.
- 실제 주문, 운영 변경과 운영 환경 부하 테스트를 금지합니다.
- 플러그인 파일은 `src/` 안에 배치합니다.
- 사실·원인 가설·추가 확인 필요를 분리합니다.
- 지표·수식·임계치·오류 코드는 단일 계약의 M-ID와 issue code를 참조하고 다른 파일에서 다시 정의하지 않습니다.
- 테스트와 제출물 정합성까지 확인해야 완료로 판단합니다.
- 로컬 Worklog·Decisionlog를 먼저 기록하고 AX Notion 포트폴리오를 동기화합니다.
- Techlog의 채택 후보는 변경안·비용·검증 영향을 먼저 사용자에게 보고하고 승인받은 뒤 기획서에 반영합니다.
- 구현 작업은 계약과 공용 인터페이스를 먼저 고정한 뒤 파일 소유권이 겹치지 않는 범위만 서브에이전트에게 분배합니다.

## 7. 좋은 지침을 유지하는 방법

- 추상적인 구호보다 실행 가능한 규칙을 적습니다.
- 상세 참고자료를 모두 복사하지 않고 필요한 문서를 링크합니다.
- 실제로 반복되는 실수나 마찰이 생길 때 규칙을 추가합니다.
- 새 외부 자료의 아이디어를 바로 기준선에 넣지 않고 검토 문서와 사용자 승인 단계를 거칩니다.
- 더 이상 맞지 않는 명령과 경로는 즉시 수정합니다.
- 서로 충돌하는 규칙을 여러 위치에 만들지 않습니다.
- 지표 의미를 바꿀 때는 단일 계약과 Decisionlog를 먼저 갱신하고 사용자 승인을 받은 뒤 fixture·코드·문서를 함께 바꿉니다.
- API 키, 토큰과 실제 계정정보를 넣지 않습니다.
- 파일 크기 제한을 고려해 짧고 정확하게 유지합니다.

## 8. 새 세션에서 확인할 사항

1. 프로젝트 루트에서 새 Codex 세션을 시작합니다.
2. 현재 지침과 작업 범위를 요약해달라고 요청합니다.
3. 로그 편집 금지, 실제 운영 접근 금지와 완료 기준이 포함되는지 확인합니다.
4. 하위 디렉터리에서 시작했다면 예상하지 않은 override가 없는지 확인합니다.

## 9. 언제 수정해야 합니까?

- 실제 테스트 명령이 확정됐을 때
- 플러그인 구조나 이름이 공식적으로 변경됐을 때
- Codex가 같은 유형의 실수를 반복했을 때
- 새로운 보안·제출 규칙이 생겼을 때
- 더 이상 존재하지 않는 파일이나 명령을 참조할 때

지침을 변경하면 기획서·README·질문지와 충돌하지 않는지도 함께 확인합니다.

## 10. Notion 기록에는 어떻게 사용합니까?

루트 `AGENTS.md`는 프로젝트 요청 한 단위가 완료될 때 로컬 기록과 Notion을 함께 갱신하도록 규정합니다.

```text
원본 대화 → logs/
작업 사실 → Worklog.md
중요 결정 → Decisionlog.md
포트폴리오 요약 → Notion MyProject·Study DB
```

- 일반 작업은 현재 MyProject Phase에 W-ID로 추가합니다.
- 중요 결정은 관련 Phase의 D-ID 하위 페이지로 만듭니다.
- 새 기술이나 공개 자료 학습만 Study DB에 기록합니다.
- 원본 대화 전문과 비밀정보는 Notion에 넣지 않습니다.
- Notion 연결이 없으면 로컬 기록을 완료하고 동기화 대기로 보고합니다.
- 생성·수정 후 페이지를 다시 읽어 중복과 누락을 확인합니다.

대상 페이지, 데이터 소스와 상세 절차는 [Notion 기록 운영 가이드](../notion-recording-guide.md)를 따릅니다.

이 규칙은 프로젝트 루트의 `AGENTS.md`를 읽는 Codex 세션에 적용되며 백그라운드 작업이나 다른 AI 도구에 자동으로 전파되지는 않습니다.

## 11. 서브에이전트 규칙에는 어떻게 사용합니까?

루트 `AGENTS.md`는 메인 에이전트가 최종 통합 책임을 유지하면서 필요할 때만 최대 세 명의 서브에이전트를 사용하도록 규정합니다.

- 계약·스키마를 먼저 고정합니다.
- 한 파일은 한 시점에 한 에이전트만 수정합니다.
- 목표, 허용 파일, 금지 사항과 검증 명령을 작업 브리프에 적습니다.
- 구현 에이전트와 최종 검증 에이전트를 가능한 한 분리합니다.
- Worklog·Decisionlog·Notion은 메인 에이전트만 갱신합니다.
- 서브에이전트 결과는 메인 에이전트가 파일과 테스트로 다시 확인한 뒤 수용합니다.

역할 풀, WP-01~WP-12, 파일 소유권과 핸드오프 양식은 [서브에이전트 운영·구현 분담 가이드](../subagent-operating-guide.md)를 따릅니다.
