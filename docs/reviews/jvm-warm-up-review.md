> 구현 주의: 이 검토의 지표 언급은 조사 절차 설명이며 실제 계산·오류 판정은 [단일 기준 계약](../openbell-guard-metrics-validation-contract.md)을 따릅니다.

## 1. 결론부터 설명합니다

이 글에서 OpenBell Guard에 가장 가치 있는 부분은 `JVM 웜업`이라는 해결 기술이 아니라 **눈에 보이는 오류를 근본 원인으로 단정하지 않고 여러 가설을 메트릭으로 반증한 조사 방식**입니다.

OpenBell Guard에는 다음 원칙을 반영할 가치가 있습니다.

- 오류 로그를 `확인된 증상`으로 먼저 기록하고 곧바로 근본 원인으로 승격하지 않습니다.
- 장애 직전의 배포·설정 변경·증설 같은 변화가 있었는지 별도 증거로 확인합니다.
- 오류율 하나가 아니라 응답 시간, 자원 사용량, 커넥션 사용·획득 시간과 외부 의존성 지연을 같은 시간축에서 비교합니다.
- 임시 완화책과 검증된 근본 조치를 구분합니다.
- 채택하지 않은 가설과 그 반대 근거도 사고 보고서에 남깁니다.

반대로 JVM 웜업 코드, HikariCP 설정, Spring `ApplicationRunner`, Kubernetes `startupProbe`와 배포 파이프라인은 OpenBell Guard MVP에 넣지 않습니다. 현재 플러그인은 Python 기반의 읽기 전용 분석 도구이며 실제 운영 애플리케이션을 배포하거나 변경하지 않기 때문입니다.

### 1.1 여기서 `MVP 반영`은 무슨 뜻입니까?

MVP의 일반적인 의미는 [플러그인 제작 가이드의 MVP 설명](../guides/plugin-build-guide.md#먼저-알아둘-말-mvp)에서 자세히 설명합니다.

이 검토에서 `MVP 반영`이라고 한 것은 **JVM 웜업을 구현한다는 뜻이 아닙니다.** OpenBell Guard의 첫 제출 가능 버전에서 다음 사항을 실제 보고서 형식과 테스트 기준에 넣는다는 뜻입니다.

- 타임아웃 로그를 곧바로 근본 원인으로 확정하지 않습니다.
- `확인된 증상`과 `원인 가설`을 다른 항목으로 출력합니다.
- 원인 가설마다 지지 근거, 반대 근거와 경쟁 가설을 요구합니다.
- 재발 방지 계획에서 임시 완화책과 근본 조치를 구분합니다.

즉, 글에서 배운 **분석 방법은 첫 버전에 포함**하고, Java·Spring·Kubernetes 같은 **백엔드 구현은 포함하지 않는다**는 의미입니다.

## 2. 출처 범위를 먼저 구분해야 합니다

이 게시물은 카카오페이증권 DevOps 팀이 작성한 호그와트 도서관 글과 달리 **카카오페이의 굿딜 결제 서비스 개발 사례**입니다. 따라서 다음과 같이 사용 범위를 제한합니다.

- 카카오페이 계열 Techlog에서 다룬 문제 해결 방식이라는 참고 자료로 사용할 수 있습니다.
- 고트래픽 금융 서비스의 배포 직후 지연을 분석한 공개 사례로 사용할 수 있습니다.
- 카카오페이증권이 JVM, HikariCP, Spring 또는 같은 배포 방식을 사용한다는 근거로는 사용할 수 없습니다.
- 공개 보도된 카카오페이증권의 개장 지연이 JVM 콜드 스타트 때문이었다고 주장할 수 없습니다.

이 구분을 지키지 않으면 기업 연관성을 강조하려다 오히려 출처의 범위를 과장하게 됩니다.

## 3. 게시물은 어떤 문제를 해결했습니까?

게시물의 흐름을 초급자 관점에서 단순화하면 다음과 같습니다.

1. 배포 직후 DB 커넥션을 얻지 못했다는 타임아웃 오류와 응답 지연이 나타났습니다.
2. 처음에는 커넥션 풀이 부족하다고 보고 크기와 대기시간 설정을 바꿨지만 문제가 해결되지 않았습니다.
3. 새 서버로 들어오는 트래픽을 점진적으로 늘리는 방식은 잠시 완화 효과가 있었지만 트래픽이 커지자 다시 문제가 나타났습니다.
4. DB 쿼리 시간은 정상인데 배포 직후에만 커넥션 사용·획득 시간이 늘고 CPU·메모리·GC 지표가 함께 변하는 것을 확인했습니다.
5. 첫 사용자 요청을 처리하는 동안 필요한 JVM 클래스가 늦게 로딩되어 커넥션 점유 시간이 길어지는 가설을 세웠습니다.
6. 사용자가 들어오기 전에 호출량이 많은 로직을 미리 실행하고, 완료된 뒤에만 트래픽을 받도록 웜업 절차를 넣었습니다.
7. 적용 전후의 커넥션·응답시간 지표와 클래스 로딩 로그를 확인해 가설을 뒷받침했습니다.

핵심 교훈은 `DB Connection Timeout`이라는 오류 이름이 실제 근본 원인을 말해 주는 것은 아니라는 점입니다. 이 사례에서 해당 오류는 커넥션 풀 자체 부족보다 애플리케이션 초기화 지연이 뒤에서 만들어 낸 결과였습니다.

## 4. OpenBell Guard와 맞닿는 지점

OpenBell Guard의 현재 설계는 이미 결과를 `확인된 사실`, `원인 가설`, `추가 확인 필요`로 분리하고 지지·반대 근거를 요구합니다. 이 글은 그 설계가 실제 운영 문제 해결 방식과 맞닿아 있음을 보여주는 보조 사례입니다.

| 게시물의 문제 해결 방식 | OpenBell Guard 적용 | 적합성 |
|---|---|---|
| 오류 메시지를 근본 원인이 아닌 증상으로 취급 | 로그 오류를 `관측 증상`으로 기록하고 원인 가설과 분리 | 높음 |
| 설정 변경 후 문제가 남는지 확인 | 채택·기각한 가설과 반대 근거 기록 | 높음 |
| 배포 직후와 안정 구간의 메트릭 비교 | 기준 구간과 장애 구간의 오류율·지연·자원 지표 비교 | 높음 |
| 임시 트래픽 완화와 근본 조치 구분 | 완화책·근본 조치·검증 조건을 각각 출력 | 높음 |
| 배포·웜업 완료 시점을 타임라인에 표시 | 향후 선택 입력인 변경 이벤트로 검토 | 중간 |
| JVM 클래스 로딩·JIT 분석 | Python 기반 범용 장애 분석에는 직접 불필요 | 낮음 |
| Kubernetes probe와 배포 파이프라인 변경 | 읽기 전용 플러그인의 안전 범위 밖 | 매우 낮음 |

## 5. 반영할 내용

### 5.1 원인 가설 형식 강화

`hypotheses.md`에는 각 후보 원인마다 최소 다음 항목을 요구합니다.

- 관측된 증상
- 원인 가설
- 지지 근거
- 반대 근거
- 경쟁 가설
- 판단에 필요한 추가 데이터
- 신뢰도

예를 들어 `DB 커넥션 타임아웃 발생`은 확인된 증상일 수 있지만, `DB 커넥션 수 부족`은 메트릭과 설정 변경 결과가 뒷받침되기 전까지 원인 가설입니다.

### 5.2 완화책과 근본 조치 분리

`remediation-plan.md`에서 다음을 구분합니다.

- 즉시 완화: 고객 영향을 줄이기 위한 임시 조치
- 근본 조치: 검증된 원인을 제거하는 변경
- 검증 조건: 적용 전후 어떤 지표가 어떻게 달라져야 성공인지
- 롤백 조건: 예상과 다른 결과가 나타날 때 중단할 기준

이 구분은 트래픽을 일시적으로 줄여 현상을 완화했지만 재발한 게시물의 사례와 직접 연결됩니다.

### 5.3 변경 이벤트는 후순위 선택 기능으로 검토

핵심 MVP가 완성된 뒤 다음과 같은 선택 입력을 추가할 수 있습니다.

```csv
event_time,event_type,service_name,change_id
2026-06-29T08:55:00+09:00,deployment,synthetic-order-api,release-42
2026-06-29T08:58:00+09:00,traffic_enabled,synthetic-order-api,release-42
```

이를 통해 장애가 배포 직후에만 나타났는지, 시장 개장 트래픽과 배포가 겹쳤는지 확인할 수 있습니다. 그러나 현재 핵심 시나리오는 개장 피크·외부 중개사 장애이므로 MVP 구현을 늦추면서까지 먼저 넣지는 않습니다.

## 6. 구현하지 않을 내용

다음 요소는 기업 기술을 보여 주기 위한 장식으로도 넣지 않습니다.

- Java·Kotlin·Spring 애플리케이션
- HikariCP 커넥션 풀 튜닝
- JVM 클래스 로딩 로그 수집기
- 실제 서비스 로직을 호출하는 웜업 API
- Kubernetes `startupProbe`, `readinessProbe`, `livenessProbe`
- 블루·그린 또는 롤링 배포 자동화
- JIT 임계값 조정과 GraalVM Native Image

이 기능들은 실제 서비스 코드와 런타임, 데이터베이스, Kubernetes 환경이 있어야 의미 있게 검증할 수 있습니다. 합성 로그만으로 흉내 내면 기술 이름은 늘어나지만 문제 해결력이나 심사 재현성은 좋아지지 않습니다.

## 7. 구현 비용 추정

아래 비용은 현재 저장소에 아직 `src/` MVP가 없는 상태에서, AI 도움을 받아 구현하고 테스트까지 완료한다는 전제의 대략적인 범위입니다. 실제 소요시간을 측정한 값이나 완료 약속은 아닙니다.

| 후보 | 예상 비용 | 검증 난이도 | 판단 |
|---|---:|---:|---|
| 가설 출력에 증상·반대 근거·경쟁 가설 추가 | 낮음, 약 1~2시간 | 낮음 | MVP에 반영 |
| 완화책·근본 조치·검증 조건 분리 | 낮음, 약 1~2시간 | 낮음 | MVP에 반영 |
| 오해를 유도하는 타임아웃 로그 합성 테스트 | 중간, 약 0.5~1일 | 중간 | 핵심 기능 후 검토 |
| 변경 이벤트 CSV 파싱과 배포 전후 비교 | 중간, 약 1~2일 | 중간 | 후순위 선택 기능 |
| Java/Spring 웜업 데모와 Kubernetes probe 재현 | 높음, 최소 3~7일 이상 | 높음 | 제외 |
| 실제 운영 배포 파이프라인 연동 | 산정 불가 | 매우 높음 | 안전 범위 밖 |

가장 높은 효과를 가장 낮은 비용으로 얻는 방법은 새로운 백엔드 스택을 추가하는 것이 아니라, 기존 Python 분석 결과와 Codex 보고서의 **가설 검증 형식**을 강화하는 것입니다.

## 8. 검증 방법

반영 효과는 다음 질문으로 확인합니다.

1. 타임아웃 오류만 주어졌을 때 플러그인이 용량 부족을 근본 원인으로 확정하지 않습니까?
2. 장애 구간과 정상 기준 구간의 메트릭을 나누어 비교합니까?
3. 지지 근거뿐 아니라 가설과 맞지 않는 반대 근거도 출력합니까?
4. 임시 완화책과 근본 조치를 구분합니까?
5. 배포 시점이나 클래스 로딩 데이터가 없으면 JVM 웜업을 권고하지 않고 `추가 확인 필요`로 남깁니까?

이 검증은 카카오페이증권의 실제 장애 원인을 재현하는 것이 아니라, 합성 데이터에서 과도한 인과 단정을 막는 품질 검증입니다.

## 9. 최종 판단

이 게시물은 **참고만**으로 분류합니다.

- 채택: 증상과 원인 분리, 경쟁 가설, 반대 근거, 적용 전후 메트릭 비교, 완화책과 근본 조치 구분
- 후순위: 변경 이벤트 입력과 배포 직후 비교
- 제외: JVM 웜업, Spring, HikariCP, Kubernetes와 배포 자동화 구현

이 판단은 기술을 억지로 붙이지 않으면서도 카카오페이 Techlog가 보여 준 실제 문제 해결 태도를 OpenBell Guard의 분석 품질에 반영하는 선택입니다.
