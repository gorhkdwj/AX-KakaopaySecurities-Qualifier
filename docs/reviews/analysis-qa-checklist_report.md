## 2. 이미 잘 설계된 부분

### 2.1 문제와 기업의 연결

- 2026년 2월 19일·27일과 3월 3일 개장 직후 지연, 3월 4일 외부 중개사 장애가 서로 다른 장애 유형으로 구분돼 있습니다.
- 회사가 주문은 정상 접수됐다고 설명한 사건에서 시세·관심종목 경로를 전체 주문장애로 확대하지 않습니다.
- 공식 Techlog의 하루 41TB·200억 건 이상 로그와 장 시작 트래픽 스파이크를 실제 카카오페이증권 운영 특성으로 사용하되, 공개 장애의 근본 원인으로 연결하지 않습니다.
- 공식 채용 공고의 로그·패킷·메트릭 분석, 지연 구간 식별과 재발 방지 업무가 플러그인 사용자·기능과 잘 맞습니다.

### 2.2 안전한 판단 원칙

- 확인된 사실, 관측된 증상, 원인 가설과 추가 확인 필요 사항을 분리합니다.
- 내부 장애와 외부 의존성 장애를 구분합니다.
- 합성 fixture의 원인을 실제 카카오페이증권 사건의 원인처럼 표현하지 않습니다.
- 실제 주문·운영 변경·운영 주소 부하 테스트를 수행하지 않는 읽기 전용 경계가 명확합니다.

### 2.3 현실적인 기술 선택

- Skill과 로컬 Python 스크립트를 중심으로 하고 MCP·외부 인증·대규모 백엔드 인프라를 제외한 선택은 해커톤 재현성에 적합합니다.
- JSONL·CSV·텍스트의 행 단위 처리와 큰 JSON 배열의 사전 차단은 낮은 비용으로 실패를 예측 가능하게 만듭니다.
- Spring Batch·MongoDB·Kafka·ClickHouse를 억지로 구현하지 않고 설계 원칙만 선별한 판단이 타당합니다.

## 3. 구현 전에 반드시 보완할 P0 항목

### P0-1. 대표 사용자 흐름을 하나로 동결해야 합니다

**현재 상태**

- 장 시작 전 준비 점검, 장애 중 분석, 장애 후 보고와 외부 장애 분석을 모두 1차 사용 시점으로 제시합니다.
- 일곱 개 Markdown 산출물을 모두 기본 출력으로 계획합니다.

**문제**

MCP나 실시간 관측 시스템 연결이 없는 플러그인은 사용자가 내보낸 파일을 분석하는 도구입니다. 따라서 `장애 발생 중 신속한 실시간 분석`을 대표 가치로 내세우면 실제 구현과 기대가 어긋날 수 있습니다. 일곱 개 문서를 모두 생성하면 템플릿·검증·정합성 비용도 커집니다.

**권고안**

- 대표 흐름을 `익명화된 incident bundle 경로 입력 → 결정론적 분석 → 근거가 연결된 사후 분석 보고서`로 고정합니다.
- 장애 중 사용은 `사용자가 최신 export 파일을 제공한 경우 가능한 보조 흐름`으로 낮춥니다.
- 기본 산출물을 다음 세 개로 축소합니다.
  - `analysis.json`: 계산 결과, 근거와 판정의 기계 판독 원본
  - `openbell-report.md`: 타임라인·영향·가설·개선안을 합친 사람용 보고서
  - `sanitization-report.md`: 값 자체를 노출하지 않는 입력 안전성 보고서
- 고객 공지와 부하 테스트 계획은 기본 파일이 아니라 보고서의 선택 절 또는 사용자 요청 시 파생 출력으로 둡니다.

**완료 조건**

한 개의 대표 프롬프트와 한 개의 fixture로 입력부터 위 세 산출물까지 끝나는 데모 경로가 기획서에 명시돼야 합니다.

### P0-2. 입력 계약을 파일 단위로 고정해야 합니다

**현재 상태**

JSONL·JSON·텍스트·CSV·YAML·Markdown을 폭넓게 받지만 로그·메트릭·서비스 구성도의 필수 필드가 완전히 정의되지 않았습니다. 로그와 메트릭은 필수로 표시하면서 불완전 데이터 시나리오도 허용해 `실패`와 `제한 분석`의 경계가 불명확합니다.

**권고안**

MVP 핵심 입력은 다음처럼 줄이는 편이 안전합니다.

| 파일 | MVP 형식 | 핵심 내용 |
|---|---|---|
| `incident.json` | JSON 객체 | 사건 ID, 기준·사고 구간, 기본 시간대, 분석 모드 |
| `logs.jsonl` | JSONL | 시각, 서비스, 경로, 상태, 지연, 외부 의존성, 메시지 |
| `metrics.csv` | CSV | 시각, 서비스, 지표명, 값, 단위 |
| `service-map.json` | JSON | 합성 서비스와 내부·외부 경로 매핑 |

- YAML과 자유 형식 Markdown·텍스트는 MVP 핵심 입력에서 빼거나 Python 의존성을 명시해야 합니다.
- ISO 8601과 UTC offset을 요구하고 시간대가 없으면 추정하지 않아야 합니다.
- 로그 또는 메트릭 하나가 없을 때는 실패가 아니라 `degraded` 상태로 결과를 만들지, 분석을 중단할지 규칙을 정해야 합니다.
- SLO가 없으면 임의 기준을 만들지 않고 `SLO 비교 불가`로 표시해야 합니다.

**완료 조건**

각 파일의 필수·선택 필드, 자료형, 허용값, 누락 처리, 오류 예시가 하나의 입력 계약 표 또는 JSON Schema에 존재해야 합니다.

### P0-3. 통계와 장애 구간 계산을 재현 가능하게 정의해야 합니다

**현재 상태**

처리량·오류율·p50·p95·p99, 기준 구간 비교와 장애 시작·회복 시각을 계산한다고 되어 있지만 계산식과 판정 규칙이 없습니다.

**문제**

라이브러리와 보간법에 따라 백분위수가 달라질 수 있으며, `행 단위 입력`만으로는 정확한 백분위수 계산의 메모리가 제한되지 않습니다. 모든 `latency_ms`를 보관하면 입력 전체 크기에 따라 메모리가 증가합니다.

**권고안**

- 시간 버킷 크기, 처리량 단위, 오류로 분류할 상태와 분모가 0일 때의 규칙을 고정합니다.
- 백분위 계산법을 예를 들어 `nearest-rank`처럼 하나로 고정하고 최소 표본 수를 정합니다.
- 기준 구간과 사고 구간은 `incident.json`에서 명시적으로 받습니다.
- 장애 시작·회복은 어떤 지표가 몇 개 버킷 연속 임계치를 넘고 내려와야 하는지 정의합니다.
- 정확한 백분위수를 유지할 경우 Phase 4 측정으로 **고정된 지원 범위**를 정하고 초과 입력을 명시적으로 거부합니다. 사용자 조정형 일반 한도까지 만들 필요는 없지만, 무제한 지원처럼 보이게 두어서는 안 됩니다.

**완료 조건**

같은 입력에 대해 구현자가 바뀌어도 동일한 숫자와 장애 구간이 나오는 계산 명세와 Golden 기대값이 있어야 합니다.

### P0-4. 원본 데이터와 Codex 사이의 보안 경계를 바로잡아야 합니다

**현재 상태**

`로컬 파일만 처리하며 기본 설정에서 네트워크 전송을 하지 않는다`고 적혀 있습니다.

**문제**

Python 스크립트가 외부 API를 호출하지 않는 것과 Codex가 파일 내용을 모델 문맥에서 처리하는 것은 다른 문제입니다. 현재 문장은 원본 로그가 어떤 환경에서도 외부로 전달되지 않는다는 보장처럼 읽힐 수 있습니다.

**권고안**

- 사용자는 원본 내용이 아니라 **파일 경로**를 플러그인에 제공합니다.
- 결정론적 스크립트가 원본을 로컬에서 검사하고 마스킹된 파생본과 `analysis.json`을 만듭니다.
- `SKILL.md`는 Codex가 마스킹 전 원본 파일을 직접 열거나 인용하지 못하도록 명시합니다.
- Codex는 마스킹된 분석 결과와 허용된 근거 조각만 읽습니다.
- `네트워크 전송 없음` 대신 `번들 스크립트는 외부 API를 호출하지 않으며 원본을 모델 입력으로 사용하지 않도록 설계한다`고 표현합니다.
- 마스킹 보고서에도 원래 토큰·계좌번호·개인정보 값을 남기지 않습니다.

**완료 조건**

원본 파일을 Codex가 읽지 않아도 전체 대표 시나리오가 동작하고, Seeded secret이 파생 입력·분석 JSON·Markdown 출력 어디에도 남지 않아야 합니다.

### P0-5. 사실 근거 연결을 기계가 검사할 수 있어야 합니다

**현재 상태**

모든 사실 문장에 파일 위치 또는 시각 근거를 연결한다고 되어 있으나, Markdown 문장과 근거를 연결하는 중간 스키마가 없습니다. 원인 가설의 `신뢰도` 기준도 정의되지 않았습니다.

**권고안**

`analysis.json`에 최소한 다음 구조를 둡니다.

```json
{
  "claims": [
    {
      "claim_id": "C-001",
      "claim_type": "confirmed_fact",
      "text": "시세 경로의 p99 지연이 사고 구간에 증가했습니다.",
      "evidence_refs": ["E-014", "E-015"]
    }
  ],
  "evidence": [
    {
      "evidence_id": "E-014",
      "source_file": "metrics.csv",
      "source_location": "rows 120-180",
      "time_window": "2026-03-03T09:00:00+09:00/2026-03-03T09:05:00+09:00"
    }
  ]
}
```

- Markdown 보고서는 이 ID를 표시하거나 내부적으로 보존해야 합니다.
- 사실 claim은 하나 이상의 근거가 없으면 생성하지 못하게 합니다.
- 가설은 지지·반대 근거와 추가 필요 데이터가 있어야 합니다.
- 신뢰도는 숫자처럼 정밀하게 보이지 않도록 `높음·중간·낮음·판단 불가`와 판정 규칙을 정의하거나 MVP에서 제거합니다.

**완료 조건**

`validate_bundle.py`가 근거 없는 사실 claim, 존재하지 않는 evidence ID와 비밀정보 잔존을 실패로 판정해야 합니다.

### P0-6. 플러그인 이름·설치 검증 규칙의 충돌을 수정해야 합니다

**현재 상태**

제출물에서 플러그인 루트 디렉터리는 `src/`이지만 `AGENTS.md`와 제작 가이드는 manifest `name`이 플러그인 디렉터리 이름과 같아야 한다고 적고 있습니다.

**공식문서 대조 결과**

현재 OpenAI 공식 Build plugins 문서는 manifest `name`을 안정적인 kebab-case 플러그인 식별자로 사용하라고 설명하며, 루트 디렉터리명과 반드시 같아야 한다고 요구하지 않습니다. 따라서 제출 폴더명이 `src`여도 manifest 이름은 `openbell-guard`를 유지할 수 있습니다.

**권고안**

- `name`과 디렉터리명이 같아야 한다는 로컬 규칙을 삭제합니다.
- `plugin.json`의 `name=openbell-guard`, `version`, `description`, `skills=./skills/`를 실제 파일과 대조합니다.
- 개발용 Marketplace에서 `src/`를 가리키는 설치 경로를 만들고, 재시작 후 새 스레드에서 발견·자동 선택·명시 호출을 각각 검증합니다.
- Marketplace 파일은 개발 도구이며 제출 ZIP 요구 구조와 분리합니다.

**완료 조건**

공식 스캐폴드·manifest 검증과 새 Codex 세션 설치 테스트가 모두 통과하고, 제출 ZIP 내부에서는 `src/.codex-plugin/plugin.json`이 유지돼야 합니다.

## 4. P1 보완 권고

1. **구현 상태 문구 정정**  
   점검 당시 기획서의 채택 경계 표는 `src/`가 없는데도 `코드·출력 계약·테스트에 구현`이라고 표현했습니다. 2026-06-30에 `설계 반영·Phase 4 구현 예정`으로 바로잡았습니다.

2. **대표 데모와 회귀 테스트 분리**  
   시나리오 A와 B를 심사용 대표 데모로 두고 C~H는 안전·오판 방지 회귀 테스트로 분류하면 발표 흐름이 선명해집니다.

3. **명령·실패 계약 추가**  
   대표 사용자 프롬프트, 스크립트 호출 예시, 출력 디렉터리, 종료 코드와 주요 오류 메시지를 미리 정의해야 합니다.

4. **고객 공지의 인간 검토 표시**  
   고객 공지는 항상 `초안`으로 표시하고 법적 책임·보상 여부를 확정하지 않으며 승인 전 외부 발송하지 않는다고 출력 계약에 명시해야 합니다.

5. **핵심 사건 근거의 출처 등급 표시**  
   현재 사건 타임라인은 회사 설명을 인용한 복수 언론 보도로 검증됩니다. 공개된 회사 공지 원문을 찾지 못하면 `회사 공식 원문 미확보·언론의 회사 설명 인용`이라고 출처 등급을 명시하는 편이 안전합니다.

6. **성능 주장의 기준 마련**  
   `신속한 분석` 또는 생산성 향상은 아직 검증되지 않았습니다. Phase 4에서 입력 규모, 실행 환경, 실행시간과 peak memory를 측정하고 그 범위 안에서만 주장해야 합니다.

## 7. Codex 공식 규격 대조 결과

- 플러그인 루트의 `.codex-plugin/plugin.json`과 Skill 디렉터리 구조는 공식 가이드와 맞습니다.
- manifest의 안정적인 kebab-case 이름, `version`, `description`, `skills` 상대경로 계획은 타당합니다.
- 현재 공식 가이드는 공유·설치 검증에 Marketplace와 재시작 후 새 스레드 확인 절차를 설명합니다.
- 가장 작은 플러그인은 manifest와 Skill로 시작할 수 있으므로 MCP 제외 결정은 문제없습니다.
- 이름과 루트 디렉터리명이 반드시 같아야 한다는 규칙은 공식 가이드에서 확인되지 않았습니다.

## 8. 권장 수정 순서

1. 대표 사용 흐름과 기본 출력 세 개를 확정합니다.
2. 입력 파일·필드·시간대·누락 처리 계약을 작성합니다.
3. 통계·장애 구간·지원 입력 범위를 정의합니다.
4. 원본→마스킹→분석 JSON→Codex 보고서의 보안 경계를 고정합니다.
5. claim·evidence 스키마와 신뢰도 규칙을 정의합니다.
6. manifest 이름 규칙과 Marketplace 설치 절차를 바로잡습니다.
7. 기획서에서 구현 완료처럼 보이는 문구를 미래형으로 수정합니다.
8. 이 변경안을 기획서·AGENTS.md·제작 가이드·Notion Phase 3에 반영한 뒤 설계를 동결합니다.

## 9. 최종 판정

OpenBell Guard의 **문제 선정은 유지하는 것이 좋습니다.** 카카오페이증권 특화성, 최근성, 공개 검증 가능성과 데이터분석 중심 강점이 모두 살아 있습니다.

점검 당시 기획서는 `무엇을 해결할지`는 잘 정했지만 `정확히 어떤 입력을 어떤 계산으로 어떤 파일에 출력할지`가 넓고 일부 모호했습니다. P0 여섯 항목을 반영해 백엔드 플랫폼을 과도하게 구현하지 않으면서도 심사자가 직접 실행·검증할 수 있는 설계로 동결했습니다.

최초 점검 당시에는 기획서를 수정하지 않았으나, 2026-06-30 승인 후 권고를 모두 반영했습니다. Phase 4에서는 이 계약을 임의로 넓히지 않고 Golden fixture와 `analysis.json`부터 구현합니다.
