## 1. 왜 이 자료가 중요합니까?

이 게시물은 카카오페이의 일반적인 기술 관심사를 소개한 글이 아니라 카카오페이증권 DevOps 팀이 실제 운영하는 차세대 로깅 플랫폼 구축기입니다. 따라서 OpenBell Guard의 기업 적합성을 뒷받침하는 직접적인 공식 자료로 사용할 수 있습니다.

특히 다음 내용이 현재 프로젝트 문제와 맞닿아 있습니다.

- 카카오페이증권은 국내장 09:00과 해외장 23:30, 서머타임 기간 22:30의 트래픽 스파이크를 시스템 설계에 반영합니다.
- 서비스 성장에 따라 하루 로그량이 약 100GB에서 41TB·200억 건 이상으로 증가했다고 설명합니다.
- 기존 플랫폼에서는 로그 유입이 수 분에서 수 시간 지연되고, 조회에 5분 이상 걸리는 문제가 있었다고 설명합니다.
- 새 플랫폼은 수집·처리·저장·아카이빙·조회 전 과정을 재설계해 로그 유입 지연을 20초 이내로 줄이고 비용을 기존 대비 14.4%로 낮췄다고 보고합니다.
- 장애 분석에서 시간 범위와 서비스 필드 조건을 결합한 조회가 전체 사용 패턴의 90% 이상이라고 설명합니다.

위 수치는 카카오페이증권이 공개한 운영 성과이며, OpenBell Guard가 독립적으로 재검증한 값은 아닙니다.

## 2. 게시물의 핵심 기술과 개념

### 2.1 OpenTelemetry와 배치 처리

기존 Filebeat·Fluentd 중심 흐름을 OpenTelemetry Collector로 전환했습니다. OTLP Protocol Buffers와 배치 전송을 사용해 메시지 오버헤드와 Kafka 처리 부담을 줄였다고 설명합니다.

프로젝트에 중요한 부분은 특정 수집 제품이 아니라 다음 원칙입니다.

- 서로 다른 로그 형식을 공통 속성으로 정규화합니다.
- 처리량·오류율·지연을 서비스와 로그 유형 단위로 비교합니다.
- 로그 발생 시점과 플랫폼 유입 시점의 차이를 별도 품질 지표로 관리합니다.

### 2.2 로그 유형과 우선순위 분리

게시물은 로그 유형과 처리 우선순위를 구분하고, 피크 시간대에 중요도가 낮은 처리 자원을 줄여 빠른 로그 처리 자원에 재배분하는 방식을 설명합니다. 또한 애플리케이션·DB·Kafka·Redis·Nginx·FEP 등 다양한 로그를 `logtype` 속성으로 라우팅합니다.

OpenBell Guard에서는 이를 인프라 스케일링 기능으로 구현하지 않습니다. 대신 다음 분석 분류에 응용합니다.

- 주문·체결처럼 고객 거래에 직접 영향을 주는 경로
- 시세·관심종목처럼 투자 판단을 지원하는 경로
- 외부 브로커·거래소처럼 회사 밖 의존성에 연결된 경로
- 관측 시스템 자체의 지연 또는 누락

### 2.3 ClickHouse의 시간·필드 중심 조회

게시물은 전문 검색보다 `시간 범위 + 서비스 필드` 조회가 대부분인 실제 사용 패턴에 맞춰 ClickHouse를 선택했다고 설명합니다. 자주 검색하는 필드를 별도 컬럼으로 만들고, 시간과 서비스 중심으로 데이터를 정렬합니다.

이 원칙은 OpenBell Guard의 근거 추적 방식과 잘 맞습니다. 플러그인은 분석 문장을 다음 정보에 연결해야 합니다.

- 입력 파일과 행 번호
- 이벤트 발생 시각
- 관측 시스템 유입 시각
- 서비스 또는 고객 경로
- 오류·지연·외부 의존성 필드

### 2.4 최근 데이터와 장기 데이터의 역할 분리

게시물은 최근 로그는 IDC ClickHouse에서 빠르게 조회하고 장기 로그는 S3로 아카이빙한 뒤 필요할 때만 AWS ClickHouse를 기동하는 구조를 설명합니다.

MVP에서는 S3나 ClickHouse를 도입하지 않습니다. 다만 다음 원칙은 유지합니다.

- 사고 구간의 상세 이벤트와 비교 기준선 데이터를 분리합니다.
- 입력이 커지면 전체를 LLM 문맥에 넣지 않고 결정론적 코드가 먼저 집계합니다.
- Codex에는 근거가 있는 요약과 필요한 원본 위치만 전달합니다.

### 2.5 로그와 메트릭의 역할 분리

게시물은 로그 검색에는 HyperDX, 시스템 메트릭에는 Grafana를 병행한다고 설명합니다. OpenBell Guard도 로그와 메트릭을 하나로 뭉개지 않고 다음처럼 사용합니다.

- 로그: 어떤 서비스·요청·의존성에서 무슨 일이 발생했는지 확인합니다.
- 메트릭: 처리량, 오류율, 지연시간, 자원 포화의 변화량을 계산합니다.
- Codex: 두 자료의 시간 관계와 서로 지지하거나 반박하는 증거를 종합합니다.

## 4. 채택할 요소

### 4.1 OpenTelemetry에서 영감을 받은 정규화 이벤트 모델

제품이나 프로토콜을 직접 도입하지 않고 다음 공통 필드를 갖는 로컬 이벤트 스키마를 정의합니다.

| 필드 | 용도 |
|---|---|
| `event_time` | 서비스에서 사건이 발생한 시각 |
| `observed_time` | 로그가 관측 파이프라인에 들어온 시각 |
| `ingestion_lag_ms` | 두 시각의 차이로 계산한 관측 지연 |
| `service_name` | 합성 또는 사용자 제공 서비스 식별자 |
| `service_path` | 인증·시세·정보·주문·정기투자·외부연계 경로 |
| `log_type` | application·nginx·database·kafka·dependency 등 입력 유형 |
| `severity` | DEBUG·INFO·WARN·ERROR 등 심각도 |
| `status` | success·error·timeout 등 결과 |
| `latency_ms` | 요청 또는 처리 지연시간 |
| `trace_id` | 사용자가 제공한 경우 서비스 간 사건 연결에 사용 |
| `dependency_type` | internal·exchange·broker 등 의존성 분류 |
| `source_file`, `source_line` | 보고서 문장의 원본 근거 위치 |

이는 OpenTelemetry 호환을 보장하는 스키마가 아니라, 게시물의 정규화 원칙을 합성 데이터 기반 MVP에 맞게 단순화한 모델입니다.

### 4.2 관측 지연을 장애 원인과 분리

로그가 늦게 들어온 현상을 실제 서비스 장애로 잘못 해석하지 않도록 `ingestion_lag_ms`를 별도로 계산합니다.

- 서비스 지연 상승 + 관측 지연 정상: 서비스 경로 이상 가능성을 검토합니다.
- 서비스 지연 정상 + 관측 지연 상승: 로깅 파이프라인 문제 가능성을 검토합니다.
- 둘 다 상승: 원인을 단정하지 않고 서비스와 관측 계층의 동시 증거를 요구합니다.

### 4.3 시장 시간대 중심 비교

합성 데이터와 분석기의 기본 시나리오에 다음 피크 구간을 반영합니다.

- 국내장 개장 전후 08:55~09:10
- 해외장 개장 전후 23:25~23:40
- 서머타임 적용 시 해외장 개장 전후 22:25~22:40

시간 구간은 공개 글에 나타난 트래픽 특성을 반영한 테스트 기본값일 뿐, 카카오페이증권의 실제 SLO나 경보 임계치로 표현하지 않습니다.

### 4.4 시간·서비스·경로 중심 근거 검색

분석 순서를 다음처럼 고정합니다.

1. 사고 시간 범위를 좁힙니다.
2. 서비스와 고객 경로별 처리량·오류율·지연을 비교합니다.
3. 내부 경로와 외부 의존성을 분리합니다.
4. 관측 지연 또는 로그 누락 여부를 확인합니다.
5. 사실, 원인 가설, 추가 확인 필요 사항을 분리해 보고합니다.

## 5. 채택하지 않을 요소

다음 구성은 좋은 기술이지만 OpenBell Guard MVP에는 넣지 않습니다.

- ClickHouse 클러스터 구축 및 운영
- Kafka Topic·파티션·Consumer Group 구성
- OpenTelemetry Collector 배포와 OTLP 전송
- HyperDX·Grafana UI 구축
- Kubernetes 자동 스케일링
- S3 아카이빙과 장기 조회용 AWS ClickHouse
- 게시물의 파티션 수, 코어 수, 버퍼 크기를 그대로 사용하는 설정

제외 이유는 다음과 같습니다.

- 현재 과제는 로그 플랫폼 자체보다 Codex 플러그인의 장애 분석 능력을 검증해야 합니다.
- 대규모 인프라는 설치·실행 부담이 커져 심사자가 짧은 시간에 재현하기 어렵습니다.
- 실제 카카오페이증권 데이터와 인프라 접근 권한이 없으므로 운영 통합을 검증할 수 없습니다.
- 인프라 구현에 시간을 쓰면 민감정보 마스킹, 근거 연결과 합성 시나리오 품질이 낮아질 수 있습니다.

## 6. OpenBell Guard에서의 위치

OpenBell Guard는 공개된 로그 플랫폼을 대체하거나 Pallas V2에 직접 연결되는 제품으로 주장하지 않습니다. 위치는 다음과 같습니다.

```text
합성 또는 사용자가 내보낸 로그·메트릭
                  │
                  ▼
       공통 이벤트 스키마 정규화
                  │
                  ├─ 서비스·고객 경로 분류
                  ├─ 처리량·오류율·지연 계산
                  └─ 관측 지연·누락 검사
                  │
                  ▼
       Codex의 근거 기반 사고 분석
                  │
                  ▼
 타임라인·가설·개선안·부하 테스트·사후 분석
```

즉, 이미 잘 수집된 관측 데이터를 사람이 검토할 수 있는 사고 판단과 재발 방지 작업으로 바꾸는 분석 계층입니다.

## 7. 검증에 추가할 시나리오

### 시나리오 E: 서비스 정상·로그 유입 지연

- 09:00 전후 실제 서비스 지연과 오류율은 정상입니다.
- `observed_time`만 늦어져 `ingestion_lag_ms`가 급증합니다.
- 기대 결과: 고객 서비스 장애로 단정하지 않고 관측 파이프라인 지연을 상위 가설로 제시합니다.

### 시나리오 F: 피크 시간대 경로별 부분 장애

- 09:00에 전체 처리량이 증가합니다.
- 시세·관심종목 경로만 느려지고 주문 경로는 정상입니다.
- 로그 유입 지연은 정상 범위입니다.
- 기대 결과: 로그 플랫폼 지연과 전체 주문장애를 배제하고 영향 경로를 한정합니다.

## 8. 최종 판단

이 게시물은 현재까지 확보한 자료 중 OpenBell Guard의 기술 설계를 가장 직접적으로 뒷받침합니다. 따라서 다음처럼 활용하는 것이 적절합니다.

- **적극 채택:** 정규화, 관측 지연, 시장 시간대, 서비스·로그 유형 분리, 시간·필드 중심 분석 원칙
- **참고만:** ClickHouse 테이블 설계, 로그 우선순위별 자원 재분배, 장기 아카이빙 전략
- **제외:** ClickStack 전체 설치, 운영 인프라 통합, 실제 환경 자동 변경

이 선택은 기업 기술을 억지로 붙이는 방식이 아니라, 카카오페이증권이 공개한 운영 맥락을 작은 로컬 플러그인에서 검증 가능한 분석 규칙으로 번역하는 방식입니다.
