## 1. 결론부터 설명합니다

이 글의 Spring Batch·MongoDB 구현을 OpenBell Guard에 그대로 넣는 것은 적합하지 않습니다. 현재 플러그인은 로컬 Python 분석기이고 합성 데이터로 재현해야 하므로 Java·Spring·데이터베이스·스레드 풀을 추가하면 구현비용과 검증 부담만 크게 늘어납니다.

다만 다음 한 가지 원칙은 직접적인 가치가 있습니다.

> 큰 입력을 한꺼번에 메모리에 올리지 않고, 가능한 형식은 한 줄씩 읽으며 처리량과 안전 한도를 기록합니다.

필수성과 구현비용을 다시 검토한 결과 다음 최소 범위만 확정합니다.

- JSONL·CSV·텍스트 파일은 Python 반복자로 한 줄씩 읽습니다.
- 입력 파일 크기, 처리·거부 레코드 수와 오류 위치를 입력 요약에 기록합니다.
- JSON 배열은 Python 표준 라이브러리만으로 완전한 스트리밍이 어렵기 때문에 검증된 `max_json_bytes` 이하에서만 전체 파싱하고, 큰 파일에는 JSONL 사용을 안내합니다.
- 일반 JSONL·CSV·텍스트의 사용자 조정형 파일·레코드 한도, 자동 변환과 파티셔닝은 MVP에서 제외합니다.

이 축소 범위는 `docs/openbell-guard-research-plan.md`에 반영했습니다.

## 2. 출처 범위를 구분해야 합니다

게시물은 카카오페이 **정산플랫폼팀**이 원장 통계 데이터를 재생성한 사례입니다. 카카오페이증권의 로그 분석 시스템이나 장애 대응 사례가 아닙니다.

- 카카오페이 계열의 대량 금융 데이터 처리 사례로 참고할 수 있습니다.
- 대량 데이터를 분할·스트리밍·일괄 처리한 성능 개선 사례로 사용할 수 있습니다.
- 카카오페이증권이 Spring Batch, MongoDB 또는 같은 Partitioning 구성을 사용한다는 근거로는 사용할 수 없습니다.
- OpenBell Guard가 수억 건의 데이터를 처리할 수 있다고 주장하는 근거로 사용할 수 없습니다.

게시물에 제시된 처리시간과 성능 향상 수치는 해당 팀이 공개한 실험 결과이며 OpenBell Guard가 독립적으로 재검증한 값이 아닙니다.

## 3. 게시물은 무엇을 해결했습니까?

초급자 관점에서 단순화하면 다음과 같습니다.

1. 과거 원장 통계 수억 건을 한꺼번에 다시 계산해야 했습니다.
2. 기간을 월 단위로 나누고, 한 달 안에서는 다시 일 단위 작업으로 분할했습니다.
3. 작업을 여러 스레드에서 병렬 실행했지만 각 스레드가 수백만 건을 메모리에 올리면서 OOM이 발생했습니다.
4. 각 조각 안에서는 MongoDB Cursor로 필요한 데이터만 순서대로 읽었습니다.
5. 결과를 한 건씩 저장하지 않고 일정량을 묶어 Bulk Write했습니다.
6. 게시물은 500만 건 실험에서 기본 방식보다 최적화 방식이 약 10.9배 빨랐다고 보고했습니다.

핵심은 `병렬 처리하면 무조건 빨라진다`가 아닙니다. 데이터를 나누더라도 각 작업이 너무 많은 데이터를 메모리에 올리거나 출력이 느리면 다시 병목과 OOM이 발생하므로 입력부터 출력까지 흐름 전체를 맞춰야 한다는 점입니다.

Spring Batch 공식 문서도 먼저 단일 스레드·단일 프로세스 방식이 실제 요구를 충족하는지 측정한 뒤 복잡한 병렬화를 고려하라고 안내합니다. 또한 동시 처리에서는 thread safety, 트랜잭션과 데이터베이스 커넥션 풀 같은 추가 제약을 함께 검토해야 합니다.

## 4. OpenBell Guard와 맞닿는 지점

| 게시물의 원칙 | OpenBell Guard 적용 가능성 | 판단 |
|---|---|---|
| 입력을 한꺼번에 메모리에 적재하지 않음 | JSONL·CSV·텍스트를 반복자로 처리 | 높음 |
| 처리 단위를 나누고 결과를 취합 | 파일·서비스·시간 구간별 결정론적 집계 | 중간 |
| 입력·처리·출력 속도를 함께 고려 | 파싱·통계·보고서 단계별 건수와 시간 기록 | 중간 |
| 현실적인 데이터로 전후 성능 측정 | 작은 fixture와 별도 성능 fixture 구분 | 중간 |
| 여러 스레드의 병렬 처리 | 작은 로컬 합성 데이터에는 불필요 | 낮음 |
| Cursor 기반 MongoDB 조회 | 로컬 파일 입력과 맞지 않음 | 매우 낮음 |
| Bulk Write | 읽기 중심 분석 플러그인에 데이터베이스 쓰기가 없음 | 매우 낮음 |

현재 Pallas 검토에는 큰 입력을 LLM 문맥에 직접 넣지 않고 Python이 먼저 집계한다는 원칙이 있습니다. 이번 글은 그 원칙을 실제 파일 읽기 방식과 입력 안전장치까지 구체화할 수 있는 자료입니다.

## 5. 최종 적용 범위

### 5.1 형식별 입력 전략

| 형식 | 제안 처리 방식 | 이유 |
|---|---|---|
| JSONL | 한 줄씩 JSON 객체 파싱 | 레코드 경계가 줄 단위여서 스트리밍 가능 |
| CSV | `csv.DictReader` 반복 처리 | Python 표준 라이브러리로 행 단위 처리 가능 |
| 텍스트 | 한 줄씩 읽고 위치 기록 | 오류 줄과 근거 위치를 유지하기 쉬움 |
| JSON 배열 | 크기 사전 검사 후 `json.load` | 표준 라이브러리만으로 배열 요소 완전 스트리밍이 어려움 |

큰 JSON 배열이 제한을 넘으면 전체를 읽다가 실패하도록 두지 않고 JSONL 변환을 안내합니다. 새 외부 의존성을 도입하지 않아 심사 재현성을 유지합니다.

### 5.2 입력 검증 보고서 보강

검증 결과 JSON 또는 별도 입력 요약에 다음을 기록합니다.

- 파일별 바이트 크기
- 처리한 레코드 수
- 거부하거나 파싱하지 못한 레코드 수와 위치
- JSON 배열에 적용한 `max_json_bytes`
- 중단 여부와 이유

`max_json_bytes` 기본값은 지금 임의로 정하지 않고 구현 시 합성 성능 fixture와 개발 환경 측정을 통해 결정합니다. 일반 행 단위 입력의 파일·레코드 한도 설정은 필요성이 확인될 때 추가합니다.

### 5.3 결정론적 안전 테스트

- `read()`·`readlines()`를 허용하지 않는 가짜 파일 객체로 JSONL·CSV 파서가 반복 읽기를 사용하는지 검사합니다.
- 정상 행과 손상 행이 섞인 fixture에서 처리·거부 건수와 오류 위치가 일치하는지 확인합니다.
- 행 단위 방식과 작은 기대 fixture의 통계 결과가 같은지 비교합니다.
- JSON 배열이 크기 한도를 넘으면 JSONL 변환 안내를 출력하는지 확인합니다.

운영 환경의 실제 메모리 절감이나 수억 건 처리 성능을 검증했다고 표현하지 않습니다.

## 6. 후순위로 둘 내용

다음은 핵심 MVP와 필수 테스트가 완료된 뒤에만 재검토합니다.

- 일반 입력의 사용자 조정형 파일·레코드 한도
- 서비스 경로·고정 시간 구간별 논리 파티션을 만들고 순차적으로 결과 취합
- 파티션별 처리 건수·소요시간·실패 위치 manifest
- 정확한 p95·p99를 대규모 입력에서 계산하기 위한 외부 정렬 또는 근사 알고리즘
- 별도 성능 benchmark와 메모리 peak 측정

특히 정확한 백분위수를 계산하려면 단순 스트리밍만으로 충분하지 않을 수 있습니다. 모든 지연값을 보관하지 않으려면 외부 정렬이나 검증된 근사 알고리즘이 필요하므로 입력 파싱과 별개의 설계 문제로 남깁니다.

## 7. 구현하지 않을 내용

- Java·Kotlin·Spring Batch 애플리케이션
- MongoDB와 `MongoCursorItemReader`
- `Partitioner`, `PartitionHandler`, Worker Step
- 멀티스레드·멀티프로세스 병렬 분석
- 데이터베이스 Bulk Insert·Update
- thread pool·grid size 자동 튜닝
- 원장 통계 재생성 기능

이 기능들은 실제 데이터베이스와 쓰기 대상, 트랜잭션·재시작 모델이 있어야 의미가 있습니다. 로컬 합성 로그 분석을 위해 추가하면 문제 해결보다 인프라 재현이 중심이 됩니다.

## 8. 구현 비용 추정

다음 시간은 현재 `src/`가 없는 상태에서 AI 도움을 받아 구현·테스트·문서화한다는 전제의 대략적인 범위이며 실제 측정값이나 완료 약속이 아닙니다.

| 후보 | 예상 비용 | 판단 |
|---|---:|---|
| 파일 크기·처리·거부 건수 입력 요약 | 약 1~2시간 | 채택 |
| JSONL·CSV·텍스트 반복자 파싱과 오류 위치 테스트 | 약 0.5~1일 | 채택 |
| JSON 배열 크기 제한과 JSONL 안내 | 약 1~2시간 | 채택 |
| 일반 입력의 사용자 조정형 파일·레코드 한도 | 약 2~4시간 | 후순위 |
| 서비스·시간별 순차 파티션과 결과 취합 | 약 1~2일 | 후순위 |
| 대규모 정확 백분위수·메모리 benchmark | 약 1~3일 이상 | 후순위 |
| Python 멀티프로세스 병렬화 | 약 2~4일 이상 | MVP 제외 |
| Spring Batch·MongoDB 재현 환경 | 최소 1~2주 이상 | 제외 |

## 9. 기획서 변경 후보

다음 위치에 최소 범위를 반영했습니다.

1. `8.5 채택 경계`: 축소 채택·후순위·제외 범위를 구분했습니다.
2. `9.1 입력`: 행 단위 처리, 입력 범위 보고와 JSON 배열 사전 크기 검사를 추가했습니다.
3. `9.2 처리 흐름`: 입력 preflight와 형식별 파싱 단계를 추가했습니다.
4. `10.1 결정론적 코드`: 파일별 크기·건수·오류 위치와 큰 JSON 보호를 추가했습니다.
5. `13 구현 단계`: 최소 입력 안전계약의 구현 항목을 추가했습니다.
6. `14.1·14.2 검증`: 입력 범위와 큰 JSON 안전 경계 시나리오·테스트를 추가했습니다.

Spring Batch, MongoDB, Partitioning, Bulk Write와 병렬 실행은 기획서의 MVP 필수 기술로 추가하지 않습니다.

## 10. 최종 판단

- **채택:** JSONL·CSV·텍스트 행 단위 입력, 파일 크기·처리·거부 건수와 오류 위치, 큰 JSON 배열의 `max_json_bytes` 사전 검사와 JSONL 안내
- **후순위:** 일반 입력의 사용자 조정형 파일·레코드 한도, 논리 파티션, 파티션 manifest, 대규모 정확 백분위수와 성능 benchmark
- **제외:** Spring Batch·MongoDB·멀티스레드·Bulk Write 구현

네 항목을 그대로 확장 기능으로 채택하지 않고, 분석 범위의 신뢰성과 JSON 전체 파싱 실패를 막는 최소 입력 계약으로 축소해 기획서에 반영했습니다.
