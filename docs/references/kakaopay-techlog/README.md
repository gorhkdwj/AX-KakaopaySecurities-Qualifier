# 카카오페이 Techlog 원문 자료 보관 위치

후속 카카오페이 Techlog 게시물의 로컬 원문 또는 사용자가 제공한 복사본을 이 폴더에 보관합니다.

## 권장 제공 방식

1. **기본값은 공식 게시물 URL만 제공하는 방식입니다.** 원문 제목·발행일·본문·코드·이미지 맥락과 최신 정정 내용을 공식 페이지에서 함께 확인할 수 있습니다.
2. 링크와 함께 `OpenBell Guard 적용성 검토`, `초급자 설명`, `특정 기술만 분석`처럼 원하는 작업을 알려주면 됩니다.
3. 공식 페이지가 차단되거나 삭제될 가능성이 있거나, 특정 시점의 원문을 고정해 비교해야 할 때만 전체 복사본을 이 폴더에 추가합니다.
4. 로컬 복사본을 제공할 때는 중간 내용을 생략하거나 표현을 고치지 않은 원문 상태를 유지하고 공식 URL도 함께 남깁니다.

전체 페이지 텍스트를 복사하면 표·코드 블록·이미지 설명·링크가 누락되거나 형식이 바뀔 수 있으므로, 공개적으로 접근 가능한 공식 페이지에는 기본적으로 권장하지 않습니다.

## 검토부터 기획서 반영까지의 순서

```text
공식 링크 확인
  → 출처 범위·적용성·구현 비용·검증 방법 검토
  → 채택·참고만·제외 분류
  → 기획서 반영 후보와 정확한 변경 위치를 사용자에게 보고
  → 사용자 승인
  → 기획서·Decisionlog·출력·테스트 계획 갱신
  → 정합성 검증 후 다음 Techlog 검토
```

- 사용자가 승인하기 전에는 `docs/openbell-guard-research-plan.md`를 변경하지 않습니다.
- `참고만`·`후순위` 항목은 검토 문서에 유지하고 MVP 필수 범위로 자동 승격하지 않습니다.
- 사용자 승인이 끝난 항목은 구현 전에 기획서와 테스트 계획에 반영해 누락을 방지합니다.

## 파일 이름

- 게시물 URL의 마지막 경로인 slug를 그대로 사용합니다.
- 예: `https://tech.kakaopay.com/post/example-article/` → `example-article.md`
- 텍스트로 저장할 수 있으면 Markdown(`.md`) 또는 텍스트(`.txt`)를 우선합니다.
- 웹페이지·PDF 등 원본 형식을 유지해야 하면 해당 형식을 그대로 둡니다.

## 자료별 역할

- `docs/references/kakaopay-techlog/`: 사용자가 제공한 원문 또는 로컬 복사본
- `docs/reviews/`: OpenBell Guard 적용 가능성, 제외 범위와 판단 근거
- `docs/guides/`: 초급자를 위한 기술·개념 해설
- `docs/SOURCES.md`: 원본 URL, 제목, 발행일 또는 확인일과 로컬 산출물 색인

## 관리 원칙

- 원문 파일은 임의로 편집·발췌하지 않습니다.
- URL만 제공된 경우 원문 파일을 만들 필요가 없습니다. 공식 URL을 확인한 뒤 검토 문서와 `docs/SOURCES.md`만 갱신합니다.
- 장문 원문을 프로젝트 문서에 중복 복사하지 않고, 검토·가이드 문서에는 필요한 내용을 요약해 작성합니다.
- API 키, 토큰, 실제 고객·계좌정보가 포함된 파일은 넣지 않습니다.
- 이 폴더는 개발 참고자료이며 최종 `submission.zip`에는 기본적으로 포함하지 않습니다.
