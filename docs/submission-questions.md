# OpenBell Guard 예선 질문 5문항 답변 초안

## 1. 무엇을, 누가, 어떤 상황에서 쓰나요?

OpenBell Guard는 카카오페이증권 같은 증권 서비스 운영·장애 분석 담당자가 사고 종료 후 내보낸 합성 또는 익명화된 사고 분석 번들을 검토할 때 사용하는 Codex 플러그인입니다.

사용자는 `incident.json`, `logs.jsonl`, `metrics.csv`, `service-map.json`으로 구성된 번들 경로를 제공합니다. 플러그인은 이를 로컬에서 검증·마스킹·분석한 뒤 서비스 경로별 영향, 장애 시작과 회복 시각, 확인된 사실, 원인 가설, 추가 확인 필요 사항을 산출합니다.

대표 상황은 국내장 개장 직후 시세·관심종목 조회가 느려졌지만 주문 경로는 정상인 부분 장애, 또는 외부 중개사·브로커 의존성 문제와 내부 서비스 정상 상태를 분리해야 하는 상황입니다.

## 2. 왜 이 문제를 선택했나요?

카카오페이증권은 공개 보도에서 2026년 초 국내장 개장 직후 지연과 외부 중개사 장애가 언급됐고, 회사 공식 Techlog와 채용 공고에서는 대규모 로그, 트래픽 스파이크, 로그·패킷·메트릭 분석, 장애 원인 분석과 재발 방지 업무가 확인됩니다.

증권 서비스 장애 분석에서 중요한 점은 모든 지연을 “전체 주문 장애”로 과장하지 않는 것입니다. 시세, 관심종목, 주문·체결, 외부 의존성 경로를 분리해야 고객 영향, 내부 조사 우선순위, 사후 보고서 문구가 정확해집니다.

OpenBell Guard는 이 문제를 데이터 분석 관점으로 해결합니다. 실제 운영 시스템을 변경하거나 주문에 개입하지 않고, 사고 번들의 지표와 근거를 계산해 사람이 검토할 수 있는 보고서 초안을 만듭니다.

## 3. 플러그인은 어떻게 작동하나요?

플러그인은 `src/skills/openbell-guard/scripts/run_openbell.py`를 중심으로 동작합니다.

1. 입력 번들의 파일 구조, 크기, UTF-8, 사고 구간과 시간대를 검사합니다.
2. 토큰, 이메일, 전화번호, 계좌번호 형태의 민감정보 패턴을 마스킹한 작업본을 만듭니다.
3. 로그와 메트릭을 행 단위로 읽고 60초 bucket에 배정합니다.
4. 요청 수, 오류 수, 오류율, p50·p95·p99 지연, 관측 지연, 기준 구간 대비 변화, CPU·메모리 맥락 지표를 계산합니다.
5. 임계치와 연속 bucket 규칙으로 `healthy`, `breach`, `unknown`, `outage_detected`를 판정합니다.
6. 모든 확인된 사실과 원인 가설에 claim ID와 evidence ID를 연결합니다.
7. `analysis.json`, `openbell-report.md`, `sanitization-report.md`, `output-validation.json`을 생성합니다.

출력 검증기 `validate_bundle.py`는 evidence 참조, claim marker, 민감정보 잔존 여부를 다시 확인합니다.

## 4. AI를 어떻게 활용했나요?

AI는 공개 자료 조사, 문제 정의, 기획서 작성, 지표 계약 설계, 코드 구현, 테스트 시나리오 설계, 문서화, 오류 원인 분석에 사용했습니다.

다만 통계 계산, 입력 검증, 민감정보 마스킹, 상태 판정은 재현 가능한 Python 코드로 구현했습니다. AI가 임의로 숫자를 계산하거나 근본 원인을 단정하지 않도록 `analysis.json`의 claim/evidence 구조와 출력 검증기를 두었습니다.

대화 로그는 `logs/` 폴더에 원본 그대로 보존하며, Worklog·Decisionlog·Troubleshootinglog에는 작업과 판단 근거만 요약했습니다.

## 5. 어떻게 검증했나요?

검증은 네 층으로 나누었습니다.

첫째, 단위·통합 테스트입니다. `python -m pytest src\tests` 기준 68개 테스트가 통과했습니다. 이 테스트는 입력 사전 검사, 마스킹, 행 파싱, 지표 계산, 상태 판정, evidence·claim, 보고서 검증, A~H 통합 시나리오를 포함합니다.

둘째, 수동 테스트입니다. case-001은 작은 입문 예제이고, case-002는 80,000행 규모의 현실형 합성 데이터입니다. case-002에서 `market_data`와 `watchlist_info`는 장애로, `order_execution`은 정상으로 판정됐고 ground truth와 일치했습니다.

셋째, 성능·회귀 benchmark입니다. P4-18에서 `logs.jsonl` 100,000행, `metrics.csv` 50,000행 합성 지원 한도 입력으로 5회 측정했고, 실행시간 중앙값 34.523738초와 Python 추적 메모리 최고값 194.280592MiB로 기준을 통과했습니다.

넷째, 제출 패키징 검증입니다. P4-19에서 `submission.zip`을 생성하고 내부 경로, 필수 파일, 로그 형식, 금지 폴더 제외 여부, 플러그인 manifest와 Skill 구조를 다시 검사했습니다. 제출 ZIP 내부 경로에서 대표 fixture를 실행했고, `output-validation.json`과 독립 검증기 기준으로 schema, evidence 참조, report claim marker, 민감정보 잔존 검사가 모두 통과했습니다.
