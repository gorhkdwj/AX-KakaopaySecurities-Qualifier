# OpenBell Guard 서브에이전트 운영·구현 분담 가이드

> 문서 상태: Phase 4 구현 전 운영 기준선  
> 작성일: 2026-06-30  
> 적용 범위: OpenBell Guard의 구현, 검토, 검증, 문서화와 제출 준비  
> 제품 범위 기준: [OpenBell Guard 조사·기획서](./openbell-guard-research-plan.md)  
> 계산·판정 기준: [지표·수식·판정 기준 계약](./openbell-guard-metrics-validation-contract.md)

## 1. 결론

OpenBell Guard에는 **메인 에이전트 1명과 필요할 때만 사용하는 서브에이전트 최대 3명** 구조가 가장 적합합니다.

서브에이전트는 프로젝트 전체를 각자 완성하는 복제 작업자가 아닙니다. 메인 에이전트가 계약, 파일 소유권과 품질 게이트를 통제하고, 서브에이전트는 서로 겹치지 않는 작은 작업 묶음만 담당합니다.

가장 중요한 원칙은 다음 세 가지입니다.

1. 계약과 내부 인터페이스를 먼저 고정한 뒤 병렬 구현합니다.
2. 한 파일은 한 시점에 한 에이전트만 수정합니다.
3. 구현한 에이전트와 최종 검증하는 에이전트를 가능한 한 분리합니다.

서브에이전트는 백그라운드에서 계속 실행되는 별도 프로그램이 아닙니다. Codex가 실행 중인 요청 안에서 메인 에이전트가 구체적인 작업을 맡기고 결과를 회수하는 협업 방식입니다.

## 2. 왜 이 구조가 필요합니까?

OpenBell Guard는 단순한 Python 스크립트 하나가 아닙니다. 다음 요소가 동시에 맞아야 합니다.

- Codex 플러그인 구조와 `SKILL.md`
- 입력·출력 스키마
- 민감정보 탐지와 마스킹
- M-001~M-017 계산
- 29개 issue code와 종료 코드
- evidence·claim 연결
- 합성 fixture와 Golden 기대값
- 새 Codex 세션의 설치·호출 검증
- README·질문지·로그·코드의 정합성

한 에이전트가 모든 요소를 연속으로 만들면 앞에서 세운 가정을 뒤에서 스스로 정답처럼 받아들일 위험이 있습니다. 반대로 여러 에이전트가 같은 파일을 동시에 수정하면 공유 작업공간에서 변경이 충돌하거나 계약이 갈라질 수 있습니다.

따라서 **계약은 직렬로 통제하고, 구현과 독립 검토만 병렬화**합니다.

## 3. 운영 구조

```text
사용자
  ↓ 범위 승인·우선순위
메인 에이전트 ─ 계약·파일 소유권·통합·최종 보고
  ├─ 입력·보안 에이전트
  ├─ 분석 엔진 에이전트
  └─ QA·검증 에이전트
        ↓
메인 에이전트의 통합 테스트·기록·Notion 동기화
```

동시에 사용할 수 있는 역할은 작업 단계마다 바뀝니다. 항상 세 명을 채우지 않습니다.

## 4. 역할과 책임

### 4.1 메인 에이전트

메인 에이전트는 최종 통합 책임자입니다.

- 사용자와 직접 대화하고 승인 범위를 확인합니다.
- 기획서, 지표 계약과 제출 규칙을 해석합니다.
- 작업 패킷과 파일 소유권을 배정합니다.
- 공용 인터페이스와 통합 파일을 관리합니다.
- 서브에이전트 결과를 직접 읽고 수용·수정 요청·기각합니다.
- 전체 테스트와 대표 실행을 수행합니다.
- `Worklog.md`, `Decisionlog.md`와 Notion을 갱신합니다.
- 최종 README, 질문지와 제출 ZIP의 정합성을 책임집니다.

### 4.2 계약·스키마 에이전트

- 입력·출력 구조와 Golden 기대값을 계약에서 파생합니다.
- 구현 코드가 아니라 “어떤 입력에 어떤 결과가 나와야 하는가”를 먼저 정리합니다.
- 새 지표나 판정이 필요하면 직접 추가하지 않고 변경 제안으로 반환합니다.

### 4.3 입력·보안 에이전트

- 입력 경로, 파일, UTF-8, 크기, 레코드와 시간대 검사를 구현합니다.
- 민감정보 탐지·마스킹과 원본 우회 금지를 구현합니다.
- 원본 파일을 수정하지 않는지 검증합니다.

### 4.4 분석 엔진 에이전트

- 정규화된 레코드를 입력받아 M-001~M-017을 계산합니다.
- 버킷 상태, 장애 시작·회복, 실행 상태와 issue code를 판정합니다.
- 계약에 없는 지표를 만들지 않습니다.

### 4.5 QA·보안 검증 에이전트

- Golden 기대값과 구현 결과를 독립 비교합니다.
- 경계값, 종료 코드, 끊어진 evidence, 비밀정보 잔존과 원본 우회를 검사합니다.
- 기능을 만든 에이전트의 설명을 그대로 믿지 않고 실행 결과를 기준으로 판단합니다.

### 4.6 문서·제출 에이전트

- 구현이 안정된 뒤 `SKILL.md`, README와 질문 5문항의 정합성을 검토합니다.
- 구현되지 않은 기능이나 실행하지 않은 검증이 문서에 들어가지 않았는지 확인합니다.
- 최종 결정과 기록은 메인 에이전트에게 넘깁니다.

## 6. 권장 실행 파동

### 파동 1: 기반 고정

- 메인: WP-01, 공용 파일과 통합 구조
- 계약·스키마 에이전트: WP-02·WP-03 초안
- QA 에이전트: SHA, 스키마 오류, 백분위와 임계치 Golden 기대값 독립 검토

완료 게이트:

- 계약 복사본 SHA가 원본과 일치합니다.
- 입력·출력·내부 인터페이스가 문서와 최소 fixture로 고정됩니다.
- 계약 변경이 필요하지 않거나 변경 필요 사항이 사용자 승인 대기로 분리됩니다.

### 파동 2: 핵심 모듈 병렬 구현

- 입력·보안 에이전트: WP-04~WP-06
- 분석 엔진 에이전트: WP-07
- QA·검증 에이전트: WP-09와 WP-03에서 고정한 테스트
- 메인: WP-08의 통합 경계와 `run_openbell.py` 연결 준비

이 파동에서는 한 에이전트가 다른 에이전트의 파일을 고치지 않습니다. 공용 인터페이스가 잘못됐다고 판단되면 코드를 임의로 맞추지 않고 메인에게 변경 제안을 반환합니다.

### 파동 3: 통합과 결함 수정

- 메인: WP-08·WP-10 통합
- QA 에이전트: 시나리오 A~H의 검은상자 테스트
- 보안 검토 에이전트: seeded secret, 원본 우회, 파생 파일 잔존 검사
- 구현 에이전트: QA가 재현한 결함만 배정받아 수정

완료 게이트:

- 대표 요청으로 기본 산출물 세 개가 생성됩니다.
- Golden 결과와 `analysis.json`이 일치합니다.
- 비밀정보 원값 잔존과 근거 없는 사실 claim이 0건입니다.

### 파동 4: 설치·제출

- 메인: 새 Codex 세션 설치·발견·호출과 최종 패키징
- 문서·제출 에이전트: README·질문지·코드·검증 결과 대조
- QA 에이전트: ZIP 내부 경로와 불필요 파일 검사

최종 승인과 사용자 보고는 메인 에이전트만 수행합니다.

## 7. 파일 소유권

### 7.1 항상 메인 에이전트가 관리하는 파일

- `AGENTS.md`
- `docs/openbell-guard-research-plan.md`
- `docs/openbell-guard-metrics-validation-contract.md`
- `Worklog.md`
- `Decisionlog.md`
- `docs/notion-recording-guide.md`
- Notion 페이지
- `logs/`

`logs/`는 메인 에이전트도 수동 편집하지 않습니다.

### 7.2 구현 중 권장 소유권

| 소유자 | 대표 파일 |
|---|---|
| 메인 | `plugin.json`, `run_openbell.py`, 공용 인터페이스, 최종 `SKILL.md` |
| 입력·보안 | `redact_inputs.py`, 입력 검사·정규화 전용 보조 모듈과 관련 테스트 |
| 분석 엔진 | `analyze_telemetry.py`, 지표·상태 계산 전용 테스트 |
| QA·검증 | `validate_bundle.py`, Golden fixture, 통합·보안·패키징 테스트 |

실제 파일이 추가되면 작업 브리프에서 소유권을 다시 명시합니다. “관련 파일 전부”처럼 열린 범위를 주지 않습니다.

## 8. 서브에이전트 작업 브리프

```markdown
작업 ID:
목표:

완료 조건:
- 사람이 확인할 수 있는 구체적 결과

허용 파일:
- 수정할 수 있는 정확한 경로

읽기 전용 참고:
- 반드시 확인할 문서와 절

금지 사항:
- 허용되지 않은 파일 수정 금지
- logs/ 편집 금지
- 계약에 없는 지표·오류 코드 추가 금지
- 실제 고객·계좌·비밀정보 사용 금지

검증:
- 실행 명령
- 기대 결과

중단 조건:
- 계약 충돌
- 보안 위험
- 허용 파일 밖 수정 필요
- 필요한 입력이 없음

반환 형식:
- 변경 요약
- 변경 파일
- 실행한 검증과 실제 결과
- 미검증 범위
- 남은 위험과 통합 주의점
```

작업 브리프는 한 번에 한 가지 책임만 포함합니다. 프로젝트 전체 검토처럼 범위가 넓으면 특정 문서 절, 최대 출력 길이와 읽기 전용 여부를 지정해 더 작은 작업으로 나눕니다.

## 10. 품질 게이트

| 게이트 | 통과 조건 |
|---|---|
| G0 기준선 | 기획서와 계약이 사용자 승인 상태이고 미승인 변경이 분리됨 |
| G1 구조 | manifest·Skill·계약 복사본·내부 인터페이스가 존재하고 SHA 일치 |
| G2 모듈 | 각 모듈의 단위 테스트와 관련 M-ID·issue code 테스트 통과 |
| G3 통합 | 대표 실행으로 기본 산출물 세 개 생성, Golden 일치 |
| G4 안전 | 원본 무변경, 원본 우회 없음, seeded secret 잔존 0건 |
| G5 의미 | 부분 장애·외부 의존성·관측 지연·판단 불가 분류가 기대와 일치 |
| G6 플러그인 | 새 Codex 세션의 발견·자동 선택·명시 호출 성공 |
| G7 제출 | README·질문지·코드·테스트·로그 설명과 ZIP 구조 일치 |

앞 게이트가 실패한 상태에서 뒤 게이트의 성공을 전체 완료로 보고하지 않습니다.

## 11. 서브에이전트를 사용하지 않는 작업

다음은 분배 비용과 충돌 위험이 더 크므로 메인 에이전트가 직접 처리합니다.

- 오타나 짧은 문장 수정
- 한 파일 안에서 끝나는 작은 변경
- 사용자 승인 여부가 필요한 범위 결정
- 지표 계약과 보안 정책 변경
- `Worklog.md`, `Decisionlog.md`와 Notion 기록
- 비밀정보 발견·노출 대응
- 최종 통합, 최종 패키징과 사용자 보고

## 12. 실패와 시간 초과 처리

1. 결과가 없거나 시간 초과가 나면 같은 큰 요청을 반복하지 않습니다.
2. 대상 문서·함수·파일과 반환 길이를 줄여 재분배합니다.
3. 부분 결과는 메인이 파일과 테스트로 확인하기 전까지 수용하지 않습니다.
4. 계약 충돌은 구현 편의로 우회하지 않고 사용자 승인 대상으로 분리합니다.
5. 보안 위험이나 비밀정보가 발견되면 관련 구현을 중단합니다.
6. 에이전트가 만든 변경을 통합할 수 없으면 메인이 안전한 부분만 다시 구현하고 미사용 결과를 기록합니다.
7. 미검증 범위와 중단 원인은 최종 보고와 Worklog에 남깁니다.

## 13. 첫 구현 착수 순서

다음 요청부터는 아래 순서로 Phase 4를 시작합니다.

1. 메인 에이전트가 공식 plugin creator 기준을 재확인하고 WP-01 스캐폴드를 만듭니다.
2. 계약·스키마 에이전트가 WP-02·WP-03을 맡아 제출용 계약과 최소 Golden 기대값을 준비합니다.
3. QA 에이전트가 계약 SHA, 입력 오류와 M-004·M-006·M-007 경계값을 독립 검토합니다.
4. G1을 통과한 뒤에만 입력·보안, 분석 엔진과 출력 검증을 파일 단위로 병렬 구현합니다.

이 방식은 처음부터 많은 코드를 동시에 만드는 것보다 시작은 조금 느립니다. 대신 계약 드리프트와 재작성을 줄여 전체 구현 시간과 제출 위험을 낮춥니다.
