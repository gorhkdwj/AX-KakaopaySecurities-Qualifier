> 목적: 가장 작은 구현 단위부터 순차적으로 진행하면서 기획 정합성, 지표 정확성, 보안 경계와 코드 경량성을 계속 검증합니다.

## 1. 결론

Phase 4의 첫 성공 목표는 **작은 합성 사고 번들 하나가 계약 기준대로 `analysis.json`을 생성하는 것**입니다.

처음부터 전체 플러그인, 모든 fixture, 최종 README와 제출 ZIP까지 한 번에 만들지 않습니다. 순서는 다음처럼 잡습니다.

1. 플러그인 골격과 계약 복사본을 만듭니다.
2. 계약 SHA와 최소 입력·출력 구조를 먼저 검증합니다.
3. 가장 작은 국내장 개장 피크 fixture와 Golden 기대값을 고정합니다.
4. 입력 검사와 민감정보 마스킹을 구현합니다.
5. M-001~M-007 중심의 기본 지표 계산으로 `analysis.json`을 만듭니다.
6. 이후 관측 지연, 기준 대비 변화, evidence·claim, 보고서, 통합 시나리오, 설치·제출 검증으로 확장합니다.

이 계획은 “빨리 많이 만들기”보다 “작게 만들고 바로 검증하기”를 우선합니다. 초보 구현자가 AI와 함께 완성해야 하는 프로젝트이므로, 각 단계가 실패해도 되돌아갈 지점이 선명해야 합니다.

## 2. 구현 원칙

### 2.1 가장 작은 단위부터 구현합니다

각 단계는 한 번에 하나의 기능 축만 다룹니다.

- 구조를 만들 때 계산 코드를 넣지 않습니다.
- 입력 검증이 끝나기 전에 분석기를 확장하지 않습니다.
- 마스킹이 통과하기 전에 Codex 보고서 생성을 연결하지 않습니다.
- Golden 기대값 없이 새 지표를 구현하지 않습니다.

### 2.2 지표는 새로 정의하지 않습니다

이 문서는 지표 수식을 다시 정의하지 않습니다. 모든 계산은 단일 계약의 M-ID를 기준으로 합니다.

- 요청·오류·지연시간: M-001~M-007
- 관측 지연: M-008~M-009
- 기준 구간과 사고 구간 비교: M-010~M-013
- 레코드 처리·거부·제외 건수: M-014
- CPU·메모리 맥락 지표: M-015
- 성능 benchmark: M-016~M-017

구현 코드, 테스트 이름과 Golden 파일은 가능한 한 M-ID 또는 issue code를 포함해 어떤 계약을 검증하는지 추적 가능하게 만듭니다.

### 2.3 경량 설계를 유지합니다

MVP는 로컬 Python 분석 도구입니다.

- Python 표준 라이브러리를 우선합니다.
- 웹서버, 데이터베이스, 큐, MCP, 외부 API, 대시보드는 만들지 않습니다.
- 복잡한 플러그인 내부 프레임워크나 의존성 주입 구조를 만들지 않습니다.
- 작은 함수와 명확한 dict/list/dataclass 수준의 데이터 구조를 우선합니다.
- “나중에 확장 가능”보다 “지금 계약대로 재현 가능”을 우선합니다.

### 2.4 지표 정의 위치와 첫 검증 단계

아래 표는 구현 단계에서 어떤 지표를 처음 검증하는지 보여줍니다. 지표의 정확한 grain, 단위, 계산법, `null` 규칙과 반올림은 모두 [지표·수식·판정 기준 계약](./openbell-guard-metrics-validation-contract.md)의 해당 M-ID를 따릅니다.

## 5. 우선 구현할 “첫 번째 얇은 세로 조각”

초기 며칠의 목표는 전체 기능의 축소판입니다.

```text
P4-01 구조
→ P4-02 계약 SHA
→ P4-03 아주 작은 fixture
→ P4-04 CLI
→ P4-05 최소 입력 검사
→ P4-09 M-001~M-007 일부 계산
→ P4-14 analysis.json 생성
→ P4-15 analysis.json 구조 검증
```

이 조각이 성공하면 아직 보고서는 없어도 다음을 증명할 수 있습니다.

- 플러그인 루트 구조가 맞습니다.
- 계약 복사본이 원본과 일치합니다.
- 입력을 읽고 검증할 수 있습니다.
- 가장 중요한 요청·오류·지연시간 지표가 Golden과 일치합니다.
- `analysis.json`을 기계적으로 검증할 수 있습니다.

이후 마스킹, evidence·claim, Skill 보고서와 전체 시나리오를 덧붙입니다.

## 6. 단계별 서브에이전트 활용

| 시점 | 메인 에이전트 | 서브에이전트 | 목적 |
|---|---|---|---|
| P4-00~P4-03 | 구조와 기준선 통합 | 계약·fixture 검토 1명 | 기획·계약과 Golden이 어긋나지 않는지 확인 |
| P4-04~P4-08 | CLI와 입력 흐름 통합 | 입력·보안 검토 1명 | raw 원본 우회, 시간대 추정, 파일 한도 누락 방지 |
| P4-09~P4-12 | 지표 계산 통합 | 지표 QA 1명 | M-ID 수식·경계·`null` reason drift 방지 |
| P4-13~P4-16 | evidence·claim·보고서 흐름 통합 | 의미 검토 1명 | 근거 없는 사실·원인 단정 방지 |
| P4-17~P4-19 | 전체 검증·제출 통합 | QA·제출 검토 1~2명 | 설치·호출·ZIP·README 정합성 확인 |

처음부터 세 명을 동시에 풀가동하지 않습니다. 코드 골격이 생기기 전에는 검토할 실제 산출물이 적기 때문입니다.

## 7. 각 단계의 공통 중단 조건

다음이 발견되면 기능을 확장하지 않고 멈춥니다.

- 지표 계약과 다른 수식·경계·반올림·상태 판정이 필요해진 경우
- 계약에 없는 새 지표를 계산된 사실로 쓰고 싶은 경우
- 원본 사고 번들을 Codex가 직접 읽어야만 진행되는 경우
- 실제 고객·계좌·비밀정보가 필요한 경우
- 운영 시스템, 실제 주문, 실제 외부 API 호출이 필요한 경우
- Golden 기대값을 구현 결과에 맞춰 완화해야만 테스트가 통과하는 경우
- `logs/` 파일을 수동 편집해야 할 것처럼 보이는 경우

이때는 코드 수정 대신 사용자에게 변경 필요성과 선택지를 보고합니다.

위 항목은 “멋있어 보이는 기능”일 수는 있지만, 현재 제출물의 핵심인 로컬 합성 사고 번들의 재현 가능한 분석을 먼저 완성한 뒤 검토합니다.

## 9. 단계별 완료 보고 형식

각 구현 단계가 끝날 때는 다음 형식으로 보고합니다.

```markdown
단계:

- 전체 위치:
- 이번 단계 태스크:

완료한 기능:
변경 파일:

실행한 검증:
검증 결과:
관련 M-ID·issue code:

트러블슈팅:
- 발생한 T-ID:
- 증상·조치·결과 요약:

서브에이전트 검토 결과:
남은 위험:

남은 태스크:
- 다음 1~3개 우선 태스크:
- 후속 전체 태스크 요약:
- 남은 예상 작업량:

다음 단계:
```

전체 위치는 `P4-02/19`처럼 현재 단계가 전체 Phase 4 작업 중 어디에 있는지 표시합니다. 남은 예상 작업량은 단계 수와 체감 난이도를 함께 적습니다. 예를 들어 `약 17단계, 높음`, `약 3단계, 중간`처럼 씁니다.

트러블슈팅 항목은 실제 문제가 발생했을 때만 `Troubleshootinglog.md`의 T-ID와 연결합니다. 문제가 없으면 `없음`으로 씁니다.

이 형식은 Worklog, Troubleshootinglog, Notion 요약의 원천으로도 사용합니다.
