# 카카오페이증권 AX 해커톤 프로젝트 지침

이 파일은 이 프로젝트에서 작업하는 Codex가 항상 따라야 할 개발 지침입니다.
플러그인의 사용자용 실행 절차는 이 파일이 아니라 `src/skills/**/SKILL.md`에 작성합니다.

## 1. 대화 및 보고 방식

- 모든 응답은 격식 있는 한국어 대화체인 합쇼체·하십시오체로 작성합니다.
- 결론과 실제 결과를 먼저 설명하고, 필요한 근거와 세부사항을 뒤에 제시합니다.
- Phase 4 이후 구현 단계 완료 보고에는 현재 단계가 수행한 태스크, 전체 Phase 내 위치, 남은 태스크 목록, 남은 예상 작업량, 검증 결과와 트러블슈팅 요약을 포함합니다.
- 기술 용어는 처음 등장할 때 비전문가도 이해할 수 있도록 쉽게 설명합니다.
- 실행하지 않은 작업이나 테스트를 완료했다고 표현하지 않습니다.
- 확인된 사실, 합리적 추론, 확인할 수 없는 사항을 명확히 구분합니다.

## 2. 프로젝트 목표와 범위

- 본 프로젝트는 AX 해커톤의 카카오페이증권 제출물입니다.
- 한 제출물에는 카카오페이증권만 다루며 다른 기업의 문제를 함께 해결하지 않습니다.
- 현재 선정한 핵심 문제는 국내장 개장 시점의 서비스 지연과 외부 중개사 장애 분석 및 재발 방지입니다.
- 제안 플러그인의 이름과 ID는 각각 `OpenBell Guard`, `openbell-guard`입니다.
- 문제 배경과 설계 기준선은 `docs/openbell-guard-research-plan.md`를 따릅니다.
- 지표 이름·grain·단위·수식·반올림·`null`·임계치·실행 상태·오류 코드는 `docs/openbell-guard-metrics-validation-contract.md`를 유일한 기준으로 사용합니다.
- 핵심 문제나 해결 범위를 변경해야 한다면 코드보다 먼저 기획서와 사용자에게 변경 이유를 알립니다.

## 3. 작업 시작 전 확인 사항

- 작업 전에 관련 기존 파일과 디렉터리 구조를 확인합니다.
- Phase 4 이후 구현·검증 작업을 시작할 때는 `Troubleshootinglog.md` 전체를 통째로 읽지 않고, 빠른 색인에서 현재 작업과 관련된 T-ID만 확인합니다.
- 반복 가능한 환경·구조 점검은 가능한 한 `python tools/preflight_check.py`로 먼저 확인하고, 경고와 오류를 구분해 보고합니다.
- 사용자가 만든 기존 변경사항을 보존하고 관련 없는 파일을 수정하지 않습니다.
- 플러그인 규격은 프로젝트 특화 해설본인 `docs/guides/plugin-overview.md`와 `docs/guides/plugin-build-guide.md`를 먼저 확인합니다.
- 로컬 해설 문서는 공식문서 기반 요약본이므로 최신 동작과 정확한 스키마가 중요할 때는 `docs/SOURCES.md`의 OpenAI 공식 원문도 확인합니다.
- 프로젝트 규칙과 플러그인 실행 규칙을 혼동하지 않습니다. 개발 규칙은 `AGENTS.md`, 설치된 플러그인의 워크플로는 `SKILL.md`에 둡니다.

## 4. 공개 근거와 조사 원칙

- 기업의 문제는 공개적으로 검증할 수 있는 자료로 입증합니다.
- 기업 공식 자료, 금융감독원·금융위원회 등 공공기관 자료, 신뢰할 수 있는 언론 보도를 우선합니다.
- 기업 내부 정보에 접근한 것처럼 표현하거나 공개 자료로 확인할 수 없는 내부 사정을 단정하지 않습니다.
- 카카오페이증권이 공개한 설명과 외부의 해석을 구분합니다.
- 합성 테스트 데이터의 장애 원인을 실제 카카오페이증권 사건의 근본 원인처럼 표현하지 않습니다.
- 외부 사실을 문서에 추가할 때 원본 URL, 문서 제목, 발행일 또는 확인일을 기록합니다.
- `docs/`에 공식문서 복사본을 추가할 때 원문은 임의로 편집하지 않고 `docs/SOURCES.md`에 출처 정보를 기록합니다. 파일이 없으면 새로 생성합니다.
- 기업 Techlog는 먼저 개별 검토 문서에서 `채택`, `참고만`, `제외`와 구현 비용·검증 방법·출처 범위를 평가합니다.
- 기획서에 반영할 가치가 있다고 판단한 채택 후보는 수정 대상 절과 예상 영향·비용을 사용자에게 먼저 제시하고 명시적 승인을 요청합니다.
- 사용자 승인 전에는 `docs/openbell-guard-research-plan.md`의 기준선을 변경하지 않습니다. 승인 후에는 기획서·결정 기록·관련 출력·테스트 계획을 갱신한 뒤 다음 Techlog 검토로 넘어갑니다.
- `참고만`과 `후순위` 항목은 사용자가 별도로 승인하지 않는 한 기획서의 MVP 필수 범위에 넣지 않습니다.

## 5. 로그 무결성과 비밀정보

- `logs/`의 기존 로그 파일을 수동으로 편집, 발췌, 삭제, 병합, 이름 변경 또는 재포맷하지 않습니다.
- 공식 로그 훅과 `tools/watch_logs.py` 같은 승인된 자동 수집기가 로그를 생성·갱신하는 것은 허용합니다.
- 일반 포매터, 린터 또는 정리 스크립트의 대상으로 `logs/`를 포함하지 않습니다.
- 로그 수집 설정을 임의로 비활성화하지 않습니다.
- API 키, 비밀번호, 액세스 토큰, 세션 쿠키, 실제 계좌정보와 개인정보를 대화·코드·문서·테스트 데이터에 입력하지 않습니다.
- 비밀정보가 발견되면 대화나 로그를 사후 편집하지 말고 노출 범위와 안전한 교체·폐기 방법을 사용자에게 알립니다.

## 6. 제출물 및 플러그인 구조

- 최종 제출 구조는 `submission.zip/src`, `submission.zip/README.md`, `submission.zip/logs`를 기준으로 합니다.
- 실제 플러그인 루트 전체를 `src/` 안에 둡니다.
- `src/.codex-plugin/plugin.json`은 필수이며 삭제하거나 다른 위치로 옮기지 않습니다.
- `plugin.json`의 `name`은 안정적인 소문자 kebab-case `openbell-guard`를 사용합니다. 제출 루트인 `src/`의 디렉터리 이름과 같을 필요는 없습니다.
- `src/skills/openbell-guard/SKILL.md`와 실제 동작 요소인 실행 코드를 포함합니다.
- `plugin.json`에는 실제로 존재하는 구성요소만 선언합니다.
- MVP는 별도 API 키나 외부 서비스 없이 로컬 합성 데이터로 재현 가능하게 만듭니다.
- MCP는 문제 해결에 필수적이고 인증·재현 방법을 제공할 수 있을 때만 추가합니다.
- 개발 참고자료인 `docs/`, 로컬 캐시, 임시 결과와 비밀정보는 최종 제출물에 기본적으로 포함하지 않습니다.
- 개발용 `.agents/plugins/marketplace.json`을 사용하더라도 `./src`를 가리키는 로컬 설치 보조 파일로만 취급하고 최종 제출물에는 포함하지 않습니다.

## 7. 구현 원칙

- 통계 계산, 입력 검증과 민감정보 탐지는 재현 가능한 결정론적 코드로 구현합니다.
- MVP 대표 흐름은 장애 종료 후 내보낸 사고 번들 분석으로 고정하며, 실시간 모니터링·자동 대응 기능으로 확장하지 않습니다.
- 입력 파일, 필드, 표준 코드, 지원 한도, `complete`·`degraded`·`fatal`과 종료 코드는 지표·판정 계약을 따릅니다.
- 가능하면 Python 표준 라이브러리를 우선하고, 새 의존성이 필요하면 목적과 검증 방법을 설명합니다.
- 원본 입력 파일은 수정하지 않고 마스킹되거나 정규화된 파생 파일을 별도로 생성합니다.
- JSONL과 CSV 입력의 오류 위치를 사용자가 이해할 수 있게 보고하고 지원하지 않는 형식을 임의로 해석하지 않습니다.
- 타임스탬프의 시간대나 UTC 오프셋이 불명확하면 추정하지 말고 입력 오류로 중단합니다.
- 한국 표준시와 UTC를 함께 다룰 때 변환 기준을 코드와 문서에 명시합니다.
- 다른 문서·`SKILL.md`·코드 주석에 수식을 다시 정의하지 않고 계약의 M-ID·issue code를 참조합니다.
- 계약에 없는 지표를 계산된 사실로 출력하지 않습니다. 새 지표·수식·판정·오류 코드를 사용하려면 계약과 Decisionlog를 먼저 갱신하고 사용자 승인을 받습니다.
- 계산 코드와 테스트 이름에는 관련 M-ID 또는 issue code를 연결해 어떤 정의를 구현하는지 추적 가능하게 합니다.
- Phase 4에서 계약의 제출용 복사본을 만들고 원본과 SHA-256이 같은지 자동 검사합니다.
- 서비스 경로를 최소한 인증·접속, 시세, 관심종목·정보, 주문·체결, 정기투자, 외부 연계로 구분합니다.
- 카카오페이증권의 실제 내부 서비스명, 아키텍처 또는 SLO를 임의로 만들어내지 않습니다.
- 오류를 숨기거나 조용히 건너뛰지 말고 실패 원인과 영향을 명시합니다.

## 8. 안전 경계

- 실제 금융상품 주문, 주문 취소, 계좌 조작 또는 고객 데이터 변경 기능을 구현하거나 실행하지 않습니다.
- 실제 운영 인프라를 자동 변경·증설·배포하지 않습니다.
- 운영 주소를 대상으로 부하 테스트를 실행하지 않습니다.
- 테스트는 합성 데이터와 로컬 환경을 기본으로 하며, 스테이징 환경은 사용자가 명시적으로 지정한 경우에만 사용합니다.
- 플러그인의 기본 동작은 읽기 전용이어야 합니다.
- Codex는 원본 사고 번들을 직접 열거나 인용하지 않고, 로컬 스크립트가 만든 마스킹 작업본과 검증된 `analysis.json`만 읽습니다.
- 마스킹 실패 시 원본 직접 읽기로 우회하지 않고 분석을 중단합니다.
- 번들 스크립트는 외부 네트워크를 호출하지 않지만 Codex 서비스 전체가 네트워크와 무관하다고 표현하지 않습니다.
- 보상 여부, 법적 책임 또는 규제 준수 여부를 자동으로 확정하지 않습니다.
- 고객 공지와 사후 보고서는 초안으로 표시하고 사람의 검토가 필요함을 명시합니다.

## 9. AI 분석 및 출력 규칙

- 모든 사실 판단에는 입력 파일 위치, 로그 시각, 메트릭 구간 등 확인 가능한 근거를 연결합니다.
- `analysis.json`을 기계 검증 기준으로 삼고 모든 확인된 사실 claim에 하나 이상의 유효한 evidence 참조를 둡니다.
- `openbell-report.md`의 사실 문장에는 `[C-001]` 형식의 claim ID를 표시하고 검증기로 끊어진 참조를 검사합니다.
- 분석 결과를 `확인된 사실`, `원인 가설`, `추가 확인 필요`로 분리합니다.
- 상관관계만으로 인과관계를 확정하지 않습니다.
- 원인 가설에는 신뢰도, 지지 근거, 반대 근거와 추가로 필요한 데이터를 함께 제시합니다.
- 주문 경로가 정상이고 시세·관심종목 경로만 느린 경우 전체 주문장애로 표현하지 않습니다.
- 내부 용량 문제와 거래소·해외 브로커 등 외부 의존성 문제를 분리해 분석합니다.
- 데이터가 부족하면 가장 그럴듯한 답을 만들어내지 말고 `판단 불가`라고 기록합니다.
- 기본 산출물은 `analysis.json`, `openbell-report.md`, `sanitization-report.md` 세 개로 제한합니다.

## 10. 테스트와 검증

- 구현 변경 후 영향 범위에 맞는 단위 테스트와 통합 테스트를 실행합니다.
- 지표 계약의 각 M-ID, 경계값, `null` reason code, 상태와 issue code를 최소 한 번 이상 자동 테스트합니다.
- 국내장 개장 피크, 외부 중개사 장애, 불완전 데이터, 비밀정보, 통계 경계와 지원 한도를 합성 fixture로 검증합니다.
- 통계 결과는 고정된 기대값과 비교하고 최소 표본, 장애 구간, 서비스 경로 분류와 종료 코드를 검증합니다.
- 사실 문장의 근거 연결, 근거 없는 근본 원인 단정과 출력의 비밀정보 잔존 여부를 검사합니다.
- 실패한 테스트를 삭제·완화해 통과시키지 말고 원인을 수정합니다.
- 테스트를 실행할 수 없으면 그 이유와 미검증 범위를 최종 보고에 명시합니다.
- 플러그인 manifest와 스킬 구조를 공식 검증기로 검사합니다.
- Codex 재시작 후 새 스레드에서 플러그인 발견, 자동 선택, 명시적 호출과 대표 사용자 요청 한 번의 전체 흐름을 각각 확인합니다.
- 최종 ZIP 내부 경로와 필수 파일을 자동 또는 수동으로 재검사합니다.

## 11. 문서와 제출 정합성

- README, 질문 5문항, 실제 코드, 테스트 결과와 대화 로그가 같은 문제와 기능을 설명해야 합니다.
- 구현되지 않은 기능을 README나 질문지에서 제공한다고 표현하지 않습니다.
- 검증 문서에는 실행 명령, 입력 fixture, 실제 결과와 실패 사항을 기록합니다.
- 합성 데이터 검증을 카카오페이증권의 실제 운영 환경 검증이라고 표현하지 않습니다.
- 구현이 기획서와 달라지면 이유와 영향을 문서에 반영합니다.
- 제출 직전 `logs/`가 원본 그대로인지, 비밀정보가 새로 포함되지 않았는지, 제출 구조가 요구사항과 일치하는지 확인합니다.

## 12. 작업 및 결정 기록

- 사용자 요청 단위의 작업을 완료하기 전에 루트 `Worklog.md`에 새 항목을 추가합니다.
- `Worklog.md`에는 날짜, 고유 ID, 요청, 수행 작업, 변경 파일, 실제 검증 결과, 현재 상태와 남은 과제를 기록합니다.
- 구현 과정에서 오류, 검증 실패, 환경 의존성 문제, 도구 제한, 인코딩 문제, 설계 충돌 또는 우회 조치가 발생하면 루트 `Troubleshootinglog.md`에 T-ID로 기록합니다.
- `Troubleshootinglog.md`에는 발생 단계, 증상, 영향, 확인한 원인, 시도한 조치, 최종 처리, 재발 방지 또는 후속 조치를 기록합니다.
- 단순한 예상 위험은 Worklog의 남은 위험에 쓰고, 실제로 발생해 작업에 영향을 준 문제만 Troubleshootinglog에 기록합니다.
- 파일을 변경하지 않은 조사·설명 작업도 프로젝트 방향, 근거 또는 후속 작업에 영향을 주면 기록합니다.
- 범위, 아키텍처, 도구, 외부 자료, 보안, 데이터 처리, 검증 또는 제출 방식에 영향을 주는 결정을 `Decisionlog.md`에 기록합니다.
- `Decisionlog.md`에는 상황, 검토한 선택지, 결정, 공개·검증 가능한 근거, 영향과 재검토 조건을 기록합니다.
- 중요한 새 결정이 없으면 불필요한 Decisionlog 항목을 만들지 말고 Worklog에 `새로운 중요 결정 없음`으로 표시할 수 있습니다.
- 두 기록은 누적 방식으로 관리합니다. 기존 항목을 삭제하거나 의미가 바뀌게 고치지 않고 오류는 새 정정 항목으로 남깁니다.
- 원본 대화 전문은 `logs/`에 보존하므로 Worklog와 Decisionlog에 대화를 중복 복사하지 않습니다.
- 모델의 비공개 사고 전개를 기록하지 않고, 사람이 검토할 수 있는 선택지·결론·근거·영향만 간결하고 구체적으로 기록합니다.
- 비밀번호, API 키, 토큰, 실제 계좌·고객정보와 기타 비밀정보를 두 기록에 넣지 않습니다.
- Worklog와 Decisionlog의 내용은 실제 파일, 테스트 결과, README, 질문지와 모순되지 않아야 합니다.
- 사용자가 지정한 Git 원격 저장소가 연결된 뒤에는 단계 완료 시 로컬 기록·Notion 동기화까지 마친 변경사항을 커밋하고 원격에 push합니다.
- GitHub push 대상에는 기본적으로 `logs/`, 로컬 AI·IDE 상태, 비밀정보와 제출용 압축 산출물을 포함하지 않습니다. `logs/`는 로컬 제출 패키징 단계에서만 원본 그대로 사용합니다.

## 13. Notion 포트폴리오 동기화

- 이 프로젝트에서 완료되는 모든 사용자 요청 단위를 기록하며 중간 진행 메시지마다 별도 페이지를 만들지 않습니다.
- 파일을 바꾸지 않은 설명·조사·계획도 프로젝트 이해, 근거 또는 방향에 영향을 주면 기록합니다.
- 로컬 `Worklog.md`와 필요한 `Decisionlog.md`를 먼저 갱신한 뒤 Notion을 동기화합니다.
- 일반 작업은 MyProject의 현재 Phase 본문에 같은 W-ID로 추가합니다.
- 범위, 아키텍처, 데이터, 보안, 검증 또는 제출 방식을 바꾸는 결정은 관련 Phase의 D-ID 하위 페이지로 만듭니다.
- 새 기술이나 외부 자료 학습이 있을 때만 Study DB를 갱신합니다.
- 생성·수정 전 기존 W-ID·D-ID를 확인하고 완료 후 해당 페이지를 다시 읽어 검증합니다.
- Notion에는 원본 대화 전문, 비공개 사고 전개, 비밀정보와 실제 고객·계좌정보를 기록하지 않습니다.
- AX 허브, `[AX]` MyProject 항목과 `Topic=조코딩 AX해커톤` Study DB 범위 밖의 기존 페이지를 수정하지 않습니다.
- Notion 연결이나 권한이 없으면 로컬 기록을 완료하고 `Notion 동기화 대기`를 Worklog와 최종 보고에 명시합니다.
- 연결이 복구되면 로컬 W-ID·D-ID와 대조해 누락된 항목만 보충합니다.
- 대상 URL, Phase 분류, 기록 형식과 실패 처리는 `docs/notion-recording-guide.md`를 따릅니다.
- 이 규칙은 이 프로젝트의 `AGENTS.md`를 읽는 Codex 세션에 적용되며 백그라운드나 다른 AI 도구에는 자동 적용되지 않습니다.

## 14. 서브에이전트 협업

- 구현·검증 작업은 필요할 때 메인 에이전트 1명과 서브에이전트 최대 3명으로 분담하며, 작은 단일 파일 수정과 사용자 승인 판단은 메인 에이전트가 직접 수행합니다.
- 메인 에이전트는 사용자 대화, 범위·우선순위 결정, 공용 인터페이스, 파일 소유권, 결과 통합, 최종 테스트와 보고를 책임집니다.
- `AGENTS.md`, 기획서, 지표 계약, `Worklog.md`, `Decisionlog.md`, Notion과 최종 통합 파일은 메인 에이전트가 관리합니다.
- 서브에이전트는 목표, 완료 조건, 허용 파일, 읽기 참고, 금지 사항, 검증 명령, 중단 조건과 반환 형식이 명시된 작업 브리프 안에서만 작업합니다.
- 한 파일은 한 시점에 한 에이전트만 수정합니다. 서브에이전트는 허용된 파일 밖의 변경이 필요하면 임의로 수정하지 않고 메인 에이전트에게 보고합니다.
- 계약과 공용 인터페이스를 먼저 고정한 뒤 입력·보안, 분석 엔진과 QA처럼 파일 소유권이 겹치지 않는 작업만 병렬화합니다.
- 구현한 에이전트와 최종 검증하는 에이전트를 가능한 한 분리하고, 서브에이전트의 완료 보고를 테스트 없이 그대로 수용하지 않습니다.
- 서브에이전트는 지표·수식·오류 판정을 새로 만들거나 계약을 임의 변경하지 않습니다. 변경이 필요하면 코드 수정을 멈추고 사용자 승인 대상 제안으로 반환합니다.
- 어떤 에이전트도 `logs/`를 수동 편집하거나 비밀정보·실제 고객·계좌정보를 작업 브리프, 코드, fixture와 보고에 넣지 않습니다.
- `Worklog.md`, `Decisionlog.md`와 Notion 동기화는 중복을 막기 위해 메인 에이전트만 수행합니다.
- 시간 초과나 불완전한 결과가 발생하면 대상 문서·함수·파일과 반환 길이를 줄여 재분배하며, 미검증 범위를 최종 보고에 명시합니다.
- 역할, Phase별 작업 패킷, 파일 소유권, 핸드오프와 품질 게이트는 `docs/subagent-operating-guide.md`를 따릅니다.

## 15. 완료 기준

작업 완료를 보고하려면 다음 항목을 충족해야 합니다.

- 요청한 코드 또는 문서가 실제 파일에 반영되어 있습니다.
- 관련 테스트와 구조 검사를 실행했으며 결과를 확인했습니다.
- 서브에이전트를 사용했다면 메인 에이전트가 변경 파일과 실제 검증 결과를 재확인하고 통합했습니다.
- 보안·개인정보·로그 무결성 규칙을 위반하지 않았습니다.
- `Worklog.md`를 갱신했으며 중요한 새 결정이 있으면 `Decisionlog.md`도 갱신했습니다.
- 구현 중 실제 문제가 발생했다면 `Troubleshootinglog.md`를 갱신했습니다.
- Notion 동기화를 완료했거나 연결 실패로 인한 대기 상태를 명시했습니다.
- Notion을 갱신했다면 사용자에게 관련 페이지 링크를 보고했습니다.
- Git 원격 저장소가 연결되어 있고 사용자가 push를 요청한 범위라면 커밋과 push를 완료했거나 실패 사유를 명시했습니다.
- 사용자에게 변경 파일, 핵심 결과, 전체 Phase 내 위치, 남은 태스크, 남은 예상 작업량, 검증 결과와 남은 한계를 간결하게 보고했습니다.
